## Specific attacks to run
1. **Upgrade storage safety.** Count the used storage slots against `__gap[46]`. Does the declared gap
   actually match the layout? On an upgrade that adds a variable, can `netuid` / `validatorHotkey` /
   `wrapperColdkey` / `ccipAdmin` be shifted or clobbered? Confirm `_disableInitializers()` (~:121)
   genuinely protects the implementation and that `initialize` (~:130) is single-shot and cannot be
   re-entered via a reinitializer gap. Compare the UUPS wiring to OZ's canonical pattern.
2. **Admin role separation.** `owner()` (Timelock) vs `ccipAdmin` (registrar). Verify `setCCIPAdmin`
   (`onlyCcipAdmin`) cannot escalate to any owner-only power, that `ccipAdmin` has no mint/upgrade/fund
   reach, and that it is correctly NOT subject to `whenNotPaused` (or that this is harmless). Check
   `pause`/`unpause` are owner-only (G-1 zero-address guards at init).
3. **Coldkey immutability.** `wrapperColdkey` is derived once at init from the AddressMapping precompile.
   Can any path (upgrade, re-init, a precompile remap) change it after init, redirecting backing? The
   X-Ray claims `test_coldkeyImmutable` â€” try to break that claim across an upgrade.
4. **The pause-asymmetry incident.** Spell out the concrete scenario: owner pauses `deposit` in response
   to a detected precompile compromise; `redeem` stays open; an attacker (or panicking honest users)
   redeem against a lying `removeStake` magnitude. Is there ANY guard that protects redeem payout
   integrity when deposits are paused, or is the pause defense fully bypassed via redeem?
5. **Conservation topology backstop.** Read `SzAlphaLockReleasePool.sol` / `SzAlphaTokenPool.sol` and
   the deploy script (`contracts/script/`). Is `localTokenDecimals == 18` enforced on-chain at
   construction (X-Ray claims yes, G-18)? Is the lock-vs-burn placement enforced anywhere on-chain, or
   only by deploy choice (seam S2)? Compare to `reference/chainlink-ccip` and
   `reference/ccip-starter-kit-foundry`: does the canonical recipe add a check this deploy omits?
6. **Lifecycle/init temporal risk.** The genesis seed deposit + timelock/ccipAdmin handoff happen in the
   deploy script, not on-chain. Is there a window between deploy and handoff where a non-final owner or
   an un-seeded state is exploitable?
