# Mission 1 — The push-path / freshness / replay adversary

MISSION: 1 — push-path integrity & freshness

## Who you are
You are the adversary (threat-model #5, stale-rate / CRE-push) who attacks the *write path* and the
*freshness contract*. You want to land a value the guards should reject, replay or reorder a push,
freeze the feed, or exploit the gap between "oracle exposes freshness" and "consumer actually checks
it." You are NOT trying to defeat DON consensus (that's off-chain trust) — you attack what the contract
itself does with whatever the Forwarder delivers.

## The claims you are attacking
- **I-1 (On-chain=Yes):** `latest.ts` is strictly monotonic — no replay, no out-of-order accept.
  `invariant_latestTsTracksMonotonicAccepted` claims it always equals the strictly-increasing
  last-accepted ts. **Find an ordering, equal-ts, or boundary case that accepts a non-increasing ts.**
- **I-2 (On-chain=Yes):** a published rate is never 0; `exchangeRate()==0` ⇒ never pushed only.
  **Find a path where a 0 (or effectively-0 after decode) lands, or where a real published value reads
  back as 0.**
- **Forwarder gating:** `_processReport` is reachable only via `ReceiverTemplate.onReport`, Forwarder-gated.
  **Confirm there is no other reachable state-mutating path, and that the inherited gate isn't loosened.**
- **S3 freshness seam (cross-contract):** the oracle exposes `fresh()` but does not enforce the consumer
  gate. `exchangeRate()` returns 0/stale **silently**. **Trace every consumer and confirm it gates on
  `fresh()` before using the rate** — `SzipNavOracle` issuance is claimed to; the exit path intentionally
  does not (the §7 asymmetry). Find a consumer (or a code path inside one) that reads `exchangeRate()`
  without a `fresh()` check.

## Specific attacks to run
1. **Timestamp boundary.** Read the guard at `_processReport:~84-89`. Is the comparison strictly-newer
   (`ts > latest.ts`) or `>=`? Can two pushes with equal ts both land? Can a `ts == now` (not-future
   boundary) land, and is `now` `block.timestamp` (manipulable ±15s by a proposer)? What if the very
   first push carries `ts == 0`?
2. **Decode / report-envelope trust.** How is `(rate, ts)` decoded from the report bytes? A malformed or
   short report, a wrong `reportType`, extra trailing bytes — does any of these decode to a garbage rate
   that still passes the non-zero / not-future / newer guards? Compare to the x402/CRE base's expected
   envelope (`reference/x402-cre-price-alerts/`).
3. **Freshness window.** `fresh()` = `latest.ts != 0 && now - latest.ts <= maxStaleness`. Is there an
   off-by-one at exactly `maxStaleness`? Can `maxStaleness` be set so large at deploy that "fresh" is
   meaningless? Can a frozen feed (no pushes) ever read as fresh?
4. **Consumer fail-open hunt (the real prize).** Grep the repo for callers of `exchangeRate()` /
   `SzAlphaRateOracle`. For each, confirm a `fresh()` gate precedes use. The X-Ray *asserts* SzipNavOracle
   issuance gates — verify it in code, and find any other consumer (a keeper, a module, a view used for a
   decision) that consumes the rate without gating. A silent-0 read in a NAV calc is a high-severity find.
5. **Replay across an upgrade/redeploy.** If the oracle is redeployed (RMN rotation, fix), can a stale
   push or a stale `latest` be carried/replayed? Is there any cross-deployment replay surface?
