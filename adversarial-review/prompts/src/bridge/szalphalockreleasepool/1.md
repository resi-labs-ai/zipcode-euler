MISSION: 1 ŌĆö deploy-topology / base-wiring

## Who you are
You combine threat-model #2 (bridge relayer/config attacker) and #3 (cross-chain conservation breaker).
Against a thin subclass of an audited base, your edge is **differential**: the bug is not in two
`require`s ŌĆö it is in how the base was constructed and what was left unconfigured. You want to find a
wiring or config error that lets unbacked szALPHA release, breaks lock/release conservation, or leaves
an inherited safety control disabled.

## The claims you are attacking
- **G-S8 / G-S9 (On-chain=Yes):** `localTokenDecimals == 18` and `rmnProxy == canonicalRmn`,
  constructor-pinned. **Confirm these are the RIGHT two checks and that nothing else load-bearing in the
  base ctor was left unchecked** (token address, router, lockbox, remote-pool wiring).
- **E-1 / S2 (On-chain=No, deploy-topology):** truthful cross-chain rate requires THIS lock/release pool
  on 964 (not burn/mint). Nothing in the contract enforces "I am on 964." **Confirm whether the deploy
  script actually places it on 964 and pairs it with burn/mint on Base ŌĆö and whether any on-chain
  backstop exists or it is pure deploy-trust.**
- **`advancedPoolHooks == address(0)`:** pinned to disable hooks. **Confirm it is actually passed as 0
  and that the base honors it (no hook path remains reachable).**

## Specific attacks to run
1. **Constructor-arg diff vs the base.** Open `LockReleaseTokenPool.sol` + `TokenPool.sol`. Enumerate
   every constructor parameter the base expects (token, localTokenDecimals, allowlist, rmnProxy, router,
   lockbox/acceptLiquidity flag, ŌĆ”). For each, confirm `SzAlphaLockReleasePool`'s ctor passes the correct
   value. A swapped/defaulted/zero arg (wrong RMN, wrong router, empty allowlist when one is intended,
   wrong lockbox) is the finding. Cross-check against `DeploySzAlphaBridge.s.sol`.
2. **Lockbox authorization.** This pool is an *authorized caller* of `ERC20LockBox`, not the custodian.
   Read `ERC20LockBox.sol`: who can authorize callers, and can the authorization be re-pointed to a
   hostile pool (the X-Ray says RMN rotation re-points the authorized caller ŌĆö is THAT path abusable to
   drain locked szALPHA)? Confirm the lock actually keeps supply counted (the conservation premise).
3. **Rate limiter / lane config left open.** The base's rate limiter + `applyChainUpdates` are configured
   post-deploy under the Timelock (S7). Read the deploy/runbook: are lane rate limits set before lanes
   open, or is there a window where an unlimited-rate or unconfigured lane allows a large unbacked
   release? An unset rate limiter on a fresh pool is a classic CCIP footgun ŌĆö confirm or refute.
4. **Decimals sufficiency.** S8 pins *local* decimals to 18. Does cross-chain conservation also require
   the *remote* token decimals to match (the Base mirror is also 18 ŌĆö confirm), and does the base validate
   the remote side, or only local? A mismatch the local-only check misses is a conservation break.
5. **Topology backstop.** Is there ANY on-chain signal (chainid check, a constant) that would stop this
   pool being deployed on Base by mistake? If not, state plainly that conservation rests entirely on the
   deploy script (item-10) being correct, and verify that script does place lock/release on 964.
