# Mission 1 — The encoder-discipline adversary (can a dangerous attribute reach the forwarded call?)

MISSION: 1 — hardcode-dangerous / inject-addressable, the op allow-list, and REPAY's double-defense

## Who you are
You attack the on-chain half of the security model: the claim that this encoder can NEVER forward a
dangerous call — no value transfer, no delegatecall, no redirected receiver/spender, no redirected REPAY
sink — because the dangerous attributes are hardcoded literals and the addressable ones are injected from
storage, with only REPAY's `to` payload-carried (and that double-defended). Your goal: ONE path where a
payload steers an attribute it must not, where the op allow-list is escapable, or where the calldata the
encoder builds doesn't actually match what the off-chain scope pins (a leak, or a silent fail-open).

## The claims you are attacking
- **I-1: value/operation/shouldRevert are constants, never payload-decoded.** The sole call site is
  `roles.execTransactionWithRole(to, 0, data, OP_CALL, roleKey, true)` (`:211`); `OP_CALL = 0` (`:42`).
  **Confirm no code path decodes `value`, `operation`, or `shouldRevert` from the report** — trace every
  branch of `_processReport:179-213`. A delegatecall or value attach must be unreachable FROM THIS CONTRACT
  (the on-chain half of the no-delegatecall guarantee; the scope is the other half). Find any branch that
  passes a non-literal into those three positions.
- **I-2: the op allow-list.** `_processReport:185-207` is an if/else on `opType ∈ {SUPPLY=1, APPROVE=2,
  REDEEM=3, REPAY=4}` with a trailing `else revert UnsupportedOpType(opType)` (`:206`). Each op builds
  exactly one `(to, selector)`: SUPPLY → `eePool.deposit(amount, warehouseSafe)`, APPROVE →
  `usdc.approve(eePool, amount)`, REDEEM → `eePool.redeem(shares, warehouseSafe, warehouseSafe)`, REPAY →
  `usdc.transfer(redemptionBox, amount)`. **Find an `opType` byte that escapes the allow-list, or an op
  whose built `(to, selector, params)` differs from what the X-Ray/scope claims** (e.g. a wrong selector, a
  target that isn't `eePool`/`usdc`, a param in the wrong position so the scope's `EqualTo`/`avatar` pin
  lands on the wrong word).
- **I-3: REPAY is double-defended.** `_processReport:197-204`: `(dest, amount) = decode(payload)`;
  `if (dest != redemptionBox) revert WrongRedemptionBox(dest)`; then it injects `redemptionBox` (NOT `dest`)
  into the transfer calldata. **Confirm the INJECTED address is the immutable `redemptionBox`, not the
  payload `dest`** — so even if the self-check were removed, REPAY can't be redirected from this contract.
  Find any path where the payload `dest` reaches the actual transfer.
- **The unbounded amounts.** SUPPLY/APPROVE/REDEEM decode `amount`/`shares` as a raw `uint256` with no
  bound. **Pressure-test APPROVE specifically:** `usdc.approve(eePool, amount)` with a huge/infinite
  `amount` leaves a large standing USDC allowance from the Safe to `eePool`. Is that a real risk (does any
  path let that approval be consumed adversarially) or is it bounded because the scope pins
  `spender == EqualTo(eePool)` (a trusted, in-scope venue) and EE only pulls on deposit? Decide whether
  APPROVE needs an amount bound or whether unbounded-approve-to-a-pinned-spender is acceptable. A CRE-supplied
  huge SUPPLY/REDEEM amount is grief bounded by the EE/modifier revert (CRE trust) — confirm it stays grief.

## Specific attacks to run
1. **Attribute-injection sweep.** Walk all four branches: is there ANY way the report payload influences
   `value`, `operation`, or `shouldRevert`? They're literals at `:211` — confirm, and confirm no branch
   builds `data` such that the modifier would interpret it as a delegatecall/value request despite the Call/0
   literals (it can't — those are separate args — but confirm the args are positional-correct vs `IRoles`).
2. **Calldata-vs-scope alignment.** For each op, compare the encoded calldata's param ORDER against the
   scope pin the X-Ray claims (SUPPLY `deposit(amount, receiver)` → scope pins receiver==avatar at param 1;
   REDEEM `redeem(shares, receiver, owner)` → receiver==owner==avatar at params 1,2; REPAY
   `transfer(to, amount)` → to==redemptionBox at param 0). A mismatch between the word the encoder fills and
   the word the scope pins is either fail-closed (DoS) or — worse — fail-open if the pin lands on a
   payload-controlled word. Work out which, per op. Confirm `IEulerEarn.deposit`/`redeem` selector + arg
   order in `contracts/src/interfaces/euler/IEulerEarn.sol` match the real EE.
3. **REPAY redirect attempt.** Set `dest = redemptionBox` (passes the self-check) but try to get a different
   address into the transfer — impossible since `redemptionBox` (immutable) is injected, not `dest`. Confirm.
   Then try `dest = attacker` → `WrongRedemptionBox` revert. Confirm both. (`test_Repay_Happy`,
   `test_Repay_RevertsOnWrongSink` — only report a NEW redirect path.)
4. **APPROVE infinite-allowance economics.** Construct the worst case: CRE sends APPROVE with
   `amount = type(uint256).max`. The Safe now has an infinite USDC allowance to `eePool`. Trace whether
   anything other than an honest `eePool.deposit` (which the warehouse itself drives) can consume it — is
   `eePool` ever attacker-influenceable, or is it the pinned, trusted EE venue? Rate: acceptable standing
   approval to a pinned spender, or a finding that APPROVE should bound the amount.
5. **Malformed / short payload.** Feed a report whose inner `payload` is too short for the op's
   `abi.decode` (e.g. SUPPLY with empty payload, REPAY with only an address). Confirm it reverts (no
   zero-read that forwards a `deposit(0, …)` or `transfer(box, 0)` silently). (`test_MalformedPayload_RevertsCleanly`
   — only report a NEW degenerate-decode that forwards instead of reverting.)

Summary. A finding must be a concrete leak the suite doesn't cover — most valuably a calldata/scope param-
order mismatch, an APPROVE-amount risk with a concrete consumption path, or a degenerate decode that forwards
instead of reverting — NOT a re-proof of the value/delegatecall rejection.
