# Mission 1 — The vAMM-mint differential adversary

MISSION: 1 — vAMM addLiquidity seam (diff vs the ICHI parent)

## Who you are
You attack the **one swapped seam** — `addLiquidity`'s routerless `transfer → IVammPair.mint` — by diffing
it line-for-line against the audited prod parent's ICHI deposit. The parent was audited; the delta is not.
Your goal: find a check the parent had that the fork dropped, or a Solidly-`mint` behavior the fork
mishandles.

## The claims you are attacking
- **The seam is a faithful, atomic swap.** The X-Ray says the build is `transfer` both legs → `mint(engine)`,
  with `minShares` the slippage floor and mint-fail rolling back atomically (`_exec` bubbles the inner
  revert). **Find a path where tokens land in the pair but `mint` doesn't credit the engine** (stranded
  value), or where a failed/partial mint is reported as success.
- **`minShares` is sufficient.** Solidly `mint` returns LP shares = `min(amount0·supply/reserve0,
  amount1·supply/reserve1)` and **donates the excess of the larger side to the pool** (irrecoverable).
  `minShares` is the ONLY guard against a mis-sized build. **Find the input where `minShares` fails to
  bound the donated excess, or where the share-math diverges from what the test asserts.**
- **No check was dropped vs the parent.** The ICHI parent did `approve → deposit`; the fork does
  `transfer → mint` (no approval). **Confirm the fork didn't lose a guard the approve/deposit path
  implied** (e.g. a leftover-allowance reset, a return-value check, a balance reconciliation).

## Specific attacks to run
1. **Atomicity of transfer-then-mint.** Trace `addLiquidity` (~`:196-221`). The legs are `transfer`'d to
   the pair *before* `mint`. If `mint` reverts (or the `_exec` of `mint` returns `(false,data)`), are the
   transferred tokens rolled back? In a Safe `exec` model each call is separate — confirm the whole
   `addLiquidity` reverts (so the transfers revert too) rather than leaving legs stranded in the pair for
   anyone's next `mint` to absorb. Compare to the parent's `deposit` atomicity.
2. **Single-sided vs both-sided build.** The X-Ray notes single = transfer+mint, both = 2 transfers+mint.
   For a single-sided build (one `deposit` is 0), does Solidly `mint` behave (it needs both reserves)? Can
   a 0-side build mint dust or revert in a way the parent's ICHI single-sided deposit didn't?
3. **Share-math + `minShares` floor.** Recompute the LP shares Solidly `mint` yields for a given
   `(deposit0, deposit1)` against live `getReserves()`; confirm the `minShares` check (`:?`) compares the
   *actual minted* shares (read how — `lpBalance` delta? mint return? the pair has no router return) and
   reverts below floor. A wrong source for "shares minted" (e.g. trusting a stale balance) is the bug.
4. **The `_exec` revert-bubble on the new path.** The X-Ray says `_exec` bubbles inner revert data (Safe
   swallows reverts → `(false,data)`; unchecked would report a failed deposit as success). Confirm the
   `mint` exec is checked the same way the parent's `deposit` exec was — the fork must not have an unchecked
   exec on the swapped call.
5. **Donate-excess griefing.** Since `mint` donates the excess of the larger side, a mis-sized build gifts
   value to the pool (i.e. to all LP holders, including the engine's own position). Quantify: is the loss
   bounded by `minShares` (the engine still gets ≥ minShares LP, so the donation is "overpaid for shares
   it got"), or can a crafted ratio make the engine donate materially while still clearing `minShares`?

## What is already proven (don't re-report as new)
The ported suite (`LpStrategyModuleDemoVAMM.t.sol`, 21/21): single/both-sided `addLiquidity` share math,
the slippage floor, mint-fail atomicity rollback, the `_exec` bubble (custom error + no-data → `ExecFailed`),
and a share-math fuzz (`testFuzz_addLiquidityShareMathAndFloor`). Confirming any of these holds goes in the
Summary, not as a finding — a finding must be a concrete gap those tests miss (e.g. an adversarial ratio the
fuzz domain didn't reach, or a stranded-token path).
