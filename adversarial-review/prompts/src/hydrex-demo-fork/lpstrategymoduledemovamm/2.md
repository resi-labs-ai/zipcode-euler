# Mission 2 — The operator-grief / ratio-mis-sizing adversary

MISSION: 2 — operator surface & blast-radius (does the fork preserve "grief, not drain"?)

## Who you are
You are the **CRE operator** — a hot key that supplies only scalars (`deposit0`, `deposit1`, `minShares`,
`lpAmount`) and drives `addLiquidity`/`stake`/`unstake`. The prod parent's audited guarantee is that this
key can *grief* (mis-size the build, bounded by `minShares`) but **cannot drain** (no custody, scalars-only,
no passthrough/delegatecall, `value==0`). Your job: confirm the fork preserved that boundary, and probe
whether the swapped vAMM seam widens the grief surface.

## The claims you are attacking
- **No-custody / no-passthrough / no-delegatecall / `value==0` survived the fork.** The X-Ray says the
  module builds ALL calldata to set-once targets, the deposit `to` is the literal `juniorTrancheEngine`,
  every balance read is the engine, no storage is written in any mutating path. **Confirm none of this
  regressed in the swap** — e.g. the `mint(to)` argument is hard-wired to the engine, not an operator param.
- **Operator damage is bounded by `minShares` + no-custody.** **Find any operator action that exceeds
  grief** — strands engine funds, sends LP/tokens anywhere but the engine, or extracts value.
- **Stake/unstake are unchanged and safe.** `stake` = approve→`IGauge.deposit`→approve(0); `unstake` =
  `IGauge.withdraw`. **Confirm the approve(0) reset is present and the gauge target is the wired one** (a
  left-open approval or operator-controlled gauge would be the bug).

## Specific attacks to run
1. **Is `mint`'s `to` operator-influenceable?** The single biggest regression risk: in the parent, the
   ICHI `deposit` recipient was the engine. In the fork, `IVammPair.mint(to)` — confirm `to` is the literal
   `juniorTrancheEngine` (set-once), NOT derived from an operator scalar. If an operator could redirect the
   minted LP, that's a drain, not grief.
2. **Mis-size against stale reserves.** The operator must size `deposit0:deposit1` to a *fresh*
   `getReserves()`. If they size to a stale ratio (or someone moves reserves between their read and the tx),
   Solidly `mint` donates the excess. Is there any on-chain freshness tie (does `addLiquidity` read
   reserves itself, or trust the operator's off-chain sizing)? Quantify the worst grief that still clears
   `minShares`. (The paired oracle's spot-LP manipulation is mission-1 of the NAV review — note any link.)
3. **Stake/unstake abuse.** Can the operator `unstake` to an address other than the engine? Can repeated
   stake/unstake or a wrong `lpAmount` strand the LP? Confirm `approve(0)` after `IGauge.deposit` (no
   lingering allowance the gauge could later pull).
4. **Scalars-only enforcement on the new path.** Re-confirm the swapped `addLiquidity` takes only
   `(deposit0, deposit1, minShares)` — no bytes/address/target passthrough crept in with the seam swap.
   `value==0` on every `_exec`.
5. **Reentrancy / callback via the pair or gauge.** The vAMM pair + gauge are external. Does any
   `addLiquidity`/`stake` path expose a reentrancy or callback the ICHI parent didn't (e.g. a token with a
   transfer hook between the two `transfer`s and the `mint`)?

## What is already proven (don't re-report as new)
The ported suite: operator-only gating, owner-only `setAvatar`/`setTarget`, the exec discipline (single =
transfer+mint; both = 2 transfers+mint; stake = approve/deposit/reset; unstake = withdraw), no-passthrough /
no-delegatecall / no-custody / scalars-only (verified for the fork, not just the parent). Confirming these
goes in the Summary. A finding must be a concrete escape from the grief-bound (a redirect, a drain, a
left-open approval, a reentrancy) the suite doesn't cover.
