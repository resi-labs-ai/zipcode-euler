# Mission 3 — The clone-lifecycle / wiring adversary

MISSION: 3 — setUp / clone-init / build-phase wiring (does the slot-reuse open a hazard?)

## Who you are
You attack the module's **lifecycle and wiring**, not its LP math: the `ModuleProxyFactory` clone `setUp`,
the mastercopy init-lock, the six `onlyOwner` setters, and — the fork-specific risk — the fact that the
`ichiVault` storage slot is **reused** to hold the vAMM pair, and `setUp` reads `token0`/`token1` **live off
the pair** (the parent took them from static ICHI config). Your goal: a clone that can be re-initialized,
mis-wired, or pointed at a hostile pair/gauge.

## The claims you are attacking
- **Clone `setUp` is single-shot + mastercopy init-locked.** The X-Ray says per-clone config is set-once
  storage in `setUp` (not immutable), mastercopy init-locked (`:35-38`). **Find a re-init or an unlocked
  mastercopy.**
- **The reused `ichiVault` slot + live `token0`/`token1` read is safe.** `setUp` decodes 5 addresses and
  reads `token0`/`token1` *off the pair*. **Find a hazard the parent (static ICHI config) didn't have** —
  e.g. a pair whose `token0`/`token1` don't match the engine's actual tokens, or a setter that re-points
  the pair without re-reading/validating token0/token1, leaving `addLiquidity` transferring the wrong legs.
- **`setAvatar`/`setTarget` are owner-only; the operator can't reach them.** Confirm the zodiac-core
  inherited gating (`:167-170`) wasn't loosened.

## Specific attacks to run
1. **Re-init / mastercopy.** Confirm `setUp` reverts on a second call (initializer latch) and the
   mastercopy is init-locked so it can't be `setUp` by anyone. Diff the latch against the parent — the
   swap shouldn't have touched it, but confirm.
2. **token0/token1 ↔ pair coherence.** `setUp` reads `token0`/`token1` from the pair. `setToken0`/
   `setToken1`/`setIchiVault`(=pair) are independent owner setters. Can the owner (or a wiring mistake)
   leave `token0`/`token1` pointing at tokens that DON'T match the currently-wired pair's reserves? Then
   `addLiquidity` would `transfer` the wrong asset to the pair. Is there any on-chain consistency check
   tying the three, or is it deploy/owner discipline only? (Owner is the Timelock — so this is misconfig,
   not an attacker path; rate accordingly, but it's a fork-specific hazard the static-ICHI parent avoided.)
3. **Setter completeness vs the parent.** The fork has 6 setters (engine/operator/pair/gauge/token0/token1).
   Diff against the parent's setter set — did the fork add `setToken0`/`setToken1` (new, because tokens are
   now pair-derived)? Confirm each is `onlyOwner`, zero-address-guarded, and re-pointable per §17.
4. **Clone vs mastercopy address confusion.** Confirm `addLiquidity`/`stake` operate on the clone's
   set-once storage (the engine Safe), never the mastercopy's zeroed slots.
5. **`avatar == target == engine` enforcement.** Confirm `setUp` sets `avatar == target == juniorTrancheEngine`
   and the exec path targets that, so the module can only ever drive the intended Safe.

## What is already proven (don't re-report as new)
The ported suite: `setUp` wiring + zero-arg guards, the mastercopy init-lock, operator-only gating,
owner-only `setAvatar`/`setTarget`. Confirming these goes in the Summary. A finding must be a concrete
wiring hazard the fork's slot-reuse / live-token-read introduces that the suite + the static-ICHI parent
don't cover (e.g. a pair/token0/token1 incoherence path).
