# Mission 1 — The spot-LP-manipulation adversary

MISSION: 1 — vAMM getReserves() spot valuation & the TWAP bracket (the swapped seam)

## Who you are
You attack the **one swapped seam** — the LP leg now valued at **spot `IVammPair.getReserves()`**, which a
swap moves in-block — by diffing it against the audited parent's ICHI `getTotalAmounts()` valuation. The
ICHI parent's LP value wasn't a raw spot reserve read; the fork's is. The only defense is the
`navEntry = max(spot, twap)` / `navExit = min(spot, twap)` bracket. Your goal: prove an in-block reserve
push moves the consumer-facing price, or that the LP leg bypasses the bracket/TWAP.

## The claims you are attacking
- **I-2 (the defining fork residual):** the vAMM LP leg is valued at SPOT via `getReserves()`, manipulable
  in-block; "the only defense is the `navEntry = max / navExit = min` bracket; the LP leg must flow through
  the TWAP for the bracket to actually fence an in-block reserve push." **Confirm the LP leg actually flows
  through the TWAP** — if `spotNavPerShare` includes the spot LP value but the TWAP accumulator does NOT
  capture the LP leg (or captures it stale), the bracket is hollow and an in-block push leaks straight into
  `navEntry`/`navExit`.
- **E-1:** the bracket must be non-profitable — you can never mint at a price below what you exit at
  (`navEntry ≥ navExit` always). **Find an LP-reserve state or push that inverts or collapses the bracket.**
- **The `_legPriceOfToken` HYDX/USDC fold is correct.** LP value = `heldShares/totalSupply × reserves`,
  reserves priced HYDX (pushed leg) + USDC (`1e30` for the 6→18-dp $1). **Find a decimals/rounding error in
  the fold, or a `totalSupply==0` / empty-LP divide-by-zero the parent's ICHI path couldn't hit.**

## Specific attacks to run
1. **In-block reserve push → consumer price.** Model an attacker who swaps against the HYDX/USDC pair to
   skew `getReserves()` within a block, then triggers an issuance (`navEntry`) or exit (`navExit`). Trace
   whether the LP leg's contribution to `spotNavPerShare` uses the live (pushed) reserves and whether the
   TWAP side lags. If `navEntry = max(spot, twap)`: a push that *raises* spot LP value raises navEntry
   (bad for issuance? it makes issuance more expensive — mint fewer shares — so who profits?). If `navExit
   = min(spot, twap)`: a push that *lowers* spot lowers navExit (exit cheaper — who profits?). Work out the
   profitable direction and whether the TWAP lag actually blunts it.
2. **Does the LP leg reach the TWAP at all?** Read how the TWAP accumulator is advanced (`poke` /
   `_processReport`) and whether it captures the LP-leg value or only the pushed leg marks. The X-Ray says
   the in-block resistance "rests on the TWAP lag … demonstrated, not exhaustively fuzzed against an
   adversarial reserve push." If the TWAP only tracks pushed marks (HYDX/USD) and the LP *quantity*
   (reserves/shares) is read spot on every NAV call with no TWAP, the bracket does NOT fence the
   reserve-ratio manipulation — only the price-mark side. **This is the crux — determine it precisely.**
3. **`getReserves()` vs ICHI `getTotalAmounts()` diff.** The parent's ICHI value came from the vault's
   accounting; the fork reads raw pair reserves. Confirm the fork didn't lose a sanity bound the ICHI path
   had (e.g. ICHI's TWAP-guarded amounts), making the fork strictly more manipulable than the audited parent.
4. **Empty / unwired / single-reserve LP.** `totalSupply == 0`, `heldShares == 0`, or one reserve 0 — does
   the valuation divide-by-zero or misprice? The X-Ray claims `test_vamm_lp_zero_when_unwired_or_empty`
   covers the zero case — confirm, and probe the near-zero / dust-reserve edges the test may skip.
5. **HYDX/USDC decimal fold.** Recompute `heldShares/totalSupply × (reserveHYDX·hydxMark + reserveUSDC·1e30)`
   for a known reserve pair; confirm the 6→18-dp USDC fold (`1e30`) and the 18-dp HYDX mark compose to the
   right 18-dp USD value (the X-Ray's worked example: 1 HYDX@$1 + 2 USDC@$1 = $3). Find a rounding direction
   that favors the issuer or the exiter.

## What is already proven (don't re-report as new)
`test_vamm_lp_leg_valuation` (the $3 example), `test_vamm_lp_zero_when_unwired_or_empty`,
`test_navEntry_max_navExit_min`, `testFuzz_spotNavFormula`. Note the X-Ray's own caveat: the in-block-push
resistance is "demonstrated, not exhaustively fuzzed against an adversarial reserve push" — so a concrete,
worked in-block-manipulation path (with the profitable direction + whether the TWAP actually blunts it) is
exactly the gap to target, not a restatement of the bracket existing.
