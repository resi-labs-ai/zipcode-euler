# Mission 2 ‚Äî The status-machine, dispatch & cross-contract-resolve adversary

MISSION: 2 ‚Äî illegal transitions, the reportType/action dispatch, and the two-call escrow sequence

## Who you are
You attack the lien status machine and the dispatcher that drives it: the claim that status follows
`None ‚Üí Bonded ‚Üí {None | Defaulted}`, `Defaulted ‚Üí {Resolved | WrittenOff}` with no reverse edges, no
re-entry, no double-count ‚Äî and that the reportType-8 action decode can't be steered to the wrong handler or
a malformed payload. The highest-value target is the cross-contract resolve/writeoff sequence: the
coordinator makes TWO calls into the escrow without pre-asserting bounds, trusting the escrow's revert to
roll the report back. Find a state where that trust is misplaced.

## The claims you are attacking
- **I-4 (status machine).** Edges and their guards: `_lock` None‚ÜíBonded (`:208` guards `!= None`),
  `_release` Bonded‚ÜíNone (`:221`), `_default` Bonded‚ÜíDefaulted (`:236`), `_recovery` Defaulted‚ÜíDefaulted
  (`:255`), `_resolve` Defaulted‚ÜíResolved (`:278`), `_writeOff` Defaulted‚ÜíWrittenOff (`:299`). **Find a
  reachable status pair where an action's guard admits a source it shouldn't** (e.g. release a Defaulted
  lien, default a Resolved one, re-recognize, heal a written-off loss), or a terminal state (Resolved /
  WrittenOff) that accepts a further action.
- **G-7 / G-8 (dispatch).** `_processReport:182-185` double-decodes: `(reportType, payload)` then
  `(action, data)`. `reportType != 8` reverts (`:183`); an action outside the enum reverts via the trailing
  `else` (`:199`). **Find a decode that admits a malformed/ambiguous payload** ‚Äî e.g. an `action` byte that
  is a valid enum value but pairs with `data` for a different action (the per-handler `abi.decode` of `data`
  is the only shape check); a `data` blob that decodes successfully for the WRONG action and corrupts a
  lien. Confirm each handler's `abi.decode(data, ...)` reverts on a too-short/mis-typed blob rather than
  reading zeros.
- **The resolve/writeoff TWO-CALL sequence (the headline).** `_resolve:286-287`:
  `if (capitalSlashAmount != 0) escrow.slashXAlphaToCapital(lienId, capitalSlashAmount);`
  `if (escrow.bondAmount(lienId) != 0) escrow.slashXAlphaToCohort(lienId);`
  The coordinator heals provision to 0 and flips status to Resolved (`:280-282`) BEFORE these escrow calls.
  It does NOT pre-assert `capitalSlashAmount <= bond` ‚Äî it relies on `slashXAlphaToCapital` reverting
  `ExceedsBond` to atomically roll back the whole report (the provision heal + status flip included).
  **Construct a state where this assumption fails:** (a) can `bondAmount(lienId)` be 0 at resolve time
  (e.g. a lien defaulted but never bonded ‚Äî is that reachable?) so BOTH escrow calls are skipped and the
  provision is healed to 0 with no bond routed? Trace whether a Defaulted lien always has a live bond.
  (b) `_writeOff:303-304` has the IDENTICAL two-call shape ‚Äî same questions.

## Specific attacks to run
1. **Default-without-bond.** Is `Defaulted` reachable for a lien whose escrow bond is 0? `_default` only
   requires status `Bonded` (`:236`), and `_lock` (`:206-215`) sets status Bonded then calls
   `escrow.lockXAlpha` ‚Äî which reverts if `amount==0` (escrow G-15) or on a failed pull (CEI). So a Bonded
   lien in the coordinator should always have a live escrow bond. CONFIRM the coordinator status and the
   escrow `bondAmount` cannot diverge (e.g. the escrow re-pointed mid-lifecycle via `setCoordinator`, or
   the coordinator re-pointed via the escrow's setter, orphaning a bond). If they CAN diverge, the resolve
   sequence can heal provision with no bond ‚Äî rate it.
2. **The ExceedsBond rollback.** Confirm an over-bond `capitalSlashAmount` in `_resolve` reverts the WHOLE
   tx (the `totalProvision -= provision`, the `provision = 0`, the status‚ÜíResolved all roll back) ‚Äî i.e.
   the escrow revert propagates and nothing is stranded. Find any state write in `_resolve` that would
   persist across the escrow revert (there shouldn't be ‚Äî it's one tx). Already pinned by
   `test_resolve_exceedsBond_atomic_rollback`; only report a NEW stranding path.
3. **Resolve ordering vs the cohort read.** `_resolve` reads `escrow.bondAmount(lienId)` AFTER the capital
   slash to decide whether to call cohort. If `capitalSlashAmount == bondAmount` exactly, the capital slash
   drives the bond to 0 and the cohort call is correctly SKIPPED (no `NoBond` revert). Confirm the
   boundary: an exact-full-bond capital slash must skip cohort, a partial must route the remainder. Find an
   off-by-one or a re-read that sees a stale value.
4. **WriteOff is terminal.** After `_writeOff` flips to WrittenOff (`:301`), confirm G-12 (`:299`,
   `:255`, `:278`) blocks any further recovery/resolve/writeoff and G-10 (`:221`) blocks release ‚Äî the lien
   is frozen with its residual. Find any action that re-touches a WrittenOff lien.
5. **Re-bond after a clean exit.** After `_release` (Bonded‚ÜíNone), the lien is None again ‚Äî can it be
   re-LOCKed (G-9 admits None)? Confirm a fresh bond is legal but a re-lock of a still-bonded/defaulted lien
   reverts. Then confirm the escrow's `BondExists` (`:160`) is the second line of defense if the coordinator
   status and escrow state ever disagree.
