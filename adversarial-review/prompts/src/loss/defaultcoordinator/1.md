# Mission 1 — The provision-arithmetic & conservation adversary (can NAV desync, over-impair, or inflate?)

MISSION: 1 — conservation, the default bound, the heal floor, and the sole-writer↔oracle seam

## Who you are
You attack the loss accounting itself: the claim that `totalProvision == Σ lienLoss[lienId].provision ==
oracle.provision()` at all times, that provision falls only by `atRisk×(1−recoveryFloor)`, heals up only by
realized receipts (or fully to 0 on resolve), is floored at 0, and never marks NAV above the un-impaired
basket. Your goal: ONE path where the three quantities desync, where provision is marked DOWN by more than
`atRisk` (over-impairment), or UP above real backing (NAV inflation — the dangerous direction; concurrent
exiters then exit-RICH and drain the basket). NAV inflation > over-impairment in severity here.

## The claims you are attacking
- **I-1 (conservation).** `totalProvision == Σ lienLoss[lienId].provision`, incl. WrittenOff residual. The
  Δ-pairs: `_default:240-242` (`provision = p; totalProvision += p`), `_recovery:259-260` (`provision -=
  reduction; totalProvision -= reduction`), `_resolve:280-281` (`totalProvision -= provision; provision =
  0`). `_writeOff` touches neither. **Find any write of `provision` or `totalProvision` that is not matched
  by the paired write**, or an action ordering where the two drift (e.g. an external call between the two
  writes that re-enters — though escrow legs are `nonReentrant`; confirm there is no re-entrant path back
  into the coordinator before both writes land).
- **S5 / E-1 (sole-writer↔oracle seam).** `navOracle.writeProvision(totalProvision)` is pushed after every
  change in `_default:244`, `_recovery:262`, `_resolve:284` — but NOT in `_writeOff` (intentional: the
  residual IS the realized loss). **Confirm `writeProvision` is called on EXACTLY the paths that change
  `totalProvision` and skipped on exactly the one that doesn't.** Find a path that changes `totalProvision`
  without pushing to the oracle (oracle stale-low → NAV inflated) or pushes a value that isn't the current
  `totalProvision`.
- **I-2 (default bound).** `p = atRisk * (1e18 - recoveryFloor) / 1e18` (`:239`), truncating (rounds DOWN →
  favorable). `recoveryFloor < 1e18` at every write site (ctor `:132`, setter `:171`). **Find an input
  where the multiply `atRisk * (1e18 - recoveryFloor)` overflows uint256** (atRisk is CRE-supplied,
  unbounded) — does an overflow revert (safe, Solidity 0.8 checked) or wrap (catastrophic under-mark)?
  Confirm 0.8 checked arithmetic catches it. Also: a zero-result default (tiny atRisk, high floor) does NOT
  revert (`:237` only guards `atRisk==0`) — is a Defaulted lien with provision 0 a problem downstream
  (status flips, no NAV move)?
- **I-3 (heal floor at 0).** `_recovery:258` clamps `reduction = recoveryProceeds >= cur ? cur :
  recoveryProceeds`; both `provision` and `totalProvision` decrement by the same clamped value. **Find a
  path where `totalProvision -= reduction` could underflow** (would revert in 0.8 — but confirm the clamp
  guarantees `reduction <= cur <= totalProvision` by I-1, so it can't), or where a recovery on one lien
  reduces `totalProvision` below another lien's contribution.

## Specific attacks to run
1. **The resolve heal.** `_resolve:280` does `totalProvision -= lienLoss[lienId].provision` then sets
   `provision = 0` — it clears the WHOLE residual to 0 (the ratified §8.4, not a fresh receipt). Confirm
   this is consistent with I-1 (the per-lien provision was part of the sum) and that after resolve the lien
   contributes 0. Is there any lien state where `lienLoss[lienId].provision > totalProvision` so this
   underflows? (By I-1 it can't — prove it or break it.)
2. **WriteOff vs the oracle.** `_writeOff` deliberately does NOT call `writeProvision` and leaves the
   residual. So after a write-off, `totalProvision` still includes it and `oracle.provision()` still matches
   (it was pushed at the prior `_default`). Confirm the oracle and `totalProvision` STAY equal across a
   write-off (no push, no change — equal). Then confirm a WrittenOff lien can never be re-touched (G-12,
   `:299`) so its residual can't be double-removed or healed.
3. **Overflow / precision on the bound.** Push `atRisk` to `type(uint256).max` with `recoveryFloor` near 0:
   does `atRisk * (1e18 - recoveryFloor)` overflow-revert before the `/1e18`? Confirm the order of ops
   (multiply-before-divide) can overflow for large atRisk and that 0.8 reverts (fail-closed) rather than
   wrapping. Rate: a CRE-supplied huge atRisk is grief (X-1) IF it reverts; it's catastrophic only if it
   wraps to a tiny provision.
4. **Cross-lien independence.** Run default on lien A, default on lien B, recovery on A, resolve on B.
   Confirm `totalProvision` tracks `provision[A] + provision[B]` through every step and the oracle gets the
   right value each time. Find an ordering where a per-lien write leaks into another lien's accounting.
5. **The unsmoothed push (accepted, but trace it).** `_default:244` pushes the new `totalProvision`
   IMMEDIATELY — a default instantly down-marks NAV with no smoothing. The X-Ray documents this as
   accepted (the exit-path interaction is out of scope). Do NOT re-report it as a vuln; only flag if you
   find a way the immediacy breaks conservation (it doesn't — it's a faithful push) rather than just being
   economically sharp.
