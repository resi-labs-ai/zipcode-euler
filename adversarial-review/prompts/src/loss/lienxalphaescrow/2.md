# Mission 2 — The reentrancy/CEI, MAX-allowance, and build-phase-wiring adversary

MISSION: 2 — the CEI + nonReentrant battery, the coordinator's MAX allowance, and the X-2 sink re-point

## Who you are
You attack the mechanics that keep destination integrity from leaking: the CEI ordering + `nonReentrant` on
every path (belt-and-suspenders for a hookless token today), the MAX xALPHA allowance the coordinator grants
this escrow, and the build-phase mutable wiring (X-2) — the four `onlyOwner` sink/caller re-points. Your
goal: a reentrancy/CEI gap that moves funds twice or strands an orphan; a use of the MAX allowance that
drains beyond the gated custody; or a wiring re-point that DRAINS rather than griefs/redirects.

## The claims you are attacking
- **CEI + `nonReentrant` on every path.** Each state-changer writes ALL state before the external transfer
  and carries `nonReentrant`: `lockXAlpha:162-165` (mappings set, then pull), `releaseXAlpha:178-181`
  (both zeroed, then send), `slashXAlphaToCapital:196-198` (subtract, then send), `slashXAlphaToCohort:215-218`
  (both zeroed, then send). xALPHA is hookless/feeless today, so the guard is defense-in-depth. **Find a
  path where an external call happens BEFORE a state write**, or where the `nonReentrant` guard is missing,
  or a cross-function re-entry (release re-enters lock, slash re-enters cohort) the battery doesn't cover.
- **The MAX-allowance inflow seam.** The ONLY way funds enter is `lockXAlpha`'s
  `safeTransferFrom(coordinator, this, amount)` (`:165`) — pulling on the MAX allowance the coordinator
  granted via `setEscrow`. The X-Ray claims this is safe because the escrow is non-sweepable and its only
  pull path is this gated lock. **Test that:** can the escrow pull MORE than `amount` from the coordinator
  (no — `amount` is the transferFrom arg), or pull at a time the coordinator didn't intend (only via a
  coordinator-driven lock — `onlyCoordinator`)? Confirm the allowance is exercised ONLY by the escrow's own
  `lockXAlpha` and only for the exact bond `amount`.
- **X-2 (build-phase wiring).** `setXAlpha:123`, `setCoordinator:130`, `setAdminSafe:137`,
  `setJuniorTrancheSafe:144` are all `onlyOwner` (Timelock), each zero-rejecting (`ZeroWiring`, G-14).
  **Find a re-point that drains rather than griefs:** re-point `adminSafe`/`juniorTrancheSafe` to an
  attacker address, then a slash routes the bond there — is that THEFT or the documented X-2 grief/redirect
  (the Timelock is trusted in the build phase, sinks re-frozen pre-prod)? Re-point `coordinator` to an
  attacker, who then drives premature release/slash — bounded to the §13 grief ceiling, or worse? Re-point
  `xAlpha` mid-lifecycle while bonds in the OLD token are escrowed — does that strand the old bonds (the
  contract now transfers the NEW token on release/slash, but holds the OLD)?
- **Token assumptions (the I-5 caveat).** The contract is token-agnostic but assumes a non-fee, non-rebase
  xALPHA: `bondAmount` records the REQUESTED amount, and the pull `safeTransferFrom(.., amount)` is assumed
  to deliver exactly `amount`. **A fee-on-transfer xALPHA would make the escrow hold less than recorded**
  (release/slash then over-sends, draining other liens). The X-Ray flags this as the X-2 token caveat —
  confirm the deployed/bridged xALPHA is feeless/non-rebasing, and rate the consequence if a future
  re-pointed `xAlpha` (via `setXAlpha`) is fee-bearing.

## Specific attacks to run
1. **Cross-function reentrancy.** Even though xALPHA is hookless, reason about a malicious future token (or
   a re-pointed `xAlpha`) with an ERC-777-style hook on transfer. On `releaseXAlpha:181` send, a hook
   re-enters: `nonReentrant` blocks same-contract re-entry (confirm the guard wraps all four). Can a hook on
   the OUTBOUND transfer re-enter a DIFFERENT escrow function before this one returns? Both mappings are
   already zeroed at `:178-179` before the send, so a re-entrant release sees `NoBond` — confirm. Walk
   `slashXAlphaToCohort` and `slashXAlphaToCapital` for the same property. The battery
   (`test_reentrancy_release_sameFn_blocked` / `_release_crossFn_blocked` / `_cohort_sameFn_blocked` /
   `_lock_reentry_blocked` / `_outer_completes_balance_moves_once`) covers this — only report a NEW
   re-entrant path the battery misses.
2. **Failed-pull orphan.** `lockXAlpha` writes the mappings (`:162-163`) BEFORE the pull (`:165`). The
   X-Ray claims a failed pull reverts the whole tx leaving NO orphan (CEI: the storage writes roll back
   with the revert). Confirm: a `safeTransferFrom` revert (insufficient allowance/balance) unwinds
   `bondAmount`/`bondOriginator`. `test_lock_failedPull_leaves_no_orphan` pins it — confirm the ordering
   genuinely makes the writes atomic with the pull (same tx, both roll back).
3. **The MAX-allowance drain question (cross-contract).** The drain concern lives on the COORDINATOR side
   (a re-pointed attacker-escrow getting the MAX allowance) — that's the coordinator's Mission 3. HERE,
   confirm the escrow side: this escrow can only `transferFrom` on its OWN allowance via `lockXAlpha`, for
   the exact `amount`, into its OWN gated custody — it cannot pull to a third party (no recipient param on
   the pull; destination is `address(this)` `:165`). So the escrow holding a MAX allowance is not itself a
   drain primitive. Confirm and state the boundary.
4. **xALPHA re-point mid-lifecycle.** `setXAlpha:123` swaps the token while a lien holds a bond in the old
   token. On the next `releaseXAlpha`/slash, the contract calls `xAlpha.safeTransfer` on the NEW token —
   sending new-token balance it may not have (revert) while the old-token bond is stranded. Trace this as an
   X-2 consequence (grief/strand under a trusted Timelock, re-frozen pre-prod) — classify, don't inflate.
5. **Zero-wiring guards.** Confirm all four setters reject the zero address (G-14, `:124/131/138/145`) and
   the ctor rejects any zero slot (G-13, `:111`). Probe whether re-pointing `coordinator` to a non-contract
   EOA bricks the escrow (the EOA can't be driven by the CRE flow) — a grief/DoS, classify it.

## What is already proven (don't re-report as new)
The 5-test reentrancy battery (`test_reentrancy_release_sameFn_blocked` / `_release_crossFn_blocked` /
`_cohort_sameFn_blocked` / `_lock_reentry_blocked` / `_outer_completes_balance_moves_once`),
`test_lock_failedPull_leaves_no_orphan` (no orphan on failed pull), `test_lock_zeroAllowance_reverts` /
`_insufficientBalance_reverts` (the pull preconditions), `test_setWiring_onlyOwner_and_updates` /
`_nonOwner_reverts` / `_zero_reverts` (the X-2 setters' gating). The CEI + `nonReentrant` belt-and-suspenders
is confirmed sound for a hookless token — say so in your Summary. A finding must be a concrete re-entrancy /
CEI / allowance / wiring drain path the battery and setter tests do NOT cover — most valuably a re-pointed
fee/rebase `xAlpha` breaking conservation, or a wiring re-point that drains rather than griefs (and if it's
just grief under the trusted Timelock, that's X-2 INFO, not a vuln).
