# Mission 1 — The redirect adversary (can the veNFT / votes / rebase leave the Safe's account?)

MISSION: 1 — `exerciseVe` recipient pin (I-1) + account-keyed statelessness (I-3)

## Who you are
You attack the whole safety story of an approval-free, account-keyed harvest leg: the freshly-minted veHYDX NFT
mints ONLY to `juniorTrancheEngine`, and every vote/reward/rebase accrues to the Safe purely as the `exec`
msg.sender — with NO tokenId handle an operator could redirect. Your goal: ONE path where the veNFT mints to a
non-Safe address, where votes are cast for a non-Safe account, or where the rebase credits a veNFT the Safe
doesn't own.

## The claims you are attacking
- **I-1 (`exerciseVe` recipient hard-pinned).** `lockVe` builds `IOptionToken.exerciseVe(amount,
  juniorTrancheEngine)` (`:208`) — arg 2 (the recipient) is the set-once `juniorTrancheEngine`. **Find any path
  where the operator can set the recipient to a non-Safe address.** The operator's ONLY input to `lockVe` is
  `amount` — confirm it can't perturb arg 2. The decoded `nftId` (`:209`) is only emitted, never stored or used
  as a redirect handle.
- **I-3 (account-keyed + stateless).** The Voter is account-keyed (no tokenId). `vote` (`:218`), `reset` (`:224`),
  `getReward` (`:199`), `exerciseVe` (`:208`), and `claim_many` (`:233`) all act on the Safe's own account/
  holdings because the Safe is the `exec` msg.sender. There is NO tokenId state. **Find a path where an action
  could accrue to a non-Safe account** — e.g. a call that takes an account/recipient param the operator supplies,
  or a stored id that could be swapped.
- **I-2 (one exec per action, value==0, Call-only, NO approvals).** Each action is a single `_exec` (`:185`) →
  `execAndReturnData(to, 0, data, Operation.Call)` — `value` literal 0 (`:186`), `Operation.Call`, `to` only ever
  the wired gauge/voter/oHYDX/rewardsDistributor. **Confirm there are NO `approve` calls anywhere** (the X-Ray
  asserts none exist) and no passthrough/delegatecall.

## Specific attacks to run
1. **Trace each operator argument.** `amount` → `exerciseVe` arg 1; `poolVote`/`weights` → `vote` args;
   `tokenIds` → `claim_many` arg. Confirm NONE reach an account/recipient arg or the `to`/`value` of any exec.
   All calls use TYPED `encodeCall` (`:199`, `:208`, `:218`, `:224`, `:233`) — a field-order regression fails to
   compile; confirm.
2. **`vote` account-keying.** `IVoter.vote(poolVote, weights)` (`:218`) takes NO account arg — the Voter votes
   with the msg.sender's (the Safe's) whole veHYDX position. Confirm there is no overload that takes a tokenId/
   account the operator could supply, and that a hostile `poolVote` array can only misallocate the SAFE's OWN
   voting weight (a grief, not a redirect of someone else's votes).
3. **`claimRebase` can't be redirected.** `claim_many(tokenIds)` (`:233`) credits each veNFT's OWN lock — the
   rebase accrues to whoever owns the NFT (the Safe). Confirm an operator passing tokenIds the Safe does NOT own
   either reverts or harmlessly credits those NFTs' real owners (not the operator). The X-Ray's I-7 says the
   array is imperfect-tolerant — confirm it can't redirect value to the operator.
4. **No tokenId state / no custody.** Confirm the module stores NO veNFT id (`:209` only emits) and holds no
   tokens. A stored id that could be swapped by a setter, or a balance accruing to the MODULE, would be a
   redirect; confirm neither exists.
