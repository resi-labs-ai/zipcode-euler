# Mission 3 — The external-integration & exec-integrity adversary

MISSION: 3 — the heaviest external surface (gauge/voter/ve/distributor), bubbled false-return (I-4), live-read targets (I-6)

## Who you are
You attack the densest external-integration seam of the fleet: five Hydrex contracts, each account-keyed, each
reached through a single `_exec`. The claims: a Safe-swallowed Hydrex revert ALWAYS hard-reverts (never reported
as success), the wired targets (`oHYDX`/`ve`) can never drift from the real dependencies, and the views read the
Safe. Your goal: ONE path where a failed claim/lock/vote/rebase reports success, where a view reads the caller
instead of the Safe, or where a live-read target desyncs from its dependency.

## The claims you are attacking
- **I-4 (exec false-return hard-reverts with bubbled data).** `_exec` (`:185-194`): the Gnosis Safe's
  `execTransactionFromModuleReturnData` catches inner reverts and returns `(false, revertData)`. `_exec` reverts
  on `!ok` — bubbling `ret` if non-empty (`:189-191`), else `ExecFailed` (`:188`). **Find a path where an inner
  Hydrex failure returns `(false, …)` yet the module proceeds** — note only `lockVe` decodes a return (`:209`);
  `claimReward`/`vote`/`reset`/`claimRebase` ignore it. A swallowed failed claim/vote that emits a success event
  would be the bug.
- **I-6 (live-read targets).** `setUp` reads `oHYDX = IGauge(gauge).rewardToken()` (`:105`) and `ve =
  IVoter(voter).ve()` (`:106`), asserts non-zero (`:107`). `oHYDX` is the `exerciseVe` target; `ve` is the floor-
  read escrow. **Find a path where `oHYDX`/`ve` drift from the real dependency** — e.g. `setOHYDX`/`setVe` (`:166`,
  `:173`) overriding to a mismatched address, or `setGauge`/`setVoter` (`:145`, `:152`) re-pointing the dependency
  WITHOUT re-deriving `oHYDX`/`ve` (they don't re-derive — is that a desync risk? Note `lockVe` would then
  `exerciseVe` on a stale oHYDX). Rate: these are `onlyOwner` (Timelock) acts (X-2); worst case is a fail-closed
  revert or a mis-targeted (but Safe-owned) action.
- **I-5 (views read `juniorTrancheEngine`).** `pendingReward` (`:239`), `voteFloor` (`:244`), `rebaseClaimable`
  (`:249`) read `juniorTrancheEngine`/a passed tokenId, NEVER the caller. **Confirm no view reads `msg.sender`**
  (a view that read the caller would mislead the off-chain sizing, not move value — but confirm).

## Specific attacks to run
1. **The swallowed-failure matrix.** For each of the 4 non-decoding actions (`claimReward`, `vote`, `reset`,
   `claimRebase`), confirm the single `_exec` (`:199`, `:218`, `:224`, `:233`) hard-reverts on `!ok` so a failed
   Hydrex call can't emit its success event (`RewardClaimed`/`Voted`/etc.). The event emits AFTER `_exec` returns
   — so a revert prevents the event. Confirm.
2. **`setGauge`/`setVoter` without re-derive.** `setGauge` (`:145`) re-points `gauge` but does NOT re-read
   `oHYDX = gauge.rewardToken()`; `setVoter` (`:152`) doesn't re-read `ve = voter.ve()`. So after a dependency
   re-point, `oHYDX`/`ve` could be stale. Trace the consequence: `lockVe` would `exerciseVe` on the OLD oHYDX
   (which may not be the new gauge's reward token), and `claimReward` would claim from the NEW gauge. Is that a
   drain (no — value still goes to the Safe), a fail-closed revert, or a silent mis-harvest? Note whether the
   Timelock must also call `setOHYDX`/`setVe` to keep them in sync (X-2, governance discipline).
3. **The empty/short return on `lockVe`.** `nftId = abi.decode(ret, (uint256))` (`:209`). Confirm `_exec`'s
   `ret.length == 0` check (`:188`) catches an empty return and a successful `exerciseVe` returns a 32-byte id.
4. **Stateless re-entrancy across the heavy surface.** No `nonReentrant`. Could a hostile gauge/voter/distributor
   (build-phase re-point) re-enter an action? The module is stateless — a re-entrant call is just a second
   operator call (operator-gated, value stays in the Safe). Rate: hostile target = X-2 wiring residual; confirm
   re-entrancy adds no new capability given statelessness + account-keying.
