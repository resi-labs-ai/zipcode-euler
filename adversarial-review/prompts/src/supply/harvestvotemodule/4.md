# Mission 4 — The privilege & wiring adversary

MISSION: 4 — access gates, the clone init-lock, and the build-phase wiring blast radius

## Who you are
You attack the access-control and configuration surface: the operator gate, the SEC-14 mastercopy lock, the
SEC-15 owner-recheck, the `setUp` guards, and the 7 `onlyOwner` wiring setters. Your goal: ONE path where a
non-operator drives an action, the mastercopy is initializable, or an owner re-point DRAINS (rather than
redirects/griefs — recalling this module moves no value out and holds no approvals).

## The claims you are attacking
- **Operator gate (`NotOperator`).** All 5 actions are `onlyOperator` (`:198`, `:206`, `:215`, `:223`, `:231`);
  the modifier checks `msg.sender == operator` (`:115-118`). **Find a non-operator path to an action.** Confirm
  the operator CANNOT call any `onlyOwner` setter or the inherited `setAvatar`/`setTarget`.
- **SEC-14 (mastercopy init-lock).** The mastercopy is locked in its constructor (`MastercopyInitLock`); a clone
  is `setUp` once under `initializer` (`:77`). **Confirm `setUp` on the mastercopy reverts and a clone inits
  exactly once.** Diff against `contracts/src/supply/szipUSD/MastercopyInitLock.sol`.
- **SEC-15 (owner-recheck on `setOperator`).** `setOperator` (`:137`) rejects `operator_ == owner` (`:139`).
  **Confirm the Timelock and the hot CRE key stay distinct** across any setter ordering.
- **`setUp` guards.** Six addresses non-zero (`:87-90`), `owner != operator` (`:91`), validated BEFORE the live
  `oHYDX`/`ve` reads (`:105-107`), THEN `_transferOwnership` (`:111`). **Find a decode/validation-order bug** — a
  zero `gauge` reaching `gauge.rewardToken()` before its check (the X-Ray notes the order is load-bearing for
  exactly this).
- **X-1 / wiring blast radius.** 7 `onlyOwner` (Timelock) setters: `setJuniorTrancheEngine` (`:130` — NOTE: does
  it sync avatar/target? read it), `setOperator`, `setGauge`, `setVoter`, `setRewardsDistributor`, `setOHYDX`,
  `setVe`, plus inherited `setAvatar`/`setTarget`. **Pressure-test:** since the module holds no funds/approvals
  and the recipient is pinned, what is the worst a re-point does — redirect FUTURE harvests (to a still-Safe-owned
  account?) or mis-target an action (fail-closed)? Is there ANY re-point that drains?

## Specific attacks to run
1. **The avatar/target sync — note the difference.** Unlike Sell/Exercise/LpStrategy, `setJuniorTrancheEngine`
   here (`:130-134`) sets `juniorTrancheEngine` but does NOT re-set `avatar`/`target` (read the function — confirm
   it lacks the `avatar = target = ...` lines the siblings have). So after a re-point, the module would still
   `exec` THROUGH the old avatar Safe while reading floors/minting to the NEW `juniorTrancheEngine`. Trace the
   consequence: `exerciseVe` recipient = new engine, but the exec drives the OLD Safe (avatar). Is that a
   redirect (the new engine isn't the exec'ing Safe — so the burn/vote happens on the old Safe but the NFT mints
   to the new engine?) or a fail-closed mismatch? This is the highest-value wiring question here — work it
   carefully against the zodiac-core `execAndReturnData` (which uses `avatar`/`target`).
2. **`setOHYDX`/`setVe` mismatch.** These override the live-derived targets directly (`:166`, `:173`). A Timelock
   could set `oHYDX` to a token the gauge doesn't reward, or `ve` to a wrong escrow. Worst case: `lockVe`
   `exerciseVe`s on a wrong oHYDX (fail-closed or a Safe-owned mis-lock), `voteFloor` reads a wrong escrow (a
   misleading view). Drain or grief? Rate (X-2, owner-only).
3. **Operator → owner escalation.** Confirm the `operator` cannot reach any `onlyOwner` function or become
   `owner`. The only operator entrypoints are the 5 actions. SEC-15 keeps operator != owner.
4. **No funds to drain.** Confirm the module holds no tokens, has no approvals, and no setter can cause value to
   leave the Safe to a non-Safe party — so the wiring blast radius is "redirect future actions / fail closed,"
   not "drain." State this explicitly if it holds.
