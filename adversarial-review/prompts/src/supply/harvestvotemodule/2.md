# Mission 2 — The permalock adversary (is `lockVe` a bounded grief or worse?)

MISSION: 2 — the `lockVe` irreversible permalock (the §13 residual, X-1)

## Who you are
You attack the one irreversible action in the fleet: `lockVe` burns sellable oHYDX into a permalocked veHYDX
position. The X-Ray classifies an operator over-lock as bounded GRIEF (value stays in the Safe, never theft).
Your goal: ONE path that ESCALATES the permalock beyond grief — value reaching a non-Safe party, the lock
amount exceeding what the operator authorized, or a permalock that bricks a load-bearing path — OR a confirmation
that the residual is correctly bounded.

## The claims you are attacking
- **X-1 (the §13 `lockVe` over-lock grief).** `lockVe(amount)` (`:206`) burns `amount` oHYDX and permalocks it
  into a Safe-owned veHYDX NFT. **Confirm the over-lock is bounded to grief:** the value (the permalocked HYDX
  governance weight) stays with the Safe (recipient pinned, mission 1), so the worst case is converting liquid
  emissions into illiquid governance power — a one-way economic loss of liquidity, NOT a theft. **Find a path that
  escalates it** — e.g. the burned oHYDX or the minted weight reaching a third party, or the lock being callable
  by a non-operator.
- **The lock amount is operator-sized.** `amount` is the operator's input (`:206`); the only guard is `ZeroAmount`
  (`:207`). There is NO upper bound. **Is an unbounded `amount` dangerous?** It can burn up to the Safe's whole
  oHYDX balance into a permalock — confirm that's bounded grief (the value is the Safe's, the lock grows the
  Safe's vote floor) and that there's no path where over-locking enables anything beyond illiquidity (e.g. does
  any other module's accounting break if the Safe's oHYDX is unexpectedly zero?).
- **The permalock is genuinely irreversible.** `exerciseVe` mints a permalocked NFT — there is no module path to
  un-lock it (it's a permalock by Hydrex design). **Confirm the module offers no false "unlock" and that the
  irreversibility is a Hydrex property, not a module bug.** Note it for ops runbooks as one-way.

## Specific attacks to run
1. **Escalation hunt.** Trace `lockVe` end to end: `_exec(oHYDX, exerciseVe(amount, juniorTrancheEngine))`
   (`:208`). The burned oHYDX comes from the Safe (msg.sender); the veNFT mints to the Safe (recipient pinned).
   Confirm there is NO step where the operator chooses where the HYDX/weight goes. If the recipient pin (mission
   1) holds, the worst case is the Safe locks its own value — grief. Find any deviation.
2. **Cross-module impact of a drained oHYDX balance.** `lockVe` can burn the Safe's entire oHYDX. Does any sibling
   module assume a non-zero oHYDX balance (e.g. `ExerciseModule.exercise`, `SellModule.sellHydx` on the post-
   exercise HYDX)? An over-lock that strands a downstream leg is still grief (the CRE sequences these) — but note
   if a permalock could DoS a needed path. Rate the realism: the single CRE operator sequences harvest → lock →
   sell, so a self-inflicted ordering is operator error (X-1), not an external attack.
3. **Re-entrancy / double-lock.** No `nonReentrant`. `lockVe` is stateless (no tally). A re-entrant second
   `lockVe` is just a second operator call (bounded by the operator gate). Confirm there's no aggregation bug
   (there's no cap to aggregate past) and no state to corrupt.
4. **The `nftId` decode.** `nftId = abi.decode(ret, (uint256))` (`:209`) is only emitted (`:210`), never used.
   Confirm a malformed return can't make `lockVe` mis-behave beyond a bad event (the X-Ray notes
   `test_lockVe_reverts_on_short/empty_return_data` — `_exec`'s length check catches empty).
