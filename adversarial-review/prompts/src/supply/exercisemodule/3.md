# Mission 3 — The exec-integrity, approval-hygiene & wiring adversary

MISSION: 3 — bubbled false-return, no-standing-approval, the live `paymentToken` derive, and the wiring blast radius

## Who you are
You attack the failure handling, approval hygiene, and configuration surface: the claims that a Safe-swallowed
revert ALWAYS hard-reverts (never reported as success), that no approval survives a swap (or a failed swap), that
the `paymentToken` can never drift from oHYDX's actual one, and that the `onlyOwner` wiring re-points redirect at
worst (not drain). Your goal: ONE path where a failed exercise reports success, an allowance dangles on oHYDX,
the approve target drifts from the real payment token, or a re-point/gate escalates to redirect/theft.

## The claims you are attacking
- **I-5 (exec false-return hard-reverts with bubbled data).** `_exec` (`:156-165`): the Gnosis Safe's
  `execTransactionFromModuleReturnData` catches inner reverts and returns `(false, revertData)`. `_exec` reverts
  on `!ok` — bubbling `ret` if non-empty (`:160-162`), else `ExecFailed` (`:159`). **Find a path where an inner
  failure returns `(false, …)` yet the module proceeds as if it succeeded** — e.g. a caller that ignores the
  bool, or a `ret` that decodes to a plausible `paymentAmount` despite a failed exercise.
- **I-4 (no standing approval — approve→exercise→reset).** Exactly 3 execs; the reset (`:187`) zeroes the oHYDX
  allowance. **Find a path where the reset does NOT run** — a revert/return between the approve (`:185`) and the
  reset (`:187`) that leaves a non-zero allowance. If the exercise exec reverts (bubbled), does the WHOLE tx roll
  back, unwinding the approve? Confirm there is no try/catch swallowing the failure while keeping the approve.
- **I-6 (`paymentToken` live-read off the option).** `setUp` reads `paymentToken = IOptionToken(oHYDX).paymentToken()`
  (`:92`) and asserts non-zero (`:93`); `setOHYDX` (`:131`) re-derives it LIVE off the new option (fail-closed,
  `:134-136`). **Find a path where `paymentToken` (the approve target) can drift from oHYDX's ACTUAL payment
  token** — e.g. `setPaymentToken` (`:143`, the direct override) setting it to a mismatched token, or a `setOHYDX`
  that updates `oHYDX` but not `paymentToken`. Rate: `setPaymentToken` is `onlyOwner` (Timelock) — a deliberate
  override; is a mismatch a drain or a fail-closed (the exercise would revert on a wrong approve)?
- **X-1 / wiring blast radius.** 4 `onlyOwner` (Timelock) setters: `setJuniorTrancheEngine` (`:113`, syncs
  `avatar`/`target`), `setOperator` (`:122`, SEC-15 owner-recheck), `setOHYDX` (`:131`, re-derives paymentToken),
  `setPaymentToken` (`:143`), plus inherited `setAvatar`/`setTarget`. **Pressure-test:** does any re-point let the
  Timelock STEAL existing value, or only redirect FUTURE exercises (the X-2 residual)?

## Specific attacks to run
1. **The atomicity matrix.** Walk the 3-exec sequence. If exec #2 (the exercise) reverts, the bubbled revert
   (`:161`) propagates out → the whole tx reverts → exec #1's approve is unwound. The module is stateless beyond
   wiring, so no partial commit. Confirm a failed exercise leaves allowance 0 because the approve rolls back too.
2. **The decode/length edge.** `paymentAmount = abi.decode(ret, (uint256))` (`:189`) decodes only exec #2's
   return. The two approves' returns are ignored. Confirm a short/empty `ret` from the exercise can't decode to a
   misleading value — `_exec`'s `ret.length == 0` check (`:159`) catches empty; a successful exercise returns a
   32-byte `paymentAmount`. The X-Ray notes `test_exercise_reverts_on_short/empty_return_data`.
3. **`setOHYDX` desync.** `setOHYDX` (`:131`) sets `oHYDX` THEN re-reads `paymentToken` off the new option,
   reverting if zero (`:134-136`). Confirm there is NO ordering where `oHYDX` is updated but `paymentToken` is
   left stale (a mismatch would approve the OLD token to the NEW option → fail-closed revert, not a drain — but
   confirm). Contrast with `setPaymentToken` (`:143`), which sets it directly without touching `oHYDX` — can a
   Timelock create a deliberate mismatch, and is the worst case a bricked exercise (grief) or a drain?
4. **avatar/target sync.** `setJuniorTrancheEngine` (`:113-119`) sets `juniorTrancheEngine = avatar = target`
   together. The inherited `setAvatar`/`setTarget` (zodiac-core) can set them independently (`onlyOwner`). Since
   the exercise recipient reads `juniorTrancheEngine` (not `avatar`), a desynced avatar drives a different Safe
   than the recipient. Is that exploitable or a governance footgun? Rate severity (X-2, owner-only).
5. **Operator → owner escalation.** Confirm the `operator` (hot key) cannot call any `onlyOwner` setter or become
   `owner`. The only operator entrypoint is `exercise`. SEC-15 (`:124`) keeps `operator != owner`.
