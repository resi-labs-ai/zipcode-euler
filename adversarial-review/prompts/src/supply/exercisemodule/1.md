# Mission 1 — The redirect adversary (can the minted HYDX leave the basket?)

MISSION: 1 — recipient pin + the fixed 3-exec shape (the §10.1 boundary)

## Who you are
You attack the load-bearing claim of a paid-exercise module: the operator supplies only scalars, and the
exercised HYDX can mint NOWHERE but the engine Safe. Your goal: ONE path where the exercise `recipient`, the
`to`/`value`/`Operation` of any `exec`, or the call shape becomes operator-influenced — i.e. HYDX minted to a
non-Safe address, an arbitrary call leaking through, a non-zero `value`, or a delegatecall/passthrough.

## The claims you are attacking
- **I-1 (recipient hard-pinned).** `exercise` builds the oHYDX call as
  `IOptionToken.exercise(amount, maxPayment, juniorTrancheEngine, deadline)` (`:186`) — arg 3 (the recipient) is
  the set-once storage `juniorTrancheEngine`, re-pointable only by `onlyOwner` (`setJuniorTrancheEngine:113`).
  **Find any path where the operator can set the recipient to a non-Safe address.** The operator's ONLY inputs
  are `(amount, maxPayment, deadline)` — confirm none of them reach the recipient arg.
- **I-2 (fixed 3-exec shape).** Exactly 3 `exec`s in order: approve (`:185`) → exercise (`:186`) → reset (`:187`).
  Each goes through `_exec` (`:156`) → `execAndReturnData(to, 0, data, Operation.Call)` — `value` is the literal
  `0` (`:157`), `Operation.Call` (never `DelegateCall`), and `to` is only ever `paymentToken` (the two approves)
  or `oHYDX` (the exercise). **Find any path where `to`, `value`, or the `Operation` becomes attacker-influenced,
  or a generic exec/call passthrough** (the only `_exec` callers are the 3 calls inside `exercise`).
- **§10.1 boundary.** No `payable` entrypoint, no `msg.value` use, no delegatecall. Confirm there is no fourth
  exec path, no fallback, and that `amount`/`maxPayment`/`deadline` flow ONLY into the exercise call's args 1, 2,
  and 4 (and the approve's amount) — never into `to`/`recipient`/`value`.

## Specific attacks to run
1. **Trace each operator argument to the exec calldata.** `amount` → exercise arg 1; `maxPayment` → the approve
   amount (`:185`) AND exercise arg 2 (`:186`); `deadline` → exercise arg 4. Confirm NONE reach arg 3 (recipient),
   the `to` of either exec, or `value`. The exercise call uses TYPED `encodeCall` (`:186`) — a field-order
   regression would fail to compile; confirm. The approves use `encodeWithSelector` (`:185`, `:187`) — confirm
   the spender is the pinned `oHYDX` and the token is the wired `paymentToken`.
2. **The approve target/spender.** Both approves call `paymentToken.approve(oHYDX, …)`. Confirm `paymentToken` is
   the LIVE-derived wired token (not operator-chosen) and the spender is the wired `oHYDX` — an approve to an
   attacker spender, or of an attacker token, would be a drain vector. (The `paymentToken` live-read is mission
   3's territory; here, confirm the approve cannot be re-pointed by the operator.)
3. **`value == 0` always.** Confirm no `exec` sends native value (`:157` literal 0). A non-zero value would let
   the module drain the Safe's ETH. There is no `payable` function and no `msg.value`; confirm.
4. **Single action, no other surface.** Confirm `exercise` is the ONLY state-moving operator entrypoint (the
   other externals are the owner setters + the `quoteStrike` view). No second path to `_exec`.

asserted), `test_state_moving_happy_path_resets_allowance` (`lastRecipient == juniorTrancheEngine`),
