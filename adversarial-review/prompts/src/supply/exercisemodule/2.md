# Mission 2 — The overpay adversary (can the strike pull exceed maxPayment, or the quote mislead?)

MISSION: 2 — `maxPayment` slippage bound (delegated to oHYDX) + the `quoteStrike` view

## Who you are
You attack the strike-payment safety: the claim that the USDC pulled for the strike can NEVER exceed
`maxPayment`, even if oHYDX misreports, and that `quoteStrike` matches the same-block charge. Your goal: ONE
path where the Safe pays more than `maxPayment`, where the `PaymentExceedsMax` re-assert can be bypassed, or
where `quoteStrike` returns a value that desyncs from the actual charge in a way that mis-sizes the cushion into
an overpay.

## The claims you are attacking
- **I-3 (`maxPayment` slippage guard, two layers).** Layer 1: the approval is `paymentToken.approve(oHYDX,
  maxPayment)` (`:185`) — the ERC-20 allowance caps the real pull at `maxPayment` regardless of what oHYDX
  computes. Layer 2: oHYDX computes the strike from its own TWAP and reverts if `paymentAmount > maxPayment`
  (delegated). Layer 3 (belt-and-suspenders): `if (paymentAmount > maxPayment) revert PaymentExceedsMax()`
  (`:194`) re-asserts on the DECODED return. **Find a path where the Safe pays more than `maxPayment`** — e.g. an
  approve that exceeds `maxPayment`, a residual allowance from a prior call that stacks, or a decode that lets a
  misreporting return slip past `:194`.
- **The decoded return is trusted only for the event/accounting, not the pull.** `paymentAmount = abi.decode(ret,
  (uint256))` (`:189`) — the X-Ray notes the approval already bounds the REAL pull; `:194` guards the
  emitted/returned value that 8-B11/8-B12 accounting reads. **Confirm a malformed/short `ret` can't decode to a
  misleadingly-small `paymentAmount`** that passes `:194` while the real pull was larger (it can't — the pull is
  allowance-capped — but verify the reasoning holds, and that `_exec`'s length handling covers the empty case).
- **I-7 (`quoteStrike == max(discounted, floor)`).** `quoteStrike` (`:203`) returns `max(getDiscountedPrice(amount),
  getMinPaymentAmount())`. 8-B5 sizes the borrow + 8-B11 sizes the `maxPayment` cushion off this. **Find an input
  where `quoteStrike` returns a value that does NOT match the same-block exercise charge** (the fork test proves
  `paymentAmount == quoteStrike` same-block — find a case it misses: a different `amount`, a floor-vs-discount tie,
  an overflow in the external reads).

## Specific attacks to run
1. **Allowance stacking.** The approve sets allowance to `maxPayment` (`:185`), oHYDX pulls `paymentAmount ≤
   maxPayment`, the reset zeroes it (`:187`). Could a residual allowance from a PRIOR `exercise` (if the reset
   was skipped due to a revert) stack with a new approve to let a pull exceed the intended `maxPayment`? Trace:
   on any revert between approve and reset, does the whole tx roll back (unwinding the approve)? (It should — no
   state, atomic. Confirm there's no partial-commit path.)
2. **The re-assert ordering.** `:194` runs AFTER the reset (`:187`) and the decode (`:189`). If `paymentAmount >
   maxPayment`, it reverts — but the pull already happened (and the reset). Since the whole tx reverts, the pull
   is unwound too. Confirm the revert at `:194` rolls back the USDC pull (it does — same tx). So `:194` is a
   guard on the RETURNED value's honesty, not a fund-recovery — confirm that framing and that it can't be the
   thing that lets an overpay commit.
3. **oHYDX trust boundary (X-1).** The strike math, the TWAP, and the `paymentAmount ≤ maxPayment` enforcement
   are oHYDX's job — immutable, Base-verified. Do NOT report "oHYDX could misreport / its TWAP could be
   manipulated" as a vuln (that's the accepted X-1 oHYDX-trust). Only escalate if you find a path where OUR code
   lets a pull exceed `maxPayment` DESPITE the allowance cap, or where the approve is mis-sized.
4. **`quoteStrike` overflow / domain.** The view does two external reads and a `>` compare (`:204-206`). No
   arithmetic of our own. Confirm no overflow/under-read can make it return 0 (would mis-size the cushion to
   "free") or a huge value. Rate: if oHYDX returns garbage, that's the oHYDX-trust residual, not our bug — unless
   our `max` selection picks the wrong branch.
