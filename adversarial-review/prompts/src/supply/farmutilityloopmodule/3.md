# Mission 3 — The exec-integrity adversary (EVC call shape, atomicity, swallowed failure)

MISSION: 3 — exec discipline + atomicity (I-7), value==0/Call-only, the `IEVC.call` on-behalf shape

## Who you are
You attack the transaction mechanics across the four loop steps: the claim that each step has a fixed exec shape,
that a failed inner EVK/router call hard-reverts (bubbled, never reported as success), that no approval dangles,
and that every `IEVC.call` is `value==0` / Call-only with the on-behalf pinned to the Safe. Your goal: ONE path
where a failed borrow/repay/withdraw reports success, an approval survives, or the EVC call shape leaks (non-zero
value, a wrong on-behalf, a generic passthrough).

## The claims you are attacking
- **I-7 (exec discipline + atomicity).** `_exec` (`:207-215`) drives `execAndReturnData(to, 0, data,
  Operation.Call)` and reverts on `!ok` — bubbling `ret` (`:211-213`) or `ExecFailed` (`:210`). The Safe's
  `execTransactionFromModule` catches inner reverts and returns `(false, revertData)`; an unchecked exec would
  swallow a failed EVC borrow/repay/withdraw (so the operator believes a borrow landed when it didn't). **Find a
  path where an inner EVK/router failure returns `(false, …)` yet the step proceeds** — note `_exec` here is
  `void` (no decode), so the only question is whether `!ok` always reverts and every step routes through `_exec`.
- **Atomicity / no dangling approval.** `postCollateral` approves the escrow (`:221`); `repay` approves the
  borrowVault then resets to 0 (`:254`, `:266`). **Find a path where an approval survives a failed step** — if the
  deposit (`:224`) or repay-call (`:258`) reverts, the whole tx rolls back (unwinding the approve). The module is
  stateless beyond wiring — confirm no partial commit. `repay`'s reset (`:266`) is the security-F13 hygiene;
  confirm a failed repay rolls back the approve too.
- **The EVC call shape (value==0, Call-only, on-behalf pinned).** Every `IEVC.call` (`:239`, `:258`, `:278`) and
  every `_exec` (`:208`) passes `value == 0` and `Operation.Call`. The `IEVC.call` 3rd arg (value) is the literal
  `0` (`:243`, `:262`, `:282`) and the on-behalf is `juniorTrancheEngine`. **Find a path where `value != 0`, a
  delegatecall leaks, or the on-behalf/target becomes operator-influenced.**

## Specific attacks to run
1. **The swallowed-failure matrix.** For each step, confirm the inner call routes through `_exec` (`:207`) which
   hard-reverts on `!ok`. `postCollateral`: 3 `_exec`s (`:221`, `:223`, `:224`). `borrow`: 2 (`:236`, `:239`).
   `repay`: 3 (`:254`, `:258`, `:266`). `withdrawCollateral`: 1 (`:278`). A failed EVK call (e.g.
   `E_AccountLiquidity`, `E_RepayTooMuch`, `PriceOracle_TooStale`) returns `(false, revertData)` → `_exec` bubbles
   it. Confirm the event (`Borrowed`/`Repaid`/etc.) emits only AFTER `_exec` returns, so a failed step can't emit
   success.
2. **Atomicity rollback.** Walk `postCollateral`: approve (`:221`) → enableCollateral (`:223`) → deposit (`:224`).
   If the deposit reverts, the whole tx unwinds the approve + the enable. The X-Ray's `test_atomicity_postCollateral_
   deposit_revert_rolls_back` / `test_atomicity_repay_call_revert_rolls_back`. Confirm no state survives a failed
   step (the module has no mutable state beyond wiring/cap).
3. **The `IEVC.call` arg shape.** `IEVC.call(target, onBehalfOfAccount, value, data)`. Confirm every call site
   passes `target` = the wired vault, `onBehalfOfAccount` = `juniorTrancheEngine` (the Safe), `value` = literal
   `0`, `data` = a typed `encodeCall` to `borrow`/`repay`/`withdraw` with receiver/owner = the Safe. A non-zero
   value or a wrong on-behalf would be the leak. The operator supplies only the scalar amount — confirm it can't
   reach `target`/`onBehalf`/`value`.
4. **The double EVC layer.** `_exec(evc, IEVC.call(...))` — the module execs THROUGH the Safe (`execAndReturnData`)
   to call the EVC, which then calls the vault on-behalf of the Safe. Confirm this two-layer routing (Safe →
   EVC → vault) keeps the Safe as both the `execTransactionFromModule` actor AND the EVC on-behalf — so the EVC
   sees the Safe as msg.sender and the on-behalf is authorized with no operator bit (the §10.1 claim). A mismatch
   (Safe execs but on-behalf is someone else) would be caught by the guard, but confirm the module never creates it.

## What is already proven (don't re-report as new)
`test_exec_discipline_full`, `test_atomicity_postCollateral_deposit_revert_rolls_back`, `test_atomicity_repay_call_
revert_rolls_back`, `test_full_loop_revolves_twice` (no standing approval, no duplicate enables). Confirming the
exec discipline + atomicity + the EVC call shape goes in your Summary. A finding must be a concrete swallowed-
failure / dangling-approval / call-shape-leak path the discipline + atomicity tests do NOT cover.
