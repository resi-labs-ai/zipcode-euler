# Mission 1 — The unbounded-borrow adversary (can a borrow escape all three bounds?)

MISSION: 1 — the F1 aggregate cap + kill-switch (I-2), the EVK health gate (I-3), the guard pin (I-5)

## Who you are
You attack the three independent controls that bound the only borrow in the fleet — F1 `borrowCap`, the EVK
account-status health check, and the `FarmUtilityBorrowGuard` account-identity pin. The borrowed USDC is shared
depositor cash. Your goal: ONE borrow path that escapes all three — over the cap, over-LTV, or on an on-behalf
account that isn't the Safe.

## The claims you are attacking
- **I-2 (F1 aggregate cap + kill-switch).** `borrow` checks `IBorrowing(borrowVault).debtOf(juniorTrancheEngine) +
  usdcAmount > borrowCap` → `CapExceeded` (`:233`), BEFORE the borrow execs. `borrowCap == 0` ⇒ every borrow
  reverts (kill-switch). `setBorrowCap` is `onlyOwner` (`:138`), NOT operator. **Find a path where a borrow
  exceeds the aggregate cap** — a `debtOf` read that's stale/low, an overflow in `debtOf + usdcAmount` (0.8
  checked → reverts, safe?), a path that reaches the borrow exec (`:239`) without the `:233` check, or a cap that
  the operator can raise (it can't — owner-only).
- **I-3 (EVK health gate).** The borrow is `IEVC.call(borrowVault, juniorTrancheEngine, 0, borrow(usdcAmount,
  juniorTrancheEngine))` (`:239-244`); the EVC's end-of-call account-status check enforces health via the router →
  LP oracle (over-LTV reverts `E_AccountLiquidity`; a stale/missing mark reverts). **Confirm the module relies on
  the EVC check and can't suppress it** — e.g. a call shape that skips the status check, or a borrow that doesn't
  go through `IEVC.call` (so no end-of-call health check fires).
- **I-5 (the guard pin).** The `FarmUtilityBorrowGuard` (on `OP_BORROW`) rejects any on-behalf != the engine Safe.
  The module always passes `onBehalfOfAccount = juniorTrancheEngine` (`:243`). **Confirm the on-behalf is the
  pinned Safe** so the guard passes for the module and blocks a third party (the third-party rejection is proven
  on the real market — see the `farmutilityborrowguard` drill). Find any path where the on-behalf could be
  operator-influenced.

## Specific attacks to run
1. **The cap check vs the borrow exec ordering.** `:233` (cap check) runs BEFORE `:236` (`enableController`) and
   `:239` (the borrow). Confirm there's no path to the borrow exec that skips `:233`. `debtOf` is read live from
   the vault (`:233`, not cached). Could an attacker make `debtOf` read low (so `debtOf + amount` passes) while
   real debt is higher? `debtOf` is the EVK's accounting — trust it (EVK base); attack only whether the module
   reads the right account (`juniorTrancheEngine`).
2. **The on-behalf / receiver pins.** `borrow`'s EVC call (`:239-244`): `onBehalfOfAccount = juniorTrancheEngine`
   (`:243`), and the inner `borrow(usdcAmount, juniorTrancheEngine)` has `receiver = juniorTrancheEngine` (`:243`).
   So the borrowed USDC goes to the Safe, on the Safe's account. Confirm NEITHER is operator-influenced (the
   operator supplies only `usdcAmount`). A receiver != Safe would send borrowed depositor cash to an attacker.
3. **Kill-switch.** Confirm `borrowCap == 0` reverts every borrow at `:233` (`debtOf + usdcAmount > 0` is true for
   any `usdcAmount ≥ 1`). Confirm only the Timelock can set the cap (`:138`) and the operator cannot raise it.
4. **The three bounds are independent.** Confirm escaping ONE doesn't escape the others: even if the cap were
   mis-set high, the EVK health check still gates over-LTV; even if health passed, the guard still requires the
   Safe on-behalf. Find a single path that defeats all three at once (the real risk).
