# Mission 4 — The privilege & wiring adversary

MISSION: 4 — access gates, the clone init-lock, and the build-phase wiring blast radius

## Who you are
You attack the access-control and configuration surface: the operator gate, the SEC-14 mastercopy lock, the
SEC-15 owner-recheck, the `setUp` guards, the F1 `setBorrowCap`, and the 6 wiring setters — several of which
re-point WHAT is borrowed against (`borrowVault`/`escrowVault`/`lpToken`). Your goal: ONE path where a
non-operator loops, the mastercopy is initializable, or a re-point DRAINS / lets a borrow escape its bounds.

## The claims you are attacking
- **Operator gate.** All 4 loop steps are `onlyOperator` (`:218`, `:230`, `:251`, `:272`); the modifier checks
  `msg.sender == operator` (`:126-129`). **Find a non-operator path to a loop step.** Confirm the operator CANNOT
  call `setBorrowCap` or any wiring setter or the inherited `setAvatar`/`setTarget`.
- **SEC-14 / SEC-15 / `setUp` guards.** Mastercopy locked (`MastercopyInitLock`); clone `setUp` once
  (`initializer`, `:86`). Eight addresses non-zero (`:102-106`), `owner != operator` (`:107`), THEN
  `avatar = target = juniorTrancheEngine` (`:110-111`), THEN `_transferOwnership` (`:122`). **Find a decode/order
  bug** in the 8-address + uint decode (`:87-100`).
- **The wiring blast radius (6 setters + the cap).** `setJuniorTrancheEngine` (`:149`, syncs avatar/target — the
  borrower-of-record invariant), `setOperator` (SEC-15, `:158`), `setEvc`, `setBorrowVault`, `setEscrowVault`,
  `setLpToken`, `setUsdc` (`:166-198`), plus `setBorrowCap` (`:138`). **Pressure-test the borrow-target re-points:**
  `setBorrowVault` changes what's borrowed FROM; `setEscrowVault`/`setLpToken` change what's collateralized;
  `setUsdc` changes the borrow/repay asset. A mismatched set (e.g. `escrowVault` for token A, `lpToken` for token
  B) could desync collateral from what the borrow vault prices. Which re-point DRAINS vs redirects/fails-closed?
  All `onlyOwner` (Timelock).

## Specific attacks to run
1. **The avatar/target sync = borrower-of-record.** `setJuniorTrancheEngine` (`:149-155`) sets `juniorTrancheEngine
   = avatar = target` together — this IS the borrower-of-record invariant (the EVC on-behalf, the receiver/owner of
   every borrow/repay/withdraw all read `juniorTrancheEngine`). The inherited `setAvatar`/`setTarget` (zodiac-core,
   onlyOwner) can set them INDEPENDENTLY. Since the loop uses the explicit `juniorTrancheEngine` slot for the EVC
   on-behalf (NOT the avatar) — confirm whether a desynced avatar/target matters here (the EVC calls go through
   `execAndReturnData` which uses avatar/target to drive the Safe, but the ON-BEHALF arg is the explicit
   `juniorTrancheEngine`). Work whether a desync (avatar = Safe A, juniorTrancheEngine = Safe B) makes the module
   exec THROUGH Safe A while borrowing on-behalf of Safe B — which the guard would reject (on-behalf != installed
   Safe) → fail-closed. Confirm it's fail-closed, not a redirect.
2. **`setBorrowVault`/`setEscrowVault`/`setLpToken` desync.** Re-pointing these changes the borrow/collateral
   targets. A Timelock that points `borrowVault` at a vault holding more depositor cash, or `escrowVault`/`lpToken`
   at a mis-priced collateral, could change the leverage profile. But the F1 cap + EVK health still gate the
   borrow on the NEW vault. Is there a re-point combination that DRAINS (borrowed cash to the Safe is still the
   Safe's — but the COLLATERAL could be under-priced, letting an over-borrow that the EVK health check should
   still block)? Frame: the EVK enforces health on whatever vault is wired; a hostile vault is an X-1-style
   wiring/trust residual (owner-only), not a new escape in THIS code. Rate.
3. **`setBorrowCap` is owner-only.** Confirm the operator can't raise the F1 cap (`:138`, `onlyOwner`) — only the
   Timelock. A cap raise is a governance act; the operator sizing within it is X-1.
4. **Operator → owner escalation.** Confirm the operator can't reach any setter or become owner. SEC-15 (`:160`)
   keeps operator != owner.
