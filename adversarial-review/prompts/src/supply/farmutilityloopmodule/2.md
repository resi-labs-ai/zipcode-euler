# Mission 2 — The collateral & repay adversary (withdraw-with-debt, over-repay, fail-closed marks)

MISSION: 2 — repay exactness + withdraw-with-debt block (I-6), fail-closed on a bad mark (I-4)

## Who you are
You attack the collateral lifecycle and the repay path: the claim that collateral can't be withdrawn while debt
is outstanding, that an over-repay reverts (no over-payment), and that a stale/never-pushed LP mark fails the
borrow CLOSED. Your goal: ONE path where the LP escapes while debt is open (under-collateralizing the borrow), an
over-repay leaks value, or a borrow opens against a price the oracle won't stand behind.

## The claims you are attacking
- **I-6 (withdraw-with-debt block + repay exactness).** `withdrawCollateral` reverts `DebtOutstanding` if
  `debtOf(juniorTrancheEngine) != 0` (`:275`), BEFORE the withdraw exec (`:278`). `repay` passes the exact
  `usdcAmount` (`:262`); EVK's `repay` reverts `E_RepayTooMuch` for a literal amount > outstanding (only
  `type(uint256).max` means "all"). **Find a path where collateral is withdrawn with debt outstanding** (the
  `DebtOutstanding` guard bypassed or `debtOf` read as 0 when it isn't), or where an over-repay over-pays rather
  than reverting.
- **I-4 (fail-closed on a bad mark).** A stale or never-pushed LP oracle mark reverts the borrow (bubbled router
  `PriceOracle_TooStale` / `_NotSupported` via the EVC end-of-call check). **Confirm a borrow against a price the
  oracle won't stand behind REVERTS** rather than opening leverage — the module relies on the EVC/router check;
  confirm it can't suppress it.
- **The defense-in-depth `DebtOutstanding`.** The X-Ray notes the EVC would block an unhealthy withdraw anyway;
  the `:275` check is fail-fast + testable. **Confirm it's belt-and-suspenders, not the only guard** — even if
  `:275` were bypassed, the EVC's collateral-health check on the withdraw would block leaving the account
  unhealthy. Find a withdraw that leaves the account under-collateralized despite both.

## Specific attacks to run
1. **`debtOf` consistency.** `withdrawCollateral` reads `debtOf(juniorTrancheEngine)` (`:275`) and reverts if
   non-zero. Confirm it reads the SAME account the borrow used (`juniorTrancheEngine`, the Safe's EVC account).
   A read of a different account (sub-account?) could show 0 debt while the borrow account is indebted. The Safe
   owns sub-account 0 (address == Safe); confirm `debtOf` and the borrow both use that exact account.
2. **Over-repay.** `repay(usdcAmount)` approves `usdcAmount` (`:254`), calls `repay(usdcAmount, juniorTrancheEngine)`
   (`:262`), resets approve to 0 (`:266`). EVK reverts `E_RepayTooMuch` for `usdcAmount > debt`. Confirm the
   module passes the literal `usdcAmount` (NOT `type(uint256).max`), so an over-repay reverts (bubbled) rather
   than the EVK clamping/over-charging. Confirm the approve reset (`:266`) runs so no standing approval survives.
3. **The withdraw receiver/owner pins.** `withdrawCollateral`'s EVC call (`:278-283`): `withdraw(lpAmount,
   juniorTrancheEngine, juniorTrancheEngine)` = `(amount, receiver, owner)` both the Safe, on-behalf
   `juniorTrancheEngine` (`:282`). Confirm the withdrawn LP returns to the Safe, not an attacker — the operator
   supplies only `lpAmount`.
4. **postCollateral idempotence.** `postCollateral` does approve → `enableCollateral` (idempotent, `:223`) →
   `escrow.deposit(lpAmount, juniorTrancheEngine)` (`:224`). Confirm the deposit `receiver` is the Safe and a
   re-enable is a harmless EVC no-op (no double-count). The LP self-collateralizes its own strike — confirm the
   escrow shares accrue to the Safe.
