# Mission 2 — The Baal-power adversary (is ragequit / mintShares reachable?)

MISSION: 2 — no `ragequit`, no `mintShares`, zero-shares forever (I-2/I-3)

## Who you are
You attack the dangerous-power surface: the Gate holds Baal `manager(2)` — which CAN mint arbitrary Loot, mint
shares, and (in general) enable ragequit — but the design claims it NEVER calls anything beyond
`mintLoot`/`burnLoot`. Your goal: ONE reachable path to `ragequit`, `mintShares`, or any Baal call that breaks
zero-shares or enables an in-kind drain of the basket.

## The claims you are attacking
- **I-3 (no `ragequit` reachable).** Exit is the CoW buy-and-burn rail (mission-irrelevant here — it's ratified);
  the Gate wires NO `ragequit`. **Read EVERY Baal call in the contract and confirm the set is exactly
  `{mintLoot, burnLoot, lootToken, avatar}`** (`:88-89`, `:118-119`, `:173`, `:204`). Find any other Baal call —
  a `ragequit`, `mintShares`, `setShamans`, `setAdminConfig`, an arbitrary `execAsBaal` — reachable from any
  entrypoint or setter. There should be none.
- **I-2 (zero-shares forever).** The Gate never calls `mintShares`, so `baal.totalShares() == 0` always. **Confirm
  no path mints shares** (only Loot). A non-zero `totalShares` would change Baal's governance/ragequit math.
- **The `manager(2)` power is bounded by non-use, not by a guard.** The danger of a Baal shaman (arbitrary loot
  mint, ragequit) is real — but the Gate's safety is that it NEVER exposes those calls. **Find a path that turns
  the held power into a reachable call** — e.g. a generic-exec/passthrough (there is none — the Gate has no
  arbitrary-call surface), or a setter that could point an entrypoint at a dangerous Baal function.

## Specific attacks to run
1. **Enumerate the Baal calls.** Grep every `baal.` / `IBaal(...)` call site: `lootToken()` (`:88`, `:118`),
   `avatar()` (`:89`, `:119`), `mintLoot` (`:173`), `burnLoot` (`:204`). Confirm that's the COMPLETE set —
   no `ragequit`, `mintShares`, `burnShares`, `setShamans`, `execAsBaal`, or any generic Baal exec. The Gate has
   no `bytes`-payload entrypoint, no delegatecall, no arbitrary-target call — so there's no way to smuggle a
   dangerous Baal call through a parameter. Confirm.
2. **The `mintLoot` recipient is the Gate itself.** `baal.mintLoot(_one(address(this)), _one(shares))` (`:173`) —
   Loot mints to the GATE, not to a user or operator-chosen address. Confirm `address(this)` (the Gate) is the
   only Loot recipient, so the Gate remains the sole Loot custodian (depositors get szipUSD, never raw Loot —
   which is what keeps ragequit out of users' hands).
3. **Could a deposit/burn amount be abused to mint excess Loot?** `mintLoot(_one(this), _one(shares))` mints
   exactly `shares` Loot (== the szipUSD minted). `creditFreeValue`-style unbounded mint? No — `shares` is
   oracle-derived from a real deposited asset (`:166`). Confirm there's no path to mint Loot without a matching
   deposited asset + szipUSD (which would over-collateralize the Gate's Loot vs szipUSD — actually a conservation
   break, mission 1 — but here confirm Loot can't be minted alone).
4. **`setBaal` re-point.** `setBaal` (`:115`) re-points the Baal substrate and re-derives `loot`/`juniorTrancheSafe`.
   Could a Timelock re-point to a malicious Baal that exposes ragequit to the Gate's existing calls? The Gate
   still only calls `mintLoot`/`burnLoot` on the new Baal — so a malicious Baal could misbehave, but that's an
   `onlyOwner` (Timelock) act re-pointing to a hostile dependency (X-2-style). Frame severity: the Gate's call set
   is unchanged; a hostile Baal is a governance/wiring residual, not a new reachable ragequit in THIS code.

gate). Confirming no ragequit/mintShares is reachable goes in your Summary. A finding must be a concrete reachable
