# Mission 3 — The issuance adversary (mispricing, custody residue, fail-open)

MISSION: 3 — NAV-proportional round-down issuance (I-4), zero custody (I-5), fail-closed + TVL cap (I-7/I-8)

## Who you are
You attack the deposit pricing and asset handling: the claim that `depositFor` issues `shares = valueOf · 1e18 /
navEntry` rounded DOWN (favoring the vault, never the depositor), routes the deposited asset straight to the
basket (zero Gate custody), fails CLOSED on a stale/unseeded oracle, and respects the TVL cap. Your goal: ONE
path where a depositor gets MORE shares than the round-down price (value extraction from stayers), where the
deposited asset doesn't reach the basket, or where a stale mark prices a deposit.

## The claims you are attacking
- **I-4 (round-down issuance).** `shares = value * 1e18 / navE` (`:166`), integer division = round DOWN. `value =
  navOracle.valueOf(asset, amount)` (`:163`); `navE = navOracle.navEntry()` (`:162`). **Find an input where a
  depositor receives MORE shares than the exact value/navE** — a rounding-UP path, a `value` that over-counts the
  deposit, or a `navE` read low. Round-down favors the vault (existing holders); rounding up would dilute them.
  `previewDeposit` (`:189`) mirrors the same math — confirm it matches `depositFor` exactly (the X-Ray's parity
  test) so the UI estimate can't mislead into an over-issue.
- **I-5 (zero Gate custody).** `IERC20(asset).safeTransferFrom(msg.sender, juniorTrancheSafe, amount)` (`:170`) —
  the asset goes STRAIGHT to the main Safe (basket), never the Gate. **Find a path where the asset (or any
  residue) stays at the Gate** — a transfer to `address(this)` instead of `juniorTrancheSafe`, a fee-on-transfer
  token leaving a partial balance, or a refund path. `_assertInvariants()` checks `zip.balanceOf(gate) == 0`.
- **I-7 (fail-closed issuance).** `navEntry()` reverts `StalePrice` if a required leg is stale; the X-Ray's SEC-04
  has it revert `RateUnseeded` on an unseeded xALPHA rate. **Confirm a stale/unseeded oracle REVERTS `depositFor`
  (issuance pauses) rather than pricing off a bad mark.** `navOracle.poke()` (`:161`) runs first — confirm it
  can't be gamed to refresh a stale mark to a manipulated value.
- **I-8 (TVL cap).** `if (grossBasketValue() + value > tvlCap) revert TvlCapExceeded()` (`:164`). **Find a path to
  exceed the cap** — a `value` that under-counts so the check passes but more value enters, or a `grossBasketValue`
  read low. Note `previewDeposit` deliberately SKIPS the cap (`:186`) — confirm that's a view-only estimate and
  the cap is enforced at execution.

## Specific attacks to run
1. **Rounding direction.** `value * 1e18 / navE` (`:166`) — confirm integer division rounds toward zero (DOWN).
   Find a `(value, navE)` where the depositor could engineer a favorable rounding (e.g. a tiny deposit that
   rounds to a disproportionate share count, or a `navE` of 1). `ZeroShares` (`:167`) catches a zero-share mint —
   confirm a dust deposit reverts rather than minting 0 Loot + 0 szipUSD (which would be a no-op, benign) or
   minting mismatched amounts.
2. **`valueOf` trust + decimals.** `value = valueOf(asset, amount)` is the oracle's job (18-dp USD). The Gate
   divides by `navE` (also 18-dp). Confirm the units reconcile (value 18-dp / navE 18-dp × 1e18 = shares 18-dp).
   A decimals mismatch would mis-issue by orders of magnitude. Distrusting the oracle's `valueOf` MARK is out of
   scope (oracle authority); attack the Gate's USE of it.
3. **Custody residue on FoT.** If `asset` were fee-on-transfer, `safeTransferFrom(msg.sender, juniorTrancheSafe,
   amount)` (`:170`) sends `amount` but the basket receives less — yet `shares` was computed off `valueOf(asset,
   amount)` (the full amount). The depositor gets shares for value the basket didn't fully receive. The whitelist
   is {zipUSD, xALPHA} — confirm neither is FoT, OR that the issuance is robust to it (it values `amount`, not the
   received delta — so a FoT asset WOULD over-issue; is the whitelist the only defense? Flag if a leg could become
   FoT via `setTokens`).
4. **`poke()` then read ordering.** `poke()` (`:161`) → `navEntry()` (`:162`) → `valueOf` (`:163`). Confirm
   `poke` refreshes the accumulator honestly (it's the oracle's keeper function) and can't be sandwiched to set a
   favorable entry mark within the same tx. `navEntry = max(spot, twap)` (the §3 buyer-conservative entry bracket
   for issuance) — confirm issuance prices at the HIGHER of spot/twap (conservative: fewer shares per dollar).
