# Mission 1 ‚Äî The conservation adversary (can Loot and szipUSD desync?)

MISSION: 1 ‚Äî the two-token conservation `totalSupply() == loot.balanceOf(gate)` (I-1)

## Who you are
You attack the load-bearing accounting identity: every Loot mint/burn is paired with an EQUAL szipUSD mint/burn,
so `szipUSD.totalSupply() == loot.balanceOf(gate)` holds under ANY interleaving. Your goal: ONE path where the
two quantities desync ‚Äî an unpaired mint/burn, an ordering where one lands without the other, or a reentrancy
that interleaves a second op between the paired writes.

## The claims you are attacking
- **I-1 (two-token conservation).** `depositFor` mints paired: `baal.mintLoot(_one(this), _one(shares))` (`:173`)
  then `SzipUSD(shareToken).mint(receiver, shares)` (`:174`) ‚Äî equal `shares`. `burnFor` burns paired:
  `baal.burnLoot(_one(this), _one(amount))` (`:204`) then `SzipUSD.burn(juniorTrancheEngine, amount)` (`:205`) ‚Äî
  equal `amount`. **Find any path where one of the pair lands without the other, or with a different amount** ‚Äî a
  revert between the two writes that commits one (no ‚Äî same tx, atomic), a reentrancy that sneaks a deposit/burn
  between them, or a `shares`/`amount` value that differs between the Loot leg and the szipUSD leg.
- **The transient asymmetry (documented).** The NatSpec (`:30-32`) notes the engine Safe's transient pre-burn
  szipUSD is excluded from the *oracle* denominator (not here), resolved on the next `burnFor`. **Confirm this is
  the ONLY asymmetry** and that it doesn't break the on-chain `totalSupply == loot.balanceOf(gate)` identity (the
  szipUSD the engine holds pre-burn is still in `totalSupply` and still matched by Loot at the Gate ‚Äî confirm).
- **Reentrancy.** Both mutators are `nonReentrant` (`:154`, `:200`). **Confirm the guard wraps the WHOLE paired
  write** so no reentrant `depositFor`/`burnFor` can interleave between the Loot mint and the szipUSD mint. The
  external calls in `depositFor` are `safeTransferFrom` (`:170`, the deposit asset ‚Äî could a malicious ERC-20
  re-enter?), `baal.mintLoot`, and `SzipUSD.mint`. Trace whether any external call lands BETWEEN the two paired
  writes and could re-enter (the `nonReentrant` should block re-entry, but confirm the ordering: the
  `safeTransferFrom` is BEFORE both mints `:170`, so a token hook re-entering hits the guard).

## Specific attacks to run
1. **The paired-write atomicity.** In `depositFor`, the order is: oracle reads ‚Üí `safeTransferFrom` (`:170`) ‚Üí
   `mintLoot` (`:173`) ‚Üí szipUSD `mint` (`:174`). If `SzipUSD.mint` reverts, the whole tx reverts (unwinding the
   `mintLoot`). Confirm no partial commit. Same for `burnFor`: `burnLoot` (`:204`) ‚Üí `SzipUSD.burn` (`:205`) ‚Äî if
   the burn reverts, `burnLoot` unwinds. The X-Ray's `test_burnFor_reverts_when_engine_underfunded` proves the
   under-funded engine case rolls back atomically ‚Äî confirm `burnLoot` doesn't leak when the szipUSD burn fails.
2. **The malicious-asset reentrancy.** `safeTransferFrom(msg.sender, juniorTrancheSafe, amount)` (`:170`) ‚Äî for a
   deposit asset with a transfer hook (zipUSD/xALPHA are the whitelist; are they hook-free?), could the hook
   re-enter `depositFor`? `nonReentrant` blocks it. But confirm the transfer is BEFORE the mints (it is, `:170`),
   so even a re-entrant attempt hits the guard before any unpaired mint. Could a re-entrant call to a DIFFERENT
   mutator (none ‚Äî only deposit/burn, both guarded) interleave?
3. **Amount equality.** `mintLoot` and szipUSD `mint` both use `shares` (`:173-174`); `burnLoot` and `burn` both
   use `amount` (`:204-205`). Confirm there's no path where the Loot amount and szipUSD amount diverge (e.g. a
   fee, a rebase, a rounding applied to one leg only). `shares` is computed once (`:166`); confirm it feeds both
   legs identically.
4. **The `_one` array helper.** `_one(address)`/`_one(uint256)` (`:210-217`) build single-element arrays for the
   Baal batch API. Confirm they build exactly one element with the right value ‚Äî a length/element bug would
   mint/burn a wrong Loot amount, desyncing.
