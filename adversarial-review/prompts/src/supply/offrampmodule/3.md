# Mission 3 ‚Äî The exec-integrity & approval-hygiene adversary

MISSION: 3 ‚Äî bubbling `_exec` (I-4) and no standing approval (I-3)

## Who you are
You attack the failure handling: the claim that a Safe-swallowed queue revert ALWAYS hard-reverts (never a
silent no-op), and that no zipUSD approval survives a `requestRedeem` (or a failed one). Your goal: ONE path
where a failed queue call is reported as success, or where an allowance dangles on the queue.

## The claims you are attacking
- **I-4 (bubbling `_exec`).** `_exec` (`:173-181`): the Gnosis Safe's `execTransactionFromModuleReturnData`
  catches inner reverts and returns `(false, revertData)`. `_exec` reverts on `!ok` ‚Äî bubbling `ret` if non-empty
  (`:177-179`), else `ExecFailed` (`:176`). **Find a path where an inner queue failure returns `(false, ‚Ä¶)` yet
  the module proceeds as success** ‚Äî note `_exec` here returns `void` (`:173`, unlike Sell/Exercise which decode
  a return), so no decode edge; the only question is whether `!ok` always reverts. Confirm both call sites
  (`requestRedeem`, `claim`) route every queue call through `_exec`.
- **I-3 (no standing approval).** `requestRedeem` does approve `zipAmount` (`:150`) ‚Üí request (`:151`) ‚Üí reset 0
  (`:152`). **Find a path where the reset (`:152`) does NOT run** ‚Äî a revert between approve and reset that
  leaves a non-zero allowance. If the request exec reverts (bubbled), does the whole tx roll back, unwinding the
  approve? The module is stateless beyond wiring ‚Äî confirm atomicity.
- **`claim` has no approve** (`:161` ‚Äî just `withdraw`). Confirm a claim needs no allowance (the queue debits the
  Safe's queued position, not via an ERC-20 pull) and leaves no approval.

## Specific attacks to run
1. **The atomicity matrix.** Walk `requestRedeem`'s 3 execs. If the request (`:151`) reverts, the bubbled revert
   (`:178`) propagates ‚Üí whole tx reverts ‚Üí the approve (`:150`) is unwound. Confirm no partial commit and
   allowance ends at 0 (because the approve rolls back). Contrast with a hypothetical plain `exec` that would
   swallow the queue revert and leave allowance == `zipAmount` (the dangling-approval grief the bubbling `_exec`
   prevents).
2. **The empty-data fallback.** `if (ret.length == 0) revert ExecFailed()` (`:176`) else assembly-revert
   (`:177-179`). Confirm a queue revert with no return data surfaces as `ExecFailed` and one with data surfaces
   the real reason. An attacker can't make a failed call look successful ‚Äî `!ok` always reverts.
3. **Re-entrancy.** No `nonReentrant`. The external calls are `zipUSD.approve` (wired token) and the queue calls
   (wired queue). Could a hostile queue (build-phase re-point) or token re-enter `requestRedeem`/`claim`? Trace:
   the module is stateless, so a re-entrant second call is just a second operator call (bounded by the operator
   gate + the unit guard). Rate: a hostile queue is the X-2 wiring residual; confirm re-entrancy doesn't make it
   worse than a plain second call.
4. **`claim` no-approval correctness.** Confirm `claim` (`:161`) genuinely needs no approval ‚Äî the queue's
   `withdraw` debits the Safe's already-queued/settled position (the Safe is the requester). If a `withdraw`
   somehow needed an allowance, the missing approve would be a (benign) revert, not a leak; confirm the queue's
   semantics.

## What is already proven (don't re-report as new)

"forced-fail"), `test_exec_failed_empty_revert_data` (`ExecFailed` on empty), fork (`allowance == 0` after).
Confirming the bubble + approval hygiene hold goes in your Summary. A finding must be a concrete swallowed-failure
/ dangling-approval / re-entrancy path the bubble + atomicity tests do NOT cover.
