# Mission 3 — The malformed-batch adversary (the all-or-nothing CRE push + forwarder identity)

MISSION: 3 — batch push write path: all-or-nothing, deviation band, SEC-01 monotonic, forwarder identity

## Who you are
You attack the WRITE side: the CRE batch leg push. Only the off-chain legs (`alphaUSD`, HYDX/USD) are pushed;
they arrive as a reportType-7 batch that must be ALL-OR-NOTHING (one bad entry reverts all) and pass a stack
of guards. The Forwarder is the trusted writer, but a malformed, replayed, or out-of-band-deviation batch
must be rejected fail-closed. Your goal: find ONE malformed batch entry that lands in `legCache` and later
mis-prices the basket, or a partial write that leaves the cache half-updated.

## The claims you are attacking
- **I-4 (batch push all-or-nothing + guarded).** `_processReport` must enforce: reportType `NAV_LEG=7`,
  length-match, future-ts rejection, per-leg invalid-leg / zero-price / deviation-band / strictly-newer-ts
  (SEC-01); **one bad entry reverts the WHOLE batch.** Walk each guard and confirm there is no early-commit
  that writes some legs before a later entry reverts (a partial write would leave a stale/new mix).
- **I-13 (forwarder identity + renounce-freeze).** `onReport` is forwarder-gated with a workflow-id gate;
  renouncing ownership freezes the setters; the correct workflow-id still writes. Confirm the identity gate
  (not just `msg.sender == forwarder` but the workflow-id) and that renounce-freeze doesn't lock out the
  Forwarder writer.
- **Accumulator ordering.** The TWAP accumulator is advanced FIRST (old spot booked before new prices apply)
  on a write. Confirm a new leg price cannot contaminate the TWAP's old interval (cross-ref Mission 1).

## Specific attacks to run
1. **Partial-write atomicity.** The batch decodes N legs and writes each to `legCache`. If entry k reverts
   (bad deviation / stale ts / zero price), are entries 0..k-1 rolled back? Solidity reverts unwind state, so
   confirm there is no external call or storage commit between entries that escapes the revert. Find any
   path that commits a subset.
2. **Deviation band vs SEC-01 gap.** The deviation band catches a price that moved too far; the SEC-01
   strictly-newer `ts` catches a same-price REPLAY the deviation band cannot (deviation == 0 passes). Confirm
   BOTH run per-leg — a same-`ts` or backdated replay of a legitimate-magnitude price must still revert
   `StaleReport`. Pin `ts <= cached → revert` (strictly-newer required).
3. **Per-leg guard completeness.** For each leg entry: zero-price → revert; invalid leg id → revert;
   `ts > now` → `FutureTimestamp`; deviation beyond band → `DeviationExceeded`; non-strictly-newer →
   `StaleReport`. Find a guard that's checked at batch level but not per-leg (or vice-versa) such that a bad
   entry slips through.
4. **Length / type confusion.** `LengthMismatch` (prices vs legs vs timestamps arrays), `InvalidReportType`
   (non-7). Can a mismatched-length batch decode without reverting and write misaligned (price[i] to leg[j])?
   Diff the decode against `ZipcodeOracleRegistry` / `SzAlphaRateOracle`.
5. **Workflow-id identity.** Confirm the forwarder gate checks the workflow-id, not just the sender — a
   different workflow on the same Forwarder must not write. Confirm renounce freezes setters but NOT the
   Forwarder push (else the oracle goes stale and bricks issuance).

## What is already proven (don't re-report as new)
`test_push_*` (8 guards), `_deviation_*` (3), `test_SEC01_nav_backdated_replay_reverts`,
`test_batch_atomicity_one_bad_entry_reverts_all` (I-4); `test_push_non_forwarder_reverts`,
`test_forwarder_immutability_and_identity` (I-13). Confirming the all-or-nothing guarded push holds goes in
the Summary. A finding must be a concrete partial-write, a per-leg guard gap, or a misaligned decode these
tests don't model. "CRE could push a bad-but-valid mark" is the accepted CRE-trust residual — classify INFO,
don't over-rate.
