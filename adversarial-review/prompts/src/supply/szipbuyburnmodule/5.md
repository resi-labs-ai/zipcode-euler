# Mission 5 — The privilege & wiring adversary

MISSION: 5 — access gates, the clone init-lock, the single-bid state, and the build-phase wiring blast radius (X-1)

## Who you are
You attack the access-control, live-bid state, and configuration surface: the operator/owner gates (note
`cancelBid` is operator-OR-owner), the SEC-14 mastercopy lock, the SEC-15 owner-recheck, the `setUp` guards, the
governed params, and the 6 wiring setters + `setCoverageGate`. Your goal: ONE path where a non-authorized caller
posts/cancels, the mastercopy is initializable, the order `receiver` is redirected off the Safe, or a re-point
DRAINS rather than redirects future bids.

## The claims you are attacking
- **Access gates.** `postBid` is `onlyOperator` (`:325`); `cancelBid` is operator OR owner (`:390` — owner is the
  emergency retract); governed params (`setDiscountBps` `:228`, `setBuybackCap` `:234`) + all wiring setters are
  `onlyOwner`. **Find a non-authorized path to post/cancel**, or an operator reaching an `onlyOwner` setter.
  Confirm the operator CANNOT change `dBps`/`buybackCap`/the wiring (only the Timelock can).
- **SEC-14 / SEC-15 / `setUp` guards.** Mastercopy locked (`MastercopyInitLock`); clone `setUp` once
  (`initializer`, `:170`). Seven addresses non-zero (`:186-189`), `owner != operator` (`:190`), `dBps` strictly
  in `(0, 10_000)` (`:191`), THEN the live VaultRelayer + domainSeparator reads off settlement (`:208-209`), THEN
  `_transferOwnership` (`:211`). `coverageGate` MAY be 0 (`:205`, gate off). **Find a decode/order bug** in the
  10-field decode (`:171-184`).
- **X-1 / wiring blast radius — the receiver redirect is the sharpest.** 6 wiring setters
  (`setJuniorTrancheEngine`/`setNavOracle`/`setSzipUSD`/`setUsdc`/`setSettlement`/`setVaultRelayer`) + `setOperator`
  + `setCoverageGate`. **Pressure-test:** `juniorTrancheEngine` is the order `receiver` AND the uid `owner` — a
  re-point (`:250`) changes WHERE bought szipUSD lands. `setVaultRelayer`/`setSettlement` change the approve
  spender / presign target. `setNavOracle` changes the pricing primitive. `setCoverageGate(0)` disables the
  outflow gate. Which of these DRAIN vs redirect-future-bids? Note: re-pointing `juniorTrancheEngine` does NOT
  re-derive `vaultRelayer`/`domainSeparator` (those are settlement-derived) — confirm a settlement re-point that
  leaves a stale `domainSeparator`/`vaultRelayer` is a fail-closed mismatch, not a drain.
- **Single-resting-bid state hygiene.** `currentUid`/`currentSellAmount` (`:124-126`) are the only mutable state
  beyond wiring/params. **Find a state-corruption path** — a post that sets state then a wiring re-point that
  strands a live bid (e.g. re-point `settlement` while a presign is live on the OLD settlement → the cancel
  targets the NEW settlement, leaving the OLD presign live + allowance dangling).

## Specific attacks to run
1. **The stranded-bid wiring race (the sharpest state question).** Post a bid (presign on settlement A, allowance
   on vaultRelayer A). Timelock calls `setSettlement(B)` (`:278`) — but does NOT re-derive `vaultRelayer`/
   `domainSeparator` (those were set in `setUp`; `setVaultRelayer` is separate). Now `cancelBid` flips presign on
   B (`:400`) + approve 0 on the cached `vaultRelayer` — leaving the OLD presign on A LIVE and possibly a dangling
   allowance. Is that a drain (the old bid can still fill from the Safe) or a recoverable ops mistake? Work the
   sequence; rate it (X-1, owner-only, but a real state-vs-wiring hazard worth flagging).
2. **Receiver redirect.** `setJuniorTrancheEngine(attacker)` (`:250`) would make a FUTURE bid's `receiver` =
   attacker (bought szipUSD to the attacker). But it's `onlyOwner` (Timelock); the X-Ray accepts the Timelock-
   redirect residual (same Timelock governs all). Confirm the operator CANNOT reach it, and that an EXISTING live
   bid's uid (already signed with the old receiver) isn't retroactively changed. Frame severity: governance
   redirect of future bids (accepted) vs a drain (would need the operator, which can't).
3. **`setCoverageGate(0)` consequence.** Confirm only the Timelock can disable the outflow gate (`:293`), and the
   operator cannot — disabling it un-gates `postBid` (exits can post while undercovered). Note the consequence
   (the single highest-value gate re-point), rate X-1.
4. **Operator → owner + cancel-by-owner.** Confirm `cancelBid`'s operator-OR-owner (`:390`) is intentional (owner
   = emergency retract) and can't be abused (owner cancelling is benign — it only retracts a bid). Confirm SEC-15
   (`:244`) keeps operator != owner.
