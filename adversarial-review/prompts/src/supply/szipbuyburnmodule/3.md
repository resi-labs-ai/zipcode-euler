# Mission 3 ŌĆö The signed-order adversary (uid integrity, field hardening, atomic post/cancel)

MISSION: 3 ŌĆö canonical GPv2 uid (I-7) + the ┬¦4 order hardening + atomic post / idempotent cancel (I-8)

## Who you are
You attack what the protocol actually SIGNS: the claim that the on-chain uid matches CoW's canonical encoding
exactly, that only the 3 validated fields vary (every other GPv2 field a pinned constant, `APP_DATA == 0`
forbidding hooks), and that posting is atomic (a presign revert rolls back the approve) while cancel is
idempotent. Your goal: ONE path where an unvalidated/operator-influenced field enters the signed order, where the
uid diverges from CoW's encoding (so the protocol signs an order it didn't validate), or where a post leaves a
dangling presign/allowance.

## The claims you are attacking
- **I-7 (canonical GPv2 uid).** `_orderUid` (`:421`) builds `structHash = keccak256(abi.encode(TYPE_HASH,
  usdc, szipUSD, juniorTrancheEngine, sellAmount, buyAmount, uint256(validTo), APP_DATA, 0, KIND_BUY, true,
  BALANCE_ERC20, BALANCE_ERC20))` (`:422-438`), `digest = keccak256(0x1901 ++ domainSeparator ++ structHash)`
  (`:439`), `uid = digest ++ juniorTrancheEngine(20) ++ validTo(4)` = 56 bytes (`:440`). **Find a divergence from
  CoW's canonical encoding** ŌĆö a wrong field ORDER, a wrong type (`uint256(validTo)` in the struct vs `uint32` in
  the uid tail), a `TYPE_HASH` mismatch, a missing/extra field, or a domainSeparator that's wrong for the chain.
  The KAT test pins the 56-byte uid against an out-of-band `cast` vector ŌĆö a transcription bug fails it; find
  what the KAT's single vector might miss (a field that varies only with a different `validTo`/amount).
- **The ┬¦4 hardening (only 3 fields vary).** The signed struct uses constants for `KIND_BUY` (`:72`), `APP_DATA
  == bytes32(0)` (`:84`, NO hooks), `feeAmount = 0` (`:432`), `partiallyFillable = true` (`:435`), `BALANCE_ERC20`
  ├Ś2 (`:436-437`), and the wired `usdc`/`szipUSD`/`juniorTrancheEngine`. Only `sellAmount`/`buyAmount`/`validTo`
  come from the operator. **Find an operator input that perturbs a constant**, or a field that is both operator-
  supplied AND unvalidated (the ┬¦4 invariant forbids this). Confirm the `receiver`/`owner` is the pinned
  `juniorTrancheEngine` (so the bought szipUSD can only land in the Safe).
- **`APP_DATA == 0` forbids hooks ŌĆö and is deliberate.** The docstring (`:75-83`) REJECTS adding a CoW pre/post-
  interaction hook to re-check coverage at fill (the undercovered-fill window, mission 4). **Confirm `APP_DATA`
  is the literal `bytes32(0)` in BOTH the validated struct and the signed uid** ŌĆö a non-zero/unconstrained
  appData could attach hooks or partner-fees the validation never saw.
- **I-8 (atomic post / idempotent cancel).** `_postBid` does two `exec`s: `usdc.approve(vaultRelayer, sellAmount)`
  (`:371`) then `setPreSignature(uid, true)` (`:374`) ŌĆö one tx, so a presign revert rolls back the approve.
  `_cancelBid` (`:396`) flips presign false + allowance 0, no-ops when `currentUid.length == 0`. **Find a path
  where the approve survives without the presign (or vice-versa), or where cancel isn't idempotent.**

## Specific attacks to run
1. **Field-by-field uid diff.** Compare `_orderUid`'s 12 struct fields (`:424-437`) against CoW's GPv2Order
   field order. Confirm `validTo` is `uint256(order.validTo)` in the struct (`:430`) but `uint32` (4 bytes) in the
   uid tail (`:440`) ŌĆö that matches CoW's `uid = digest ++ owner ++ validTo(uint32)`. Confirm `owner` in the uid
   is `juniorTrancheEngine` (the receiver), matching the KAT's "owner==engine". Find any field the KAT vector
   wouldn't distinguish.
2. **The validated struct == the signed struct.** `_postBid` validates `order` then calls `_orderUid(order)`
   (`:368`) and signs THAT uid (`:374`). Confirm the SAME `order` struct is validated and hashed ŌĆö no field is
   re-derived or defaulted between validation and hashing (so "the module signs exactly what it validated").
3. **Atomicity matrix.** If `setPreSignature` (`:374`) reverts, the whole tx reverts ŌåÆ the approve (`:371`) is
   unwound ŌåÆ no dangling VaultRelayer allowance. Confirm. If the approve reverts, no presign is set. Then
   `currentUid`/`currentSellAmount` are set AFTER both execs (`:381-382`) ŌĆö so a revert leaves no live-bid state.
   Confirm `BidAlreadyLive` (`:333`) blocks a second post while one is live, and a cancel/partial-fill re-arms.
4. **Cancel idempotence + the dangling-allowance risk.** `_cancelBid` (`:396`) returns early if no uid (`:398`),
   else flips presign false (`:400`) + approve 0 (`:406`) + deletes state. Confirm a cancel with no live bid is a
   true no-op (no revert, no exec). Confirm cancel resets the VaultRelayer allowance to 0 (so a cancelled bid
   leaves no spendable approval).
