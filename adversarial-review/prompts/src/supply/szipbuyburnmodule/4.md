# Mission 4 — The coverage-gate & two-doors adversary

MISSION: 4 — the coverage path-lock outflow gate (I-5) + the undercovered-fill window + two-doors-one-guard-set (I-9)

## Who you are
You attack the outflow gating and the dual entry: the claim that `postBid` is blocked while `!covered()`, that
the (deliberately un-gated-at-fill) undercovered-fill window is genuinely safe, and that the CRE `POST_BID`
report door enforces the IDENTICAL `_postBid` validations as the operator door. Your goal: ONE path where a bid
posts while coverage is below the floor, where an undercovered FILL actually breaches coverage, or where the two
doors diverge so the report path skips a guard.

## The claims you are attacking
- **I-5 (coverage path-lock outflow gate).** `_postBid` reads `gate = coverageGate` and reverts `Undercovered`
  if `gate != address(0) && !ICoverageGate(gate).covered()` (`:340-341`). The gate is `DurationFreezeModule.
  covered()`. Gate-off (0) = ungated (M1 legacy). **Find a path where `postBid` posts while `!covered()`** — the
  gate read mis-evaluated, or a coverage state that reads `covered()` true when the floor is breached (cross-check
  with `DurationFreezeModule` mission 2's `covered()` direction).
- **The undercovered-fill window (documented accepted design).** `postBid` is coverage-gated, but a FILL after
  coverage drifts is NOT (CoW hooks forbidden by `APP_DATA == 0`). The docstring (`:75-83`) argues this is safe:
  the USDC the bid spends is free-side value `coverageValue()` ALREADY EXCLUDES (it's engine-Safe free-side, not
  committed sidecar value), and the window is bounded by NAV `maxAge`. **Stress-test the argument:** is the USDC
  the bid spends genuinely free-side (excluded from `coverageValue`)? If a fill could consume value the coverage
  numerator counts, the undercovered fill WOULD breach the floor. Confirm the bid's `sellToken=USDC` from the
  engine Safe is free-side, not committed/path-locked.
- **I-9 (two doors, one guard set).** The operator door `postBid` (`:325`) and the CRE door `_processReport`
  `POST_BID` (`:312-314`) BOTH call `_postBid` — so every validation (single-bid, zero, cap, coverage, validTo,
  freshness fence, discount) runs identically. **Find a guard that one door enforces but the other skips** — e.g.
  a check in the `postBid` wrapper (`:325-327`) that ISN'T in `_postBid`, or a report-path decode (`:311-313`)
  that constructs the `GPv2OrderInput` differently. The X-Ray's `test_CTR01_report_postBid_equals_operator_postBid`
  proves byte-identical uid/sellAmount — find an input where they'd differ.

## Specific attacks to run
1. **The gate evaluation.** `gate != address(0) && !covered()` (`:341`). Confirm a gate of 0 (off) skips the
   check (legacy, intended) and a wired gate calls `DurationFreezeModule.covered()`. Find any path where
   `covered()` returns true while the freeze floor is actually breached (this is the seam to DurationFreezeModule
   — a `covered()` that's too lenient lets exits drain coverage). Cross-reference the double-squeeze (DurationFreeze
   mission 2): `covered()` should be conservative.
2. **Free-side vs committed USDC (the fill-window crux).** The bid spends engine-Safe USDC. `coverageValue() =
   committedValue() (sidecar) + pathLockedLpEquity() (main-only LP)`. Confirm the engine-Safe free USDC the bid
   spends is NOT in either term — so a fill consuming it can't lower `coverageValue` below the floor. If ANY of
   the spent USDC were committed/path-locked, the undercovered fill would breach. Verify the partition holds
   (the X-Ray's SEC-02 partition `gross - coverage == free`).
3. **Two-doors decode equivalence.** `_processReport` (`:310`) decodes `(uint8 t, bytes payload)`, then for
   `POST_BID` decodes `(sellAmount, buyAmount, validTo)` (`:313`) and calls `_postBid(GPv2OrderInput(...))`
   (`:314`). The operator path calls `_postBid(order)` from a `calldata` struct (`:326`). Confirm both reach
   `_postBid(memory)` with identical semantics — no validation in the `external postBid` wrapper that the report
   path bypasses, and the report decode can't produce a malformed struct the operator path couldn't.
4. **The report door's forwarder gate.** The CRE door is reached via the inherited `CloneReportReceiver.onReport`
   (forwarder-gated). Confirm `_processReport` (`:310`) is `internal override` and only reachable through the
   gated `onReport` (so an un-forwarded report can't reach `_postBid`). The receiver half is the `clonereportreceiver`
   drill — here, confirm THIS module's `_processReport` doesn't add an un-gated path. Unknown types revert
   `UnsupportedReportType` (`:318`).

## What is already proven (don't re-report as new)
`test_postBid_coverage_gate` (gate false → `Undercovered`, true → posts, off → ungated),
`test_CTR01_report_postBid_equals_operator_postBid` (byte-identical uid/sellAmount), the CTR-01 receiver block
(see the `clonereportreceiver` drill). Confirming the gate blocks undercovered posts + the two doors share one
guard set goes in your Summary. A finding must be a concrete undercovered-post / breaching-fill / door-divergence
path the coverage-gate + two-doors tests do NOT cover. The undercovered-fill window and `APP_DATA==0` are
ACCEPTED design points — note them as such unless you actually break the free-side-USDC argument.
