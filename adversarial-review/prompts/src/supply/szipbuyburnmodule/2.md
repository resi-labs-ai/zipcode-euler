# Mission 2 — The stale-mark adversary (can a resting bid fill against an old NAV?)

MISSION: 2 — the SEC-13 leg-anchored NAV-freshness fence (I-3) + the freshness gate (I-6)

## Who you are
You attack the time dimension of exit safety: a bid is signed now but RESTS until `validTo`, and CoW hooks are
forbidden (`APP_DATA == 0`), so there is no fill-time re-check. The SEC-13 fence anchors `validTo` to the oldest
required leg's timestamp so a fill lands against a mark at most `maxAge` old (not `2·maxAge`). Your goal: ONE path
where a resting bid can fill against a NAV mark older than `maxAge`, or where the fence's arithmetic underflows /
fails OPEN at an edge.

## The claims you are attacking
- **I-3 (SEC-13 leg-anchored fence).** `_postBid` reads `anchor = oldestRequiredLegTs()` (`:351`) and reverts
  `ValidToBeyondNavFreshness` if `order.validTo > anchor + maxAge()` (`:352`). The rationale (`:344-350`): the
  legs feeding `navExit` may already be up to `maxAge` old at post-time (`fresh()` only requires age ≤ `maxAge`),
  so anchoring the ceiling to POST-time (`now + maxAge`) allowed a worst-case fill-time mark age of `2·maxAge`.
  Anchoring to the oldest-leg timestamp caps it at `maxAge`. **Find a path where a fill lands against a mark older
  than `maxAge`** — a `validTo` that passes the fence but allows a `2·maxAge` fill, or an `anchor` that reads
  newer than the actual oldest leg.
- **The edge cases must fail CLOSED (no underflow).** The fence is pure ADDITION (`anchor + maxAge`, never
  subtraction), so the oldest-leg-age == maxAge / unset-leg (anchor 0) / maxAge == 0 edges fail closed via this
  fence PLUS the `validTo > block.timestamp` check (`:342`). **Find an edge where the fence underflows, reverts
  spuriously (grief), or — worse — admits a `validTo` it shouldn't** (e.g. `anchor == 0` for an unset leg making
  `anchor + maxAge == maxAge`, a small number, so `validTo > maxAge` reverts for any realistic future timestamp —
  confirm that's fail-CLOSED, not open).
- **I-6 (freshness gate).** `if (!fresh()) revert StaleNav()` (`:353`) — a stale / never-pushed NAV reverts.
  `fresh()` requires both required legs within `maxAge`. **Confirm a never-pushed or stale NAV reverts at post**
  (the X-Ray's `test_freshness_gate_stale_reverts` / `_never_pushed_reverts`), and that the fence (`:352`) binds
  BEFORE `BadValidTo` (`:342`) when `anchor + maxAge < now + MAX_BID_TTL` (the X-Ray's `test_freshness_fence_binds_
  before_BadValidTo`).

## Specific attacks to run
1. **The 2·maxAge window — try to reopen it.** At post-time, suppose the oldest required leg was pushed at `T`
   and `now = T + maxAge` (still `fresh()`). The fence requires `validTo ≤ T + maxAge = now`. But `validTo >
   now` is also required (`:342`). So a leg already at the freshness limit forces `validTo` into an empty window →
   no bid posts (fail-closed). **Confirm there's no `validTo` that satisfies both `validTo > now` AND `validTo ≤
   anchor + maxAge` when the leg is near `maxAge` old** — i.e. the fence genuinely caps fill-age at `maxAge`. The
   X-Ray's `test_SEC13_two_maxAge_window_closed` / `_fill_age_capped_at_maxAge`.
2. **`oldestRequiredLegTs` trust.** `anchor` is read from the oracle (`:351`). If the oracle reports an anchor
   NEWER than the true oldest leg, the fence over-permits (a later `validTo` allowed → older fill). Is that an
   oracle-trust issue (X-2-style, out of scope) or does the module mis-read it? Attack only the module's USE — it
   reads `oldestRequiredLegTs()` directly and adds `maxAge`. Confirm no off-by-one in the comparison (`>` vs `>=`
   at `:352`) that admits an extra-old fill.
3. **maxAge == 0 / anchor == 0 edges.** If `maxAge() == 0`, the fence is `validTo > anchor` → with `validTo >
   now` required, any future `validTo > anchor` (anchor likely ≤ now) → reverts `ValidToBeyondNavFreshness`
   (fail-closed: no bid). If `anchor == 0` (unset leg), `validTo > maxAge` reverts for any realistic timestamp
   (fail-closed). Confirm both edges REVERT (no bid) rather than admitting an unbounded-age fill. The X-Ray's
   `_edge_legs_at_freshness_limit_fail_closed`.
4. **Ordering with the price read.** The fence (`:352`) and `fresh()` (`:353`) run BEFORE `navExit()` is read
   (`:356`). Confirm a stale NAV can't be priced — the `fresh()` gate blocks it. And confirm `navExit()` itself
   (which does NOT revert on staleness) is only reached after `fresh()` passes, so a stale mark never prices a bid.
