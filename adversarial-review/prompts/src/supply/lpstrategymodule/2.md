# Mission 2 — The redirect & pin-integrity adversary (can LP/legs/shares leave the basket?)

MISSION: 2 — deposit `to` + balance-read pin (I-1), live-read token legs (I-5), stake/unstake credit-the-Safe (I-6)

## Who you are
You attack the destination integrity across the LP lifecycle: the claim that every minted LP share, every
returned leg, every staked/unstaked position, and every balance read is hard-pinned to `juniorTrancheEngine`,
and that the approval/deposit token legs can never drift from the vault's actual `token0`/`token1`. Your goal:
ONE path where LP/legs/shares reach a non-Safe address, where a token leg is mispinned, or where stake/unstake
credits the wrong account.

## The claims you are attacking
- **I-1 (deposit `to` + all balance reads pinned).** `addLiquidity` builds `IICHIVault.deposit(deposit0,
  deposit1, juniorTrancheEngine)` (`:232`) — arg 3 (the `to`) is the set-once `juniorTrancheEngine`.
  `removeLiquidity` builds `IICHIVault.withdraw(shares, juniorTrancheEngine)` (`:270`). The views
  `stakedBalance`/`lpBalance` read `juniorTrancheEngine` (`:297`, `:302`). **Find any path where the operator can
  set the `to`/recipient to a non-Safe address.** The operator's ONLY inputs are scalar amounts — confirm none
  reach an address arg.
- **I-5 (live-read token legs).** `setUp` reads `token0 = IICHIVault(ichiVault).token0()` and `token1 =
  ...token1()` (`:113-114`), asserts non-zero (`:115`). These are the approve targets for the deposit legs
  (`:225`, `:228`). **Find a path where `token0`/`token1` (the approve targets) drift from the vault's ACTUAL
  legs** — e.g. `setToken0`/`setToken1` (`:161`, `:168`) overriding them to a mismatched token, or a `setIchiVault`
  (`:147`) that re-points the vault WITHOUT re-deriving the legs (it doesn't re-derive — is that a desync risk?).
- **I-6 (stake/unstake credit the Safe).** `stake` approves the gauge then `IGauge.deposit(lpAmount)` (`:280-282`);
  `unstake` does `IGauge.withdraw(lpAmount)` (`:290`). No `to`/recipient arg — the gauge credits/debits the Safe
  because the Safe is the `exec` msg.sender. **Confirm there is no tokenId/position state and the gauge can't
  credit a non-Safe account.** No EVC/borrow leg anywhere.

## Specific attacks to run
1. **Trace operator args to calldata.** `deposit0`/`deposit1`/`minShares`/`lpAmount`/`shares`/`minAmount0/1` are
   all scalars → they reach the `deposit`/`withdraw`/gauge amount args and the slippage floors, NEVER the `to`
   arg or an approve spender. The vault/gauge calls use TYPED `encodeCall` (`:232`, `:270`, `:281`, `:290`) — a
   field-order regression fails to compile. Confirm.
2. **`setIchiVault` without leg re-derive.** `setIchiVault` (`:147`) re-points `ichiVault` but does NOT re-read
   `token0`/`token1` (unlike ExerciseModule's `setOHYDX` which re-derives `paymentToken`). So after a vault
   re-point, the approve legs (`:225`, `:228`) could approve the OLD tokens to the NEW vault. Is that a drain
   (approve of a token the new vault doesn't want → the deposit reverts, fail-closed) or a real mismatch? Rate:
   it's an `onlyOwner` (Timelock) act (X-2), and the worst case is a bricked deposit, not a leak — but confirm,
   and note whether the Timelock must also call `setToken0`/`setToken1` to keep them in sync.
3. **The approve targets/spenders.** The deposit approves call `token0.approve(ichiVault, …)` / `token1.approve(
   ichiVault, …)` (`:225`, `:228`); the stake approve calls `ichiVault.approve(gauge, lpAmount)` (`:280`).
   Confirm the spender is the wired vault/gauge and the token is the wired leg — never operator-chosen.
4. **No custody.** Confirm the module holds NO tokens/LP itself (`:31` — the Safe holds everything) and writes NO
   storage in any mutating path. A balance that accrued to the MODULE instead of the Safe would be a redirect;
   confirm every `to`/balance-read is `juniorTrancheEngine`.
