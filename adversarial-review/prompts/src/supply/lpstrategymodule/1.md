# Mission 1 — The coverage path-lock adversary (can floor-backing LP be dissolved?)

MISSION: 1 — the coverage gate seam to `DurationFreezeModule` (X-1)

## Who you are
You attack the distinctive, load-bearing property of this module: `removeLiquidity` may ONLY liquefy LP that is
*excess* over the coverage floor. Dissolution converts path-locked LP into exitable legs, so it must respect the
same floor `DurationFreezeModule` gates `release`/exit by. Your goal: ONE path where a breaching burn (one that
would drop coverage below the liability floor) dissolves anyway — the gate bypassed, mis-read, mis-sized, or
desynced from the freeze module's LP-in-place accounting.

## The claims you are attacking
- **X-1 (coverage path-lock).** `removeLiquidity` reads `address gate = coverageGate` (`:268`) and reverts
  `Undercovered` unless `gate == address(0) || ICoverageGate(gate).lpBurnKeepsCovered(shares)` (`:269`). The gate
  is `DurationFreezeModule`. **Find a path where a burn that the freeze module would consider floor-breaching
  passes the gate** — e.g. the `shares` passed to `lpBurnKeepsCovered` (`:269`) not matching the `shares` actually
  withdrawn (`:270`), a units/decimals mismatch between what the gate expects and what the vault burns, or the
  gate reading a different LP valuation than the actual dissolution returns.
- **Gate == 0 is "OFF" (legacy).** `coverageGate == 0` ⇒ ungated (`:269`, the M1 pre-wiring state). **Confirm
  this is intentional and that a prod deployment wires the gate** (a gate left at 0 means `removeLiquidity` is
  unconstrained — flag as a config concern, not a code vuln, unless you find a path that SILENTLY zeroes it).
- **The seam must line up with the freeze module.** Read `DurationFreezeModule.lpBurnKeepsCovered` — confirm it
  values the same LP, in the same units, against the same floor that `LpStrategyModule.removeLiquidity` is about
  to dissolve. **Find a desync**: the gate says "covered" for a `shares` value that the vault `withdraw` (`:270`)
  decomposes into MORE underlying value than the gate accounted for (so coverage actually drops below floor), or
  a rounding direction that favors dissolution.

## Specific attacks to run
1. **`shares` consistency.** `:269` passes `shares` to `lpBurnKeepsCovered`; `:270` passes the SAME `shares` to
   `IICHIVault.withdraw(shares, juniorTrancheEngine)`. Confirm they are identical (no transform between the gate
   check and the withdraw) — a gate checking `shares` while the withdraw burns a different amount would be the
   core break. Confirm the gate check runs BEFORE the withdraw (it does, `:269` before `:270`) so a failing gate
   prevents the burn.
2. **Read the freeze module's `lpBurnKeepsCovered`.** Open `DurationFreezeModule.sol`. Does it value `lpShares`
   by the same vault accounting (`pathLockedLpEquity` / the vault's per-share value) that `withdraw` will return?
   Find a state where `lpBurnKeepsCovered(shares)` returns true but burning `shares` actually drops coverage below
   the floor — e.g. a stale oracle mark on the LP, a per-share value that moved between the gate read and the
   withdraw, or the gate counting LP that the withdraw doesn't actually remove.
3. **Re-entrancy across the seam.** The gate read (`:269`) and the withdraw (`:270`) are separate external calls
   (gate is a staticcall view; withdraw mutates). Could the withdraw (or a hostile vault) re-enter
   `removeLiquidity` between a passing gate check and the actual burn, to dissolve more than the gate allowed?
   The module is stateless — but the FREEZE module's accounting may change mid-tx. Trace whether two interleaved
   `removeLiquidity` calls can each pass the gate on the same excess and together breach the floor.
4. **Gate OFF silent path.** Confirm the ONLY way `coverageGate` becomes 0 is the explicit `onlyOwner`
   `setCoverageGate(0)` (`:176`) or a `setUp` with `coverageGate_ == 0` (`:118`, allowed). There is no path for
   the operator to zero it. A Timelock turning the gate off is the X-2 residual (governance), not an attack.

## What is already proven (don't re-report as new)
`test_removeLiquidity_coverage_gate` — all three states: gate `false` → `Undercovered`, gate `true` → dissolves,
gate OFF (0) → ungated. Confirming the seam lines up with `DurationFreezeModule` goes in your Summary. A finding
must be a concrete gate-bypass / shares-desync / valuation-desync path the 3-state test does NOT cover — most
valuably a mismatch between what `lpBurnKeepsCovered` accounts for and what `withdraw` actually dissolves.
