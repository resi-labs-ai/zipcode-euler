# Mission 2 — The malformed-push adversary (write path: gate, type, value, monotonicity)

MISSION: 2 — write path: forwarder gate, reportType pin, value guards, SEC-01 strictly-newer

## Who you are
You attack the WRITE side: the only mutator of the cache. The Forwarder is the trusted writer (ratified
base) — but a malformed payload, a wrong reportType, a replayed/out-of-order mark, or a non-Forwarder caller
must all be rejected fail-closed. Your goal: find ONE malformed or hostile push that lands in the cache and
later prices a borrow.

## The claims you are attacking
- **I-2 (forwarder-only writes).** `onReport` is gated in `ReceiverTemplate` (`InvalidSender` for any
  non-Forwarder). **Confirm THIS contract adds no second write path** that bypasses the gate (a public
  setter that writes the cache, an inherited mutator left exposed).
- **I-3 (reportType pinned).** `_processReport` must reject any report whose type != `LP_MARK = 7`
  (`InvalidReportType`). **Confirm the type check precedes the decode/write**, and that decoding a
  wrong-shaped payload as the expected struct cannot silently succeed.
- **I-4 (write-value guards).** `_writePrice` must reject: `mark == 0` → `PriceOracle_InvalidAnswer`;
  `mark > uint208.max` → `PriceOracle_Overflow`; `ts > now` → `FutureTimestamp`. **Walk each guard and
  confirm the bound is exact** (e.g. `uint208.max` not `uint256`, a future-ts a single second ahead is
  rejected).
- **I-1 (SEC-01 strictly-newer write).** A mark whose `ts` is NOT strictly newer than the cached one must
  revert `StaleReport`. This blocks replay, out-of-order delivery, AND a stale-but-higher mark from
  over-crediting collateral (the price-only deviation/value checks cannot catch a same-`ts` replay). **Pin
  the comparison: `ts <= cache.timestamp → revert`** (strictly-greater required), and confirm the first
  write (`cache.timestamp == 0`) is allowed.

## Specific attacks to run
1. **Strictly-newer off-by-one.** Is the guard `ts > cache.timestamp` (correct) or `ts >= cache.timestamp`
   (would allow an equal-ts replay over-crediting)? Confirm a backdated and an equal-ts mark both revert,
   and a `+1` second mark succeeds.
2. **Value-guard bounds.** `mark == 0`, `mark == uint208.max` (boundary — allowed?), `mark == uint208.max+1`
   (must overflow-revert), `ts == now` (allowed), `ts == now+1` (future, rejected). Find any boundary that
   admits a value the read path then mis-uses.
3. **Type / decode confusion.** Push a report typed `REVALUATION = 3` (the registry's type) or any non-7
   type — confirm `InvalidReportType` before any decode. Then: can a payload of the wrong length/shape decode
   into `(mark, ts)` without reverting (abi.decode tolerance)? Diff the decode against `ZipcodeOracleRegistry`
   / `SzAlphaRateOracle`'s `_processReport`.
4. **Second write path.** Grep the full surface for any function that writes `cache` other than the
   Forwarder-gated `onReport→_processReport→_writePrice` chain. The X-Ray claims the Forwarder is the ONLY
   writer (no controller-seed, unlike the registry) — verify that delta holds and no seed path leaked in.
5. **Monotonic vs deviation.** This oracle has no price-deviation band (unlike `SzipNavOracle`); the only
   over-credit defense on the write is the strictly-newer `ts`. Confirm that's sufficient given the read-side
   staleness window — can a single bad-but-fresh mark with a valid newer `ts` over-value within the window
   before the next push? (That's the accepted CRE-trust residual — classify it, don't over-rate it.)

## What is already proven (don't re-report as new)
`test_onReport_nonForwarder_reverts` (I-2), `test_processReport_wrongType_reverts` (rejects `REVALUATION=3`),
`test_writePrice_zeroMark_reverts` / `_overflowMark_reverts` / `_futureTs_reverts` (I-4), and the four SEC-01
tests `test_SEC01_lp_firstWrite_succeeds` / `_backdated_mark_reverts` / `_equalTs_reverts` /
`_strictlyNewer_succeeds` (I-1). Confirming these hold goes in the Summary. A finding must be a concrete
malformed-push acceptance or a leaked second writer these tests don't cover.
