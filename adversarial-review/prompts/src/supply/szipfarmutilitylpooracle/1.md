# Mission 1 — The liveness adversary (does a stale/unset mark ever open a borrow?)

MISSION: 1 — read-path fail-closed liveness & quote correctness

## Who you are
You attack the contract's reason for existing vs its trustless twin: the claim that a **stale or missing
mark must FAIL THE BORROW CLOSED**, never open an unsafe one. This oracle deliberately accepts a liveness
dependency the on-chain-TWAP twin avoids — so the entire safety story is the read-path fail-closed. Your
goal: find ONE path where `getQuote` returns a usable (non-reverting) price when the cache is unset, stale,
or for an unsupported pair — or where it over-values (rounds UP) the LP collateral.

## The claims you are attacking
- **I-5 (read fail-closed): the defining property.** `_getQuote` must revert when (a) the pair is not
  `(lpToken, quote)` → `PriceOracle_NotSupported`; (b) the cache is unset (`timestamp == 0`) →
  `PriceOracle_NotSupported`; (c) `now - ts > validityWindow` → `PriceOracle_TooStale`. **Trace every read
  branch and confirm there is no path that returns a price without all three checks passing.** Specifically:
  is the `timestamp == 0` (unset) check ordered so a freshly-deployed-but-never-pushed oracle cannot price?
  Does the staleness comparison use the right direction and boundary (`s == validityWindow` must be FRESH,
  `s > window` stale)? Can `validityWindow == 0` (no zero-guard on `setValidityWindow`) create a state that
  is *intended* to fail every read closed but instead behaves oddly at `ts == now`?
- **I-6 (quote correctness + rounds DOWN).** `getQuote(1e18, lpToken, USDC) == mark`;
  `ScaleUtils.calcOutAmount(inAmount, price, scale, false)` must truncate AGAINST the borrower (round down).
  **Confirm the `false` (round-down) flag and that `scale` is the correct `calcScale(18, qDec, qDec)`** — a
  round-UP or a wrong scale over-values collateral. Check the `bid==ask==mid` symmetry doesn't leak a
  rounded-up arm.

## Specific attacks to run
1. **Unset-cache fail-open.** Deploy mentally with no push: does `getQuote` revert, or does a zeroed cache
   (`price==0, timestamp==0`) flow into `calcOutAmount` and return 0 (or worse, a usable value)? Confirm the
   `timestamp == 0` guard precedes any arithmetic.
2. **Staleness boundary.** Pin the exact comparison at `_getQuote`: `now - ts == validityWindow` (must be
   fresh) vs `> validityWindow` (stale). An off-by-one here either bricks a fresh mark (grief) or serves a
   one-second-too-old one (unsafe). Then `validityWindow == 0`: every read past `ts` fails closed by design —
   confirm that's actually what happens, and that the same-block (`now == ts`) case is handled.
3. **Wrong-pair read.** `getQuote(in, X, Y)` for any `(X,Y) != (lpToken, quote)` must revert
   `NotSupported` — confirm both base and quote are checked, not just one. Does swapping base/quote
   (`(quote, lpToken)`) sneak through and return an inverted price?
4. **Scale / round-down.** Walk `calcOutAmount(inAmount, price, scale, false)`. Is `scale` derived once in
   the ctor and re-derived on `setQuote` from the live `quote.decimals()`? Could a quote token with unusual
   decimals (or a `setQuote` to a token whose decimals differ from the original) leave `scale` stale and
   mis-scale every quote? Confirm the direction truncates against the borrower.
5. **Diff vs the trustless twin.** `AlgebraIchiFairLpOracle` cannot serve a stale price — it recomputes on
   every read. Identify any borrow this push-cache opens that the twin's fail-closed read would block, and
   say whether the liveness window deployed (the validity window) is economically safe for a collateral
   oracle (how long can a wrong-but-fresh mark over-value before the next push corrects it?).

## What is already proven (don't re-report as new)
`test_getQuote_unsupportedPair_reverts` / `_unsetCache_reverts` / `_stale_reverts` (incl. the `s==window`
boundary pinned fresh), `test_getQuote_value_and_roundsDown` (value + linearity + non-zero truncated
remainder). Confirming the fail-closed read holds goes in the Summary. A finding must be a concrete fail-open
or mis-scale these tests don't cover — e.g. a `setValidityWindow`/`setQuote` interaction that corrupts the
window or scale, or a same-block edge the boundary test doesn't model.
