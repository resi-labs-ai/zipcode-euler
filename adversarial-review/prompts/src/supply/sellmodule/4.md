# Mission 4 — The privilege & wiring adversary (can a re-point or the operator escalate to redirect/theft?)

MISSION: 4 — access gates, the clone init-lock, and the build-phase wiring blast radius

## Who you are
You attack the access-control and configuration surface: the operator gate, the SEC-14 mastercopy lock, the
SEC-15 owner-recheck, the `setUp` guards, and the 7 `onlyOwner` wiring setters. Your goal: ONE path where a
non-operator drives a swap, the mastercopy is initializable, an owner-set re-point DRAINS (not merely
redirects/griefs), or the operator escalates from "sizing scalars" to "choosing the destination."

## The claims you are attacking
- **Operator gate (`NotOperator`).** All 3 swap legs are `onlyOperator` (`:216`, `:231`, `:253`); the modifier
  checks `msg.sender == operator` (`:198-201`). **Find a non-operator path to `_swap`** (it's `private` — only
  the 3 gated entrypoints reach it). Confirm the operator CANNOT call the owner-only setters
  (`setMaxSellHydx`, the 7 wiring setters, the inherited `setAvatar`/`setTarget`).
- **SEC-14 (mastercopy init-lock).** The mastercopy is locked in its constructor (`MastercopyInitLock`); a clone
  is `setUp` once under `initializer` (`:89`). **Confirm `setUp` on the mastercopy reverts and a clone can be
  initialized exactly once** (no re-init, no `setUp` after deploy). Diff the lock against
  `contracts/src/supply/szipUSD/MastercopyInitLock.sol` — does the mixin actually fire in this contract's
  constructor chain?
- **SEC-15 (owner-recheck on `setOperator`).** `setOperator` (`:155`) rejects `operator_ == owner` (`:157`,
  `OwnerIsOperator`) — the same invariant `setUp` enforces (`:110`). **Confirm the owner can never become the
  operator (or vice-versa) via any setter ordering**, so the Timelock and the hot CRE key stay distinct.
- **`setUp` guards.** All 8 decoded addresses non-zero (`:104-109`), `owner != operator` (`:110`),
  `maxSellHydx > 0` (`:111`), ALL validated BEFORE any use, THEN `_transferOwnership(owner)` (`:127`). **Find a
  decode/validation-order bug** — a zero address used before its check, a missing field, a decode that
  mis-aligns the 8 addresses + uint256 (`:100-102`).
- **X-1 / build-phase wiring blast radius.** 7 `onlyOwner` (Timelock) setters: `setJuniorTrancheEngine` (`:146`,
  also syncs `avatar`/`target`), `setOperator`, `setSwapRouter`, `setHydx`, `setUsdc`, `setZipUSD`, `setXAlpha`,
  plus inherited `setAvatar`/`setTarget`. Each rejects `address(0)` and emits `WiringSet`. **This is the key
  pressure-test:** re-pointing `juniorTrancheEngine` moves the swap `recipient` AND the avatar/target Safe — so
  a Timelock that re-points it redirects all future output. Is that a DRAIN or a documented governance redirect?
  Re-pointing `swapRouter` points the module at an attacker router (could re-enter — see mission 2). Frame each
  re-point: does it let the Timelock STEAL existing value, or only redirect FUTURE swaps (the X-2 residual,
  closed by the pre-prod re-freeze)?

## Specific attacks to run
1. **The avatar/target sync.** `setJuniorTrancheEngine` (`:146-152`) sets `juniorTrancheEngine = avatar =
   target` together. Confirm they CAN'T desync (a path that sets one but not the others would let the module
   drive Safe A while recipient is Safe B). The inherited `setAvatar`/`setTarget` (zodiac-core `Module`) can set
   them INDEPENDENTLY and are `onlyOwner` — does that let the owner desync avatar/target from
   `juniorTrancheEngine`? The X-Ray notes (`:203-206`) these are not hard-locked (vendored deps stay pristine).
   Rate: a desync is an owner(Timelock) act, and `recipient` reads `juniorTrancheEngine` (not `avatar`) — so a
   desynced avatar drives a different Safe than the recipient. Is that exploitable or just a governance footgun?
2. **`setSwapRouter` → hostile router.** A re-pointed `swapRouter` becomes both the approve spender (`:292`) and
   the swap target (`:293`). A hostile router could take the approve and not deliver, or re-enter. Is this a
   DRAIN (the approve is exact-`amountIn` and reset to 0 — bounded to one swap's input) or bounded grief?
   Compare to the §10.1 promise: the router is wired/trusted post-freeze; pre-freeze re-point is X-2.
3. **Operator → owner escalation.** Confirm there is no path for the `operator` (hot key) to call any
   `onlyOwner` function or to become `owner`. The only operator entrypoints are the 3 swaps. The owner is the
   Timelock set at `setUp` and changed only via the inherited `Ownable` transfer (`onlyOwner`).
4. **The §10.1 residual (X-1) — don't over-rate.** The operator sizing `(amountIn, minOut, deadline)` within the
   pins + the cap is the ACCEPTED operator-sizing trust. Do NOT report it as a vuln. Only escalate if you find a
   re-point or gate path that turns the bounded sizing into a redirect-to-attacker or a drain.
