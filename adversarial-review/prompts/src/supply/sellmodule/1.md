# Mission 1 — The redirect adversary (can the swap output leave the basket, or the pair be swapped out?)

MISSION: 1 — recipient + pair pin (the §10.1 boundary): destination integrity

## Who you are
You attack the single load-bearing claim of a swap-leg module: the operator supplies only scalars, and the
swap output can land NOWHERE but the engine Safe, on NO pair but the per-entrypoint hard-pinned one. Your
goal: ONE path where the swap `recipient`, `tokenIn`/`tokenOut`, `deployer`, `limitSqrtPrice`, or `value`
becomes operator-influenced — i.e. output to a non-Safe address, an arbitrary pair, a non-zero-`value` send,
or a generic call/exec passthrough that the §10.1 boundary forbids.

## The claims you are attacking
- **I-1 (recipient + pair hard-pinned).** Every `_swap` builds the `ExactInputSingleParams` with
  `recipient: juniorTrancheEngine` (`:302`), `deployer: address(0)` (`:301`), `limitSqrtPrice: 0` (`:306`),
  and `tokenIn`/`tokenOut` passed by the entrypoint — `sellHydx` pins `(hydx, usdc)` (`:220`), `buyXAlpha`
  pins `(zipUSD, xAlpha)` (`:234`), `sellXAlpha` pins `(xAlpha, zipUSD)` (`:256`). **Find any path where the
  operator can set `recipient` to a non-Safe address, swap an arbitrary `(tokenIn, tokenOut)`, set a non-zero
  `deployer`/`limitSqrtPrice`, or otherwise reach a pair the three entrypoints don't pin.** The operator's
  ONLY inputs are `(amountIn, minOut, deadline)` — confirm none of them can perturb a pinned field.
- **§10.1 boundary (no passthrough / value==0 / no delegatecall).** `_exec` (`:266`) drives the Safe via
  `execAndReturnData(to, 0, data, Operation.Call)` — `value` is the literal `0` (`:267`), `Operation.Call`
  (never `DelegateCall`), and `to` is only ever `tokenIn` (the approve) or `swapRouter` (the swap). **Find any
  path where `to`, the `value`, or the `Operation` becomes attacker-influenced, or where the module exposes a
  generic exec/call passthrough** (it shouldn't — the only `_exec` callers are the 3 approves + the
  `exactInputSingle` inside `_swap`).
- **The recipient is the SET-ONCE `juniorTrancheEngine`, not a parameter.** `recipient: juniorTrancheEngine`
  reads the storage slot, re-pointable only by `onlyOwner` (`setJuniorTrancheEngine:146`, which also syncs
  `avatar`/`target`). Confirm there is NO function-argument recipient anywhere and the operator cannot reach
  the setter (that owner-gated re-point is mission 4's territory — here, prove the operator path has no
  recipient knob at all).

## Specific attacks to run
1. **Trace every argument the operator controls to the exec calldata.** `(amountIn, minOut, deadline)` flow
   into `ExactInputSingleParams.amountIn` / `.amountOutMinimum` / `.deadline` — confirm NONE of them reach
   `recipient`/`tokenIn`/`tokenOut`/`deployer`/`limitSqrtPrice`/`to`/`value`. Look for any aliasing,
   mis-ordered struct field, or `encodeWithSelector`-vs-`encodeCall` mismatch that could let a scalar land in
   a pinned field. (Note `_swap` uses TYPED `encodeCall` for the swap `:295` — a field-order regression would
   fail to compile — but the two approves use `encodeWithSelector` `:292,:311`; confirm those targets/spenders
   are pinned.)
2. **The approve targets.** The two `approve` `_exec`s (`:292`, `:311`) call `tokenIn.approve(swapRouter, …)`.
   Confirm `tokenIn` is the pinned per-entrypoint token (not operator-chosen) and the spender is the wired
   `swapRouter` — an approve to an attacker spender would be a drain. Confirm the spender is the
   locally-cached `router = swapRouter` (`:291`) read once, so it can't be swapped mid-sequence.
3. **Can a non-pinned pair be reached?** The three entrypoints hard-code their `(tokenIn, tokenOut)`. Confirm
   there is no fourth path, no fallback, no `setUp`-time pair-selection that the operator influences, and that
   `buyXAlpha`/`sellXAlpha` are exact reverses on the SAME wired pair (zipUSD↔xAlpha) — so the operator can't,
   say, sell HYDX into the POL pair or route USDC out of the basket.
4. **`value == 0` always.** Confirm no `exec` path sends native value (`:267` literal 0) — a non-zero value
   would let the module drain the Safe's ETH. There is no `payable` entrypoint and no `msg.value` use; confirm.
