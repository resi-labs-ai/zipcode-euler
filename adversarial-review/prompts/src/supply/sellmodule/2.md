# Mission 2 — The dump adversary (can one sell exceed the cap and crater HYDX?)

MISSION: 2 — the `maxSellHydx` size cap (the distinctive control)

## Who you are
You attack the one control unique to this module: the per-call HYDX size ceiling. `minOut` bounds PRICE
(slippage), never SIZE — so without `maxSellHydx` a compromised operator could dump the whole HYDX basket in
one tx (`minOut = 1`) and crater HYDX, which the protocol is long via veHYDX + the LP. Your goal: ONE path
where a single `sellHydx` moves more than `maxSellHydx`, where the cap fails to bind, or where the cap is
applied to the wrong leg / can be set to a value that bricks or neuters it.

## The claims you are attacking
- **I-2 (`maxSellHydx` size cap).** `sellHydx` reverts `ExceedsMaxSell` when `amountIn > maxSellHydx` (`:219`),
  checked BEFORE `_swap`. **Find a path where `amountIn > maxSellHydx` still executes** — an integer edge, a
  re-entrant second sell that aggregates past the cap, a way to call `_swap(hydx, usdc, …)` that bypasses the
  `:219` guard (note: only `sellHydx` reaches the `(hydx, usdc)` pair, and `_swap` is `private`). Is the cap a
  per-CALL ceiling only (no running accumulator — confirmed at `:62`, "NOT a running accumulator")? If so, is
  the per-epoch throughput genuinely off-chain (8-B11/8-B12), and does that leave any on-chain dump vector the
  cap doesn't bound (e.g. N sequential at-cap sells in one block)?
- **The cap binds the RIGHT leg.** `buyXAlpha`/`sellXAlpha` are deliberately NOT capped (`:243-246` — different
  token, our own POL, bounded upstream). Confirm the cap is on the HYDX sell ONLY and that the uncapped legs
  cannot be coerced into selling HYDX (they pin zipUSD↔xAlpha — cross-check with mission 1's pair pins). A cap
  on the wrong leg, or an uncapped leg that can reach HYDX, defeats the control.
- **I-4 / `setMaxSellHydx` (`:133`).** `onlyOwner` (Timelock, not the operator), reverts `ZeroAmount` on a zero
  cap. `setUp` also asserts `maxSellHydx_ > 0` (`:111`). **Find a value the cap can be set to that bricks
  `sellHydx` (a zero cap → every sell reverts) or neuters it (a `type(uint256).max` cap → no effective
  ceiling).** Is the zero-guard the only bound? A Timelock-set huge cap is governance's call (note severity),
  but confirm the operator cannot reach the setter.

## Specific attacks to run
1. **The boundary.** Confirm `amountIn == maxSellHydx` is ALLOWED (`>`, not `>=`, at `:219`) and
   `amountIn == maxSellHydx + 1` reverts. A strict-vs-non-strict flip would be an off-by-one in the wrong
   direction (it isn't dangerous either way here, but confirm the intent matches the test).
2. **Aggregation / reentrancy.** There is NO `nonReentrant` on `sellHydx`. The cap is per-call. Can a single
   external call result in two `sellHydx` executions that together exceed the cap? Trace `_swap`'s 3 `exec`s —
   the only external calls are `tokenIn.approve` (our wired token) and `swapRouter.exactInputSingle` (the wired
   router). Could a malicious `swapRouter` (build-phase re-point) or a hostile token re-enter `sellHydx`? Rate
   the realism: both are wired/trusted post-freeze; a re-entrant dump requires a hostile router, which is the
   X-2 wiring residual — but confirm CEI/exec ordering doesn't make it worse than a plain second operator call.
3. **The uncapped legs can't reach HYDX.** Confirm `buyXAlpha`/`sellXAlpha` cannot be made to sell HYDX (they
   call `_swap(zipUSD, xAlpha, …)` / `_swap(xAlpha, zipUSD, …)` — no HYDX). If an operator could route HYDX
   through an uncapped leg, the cap is moot. (This overlaps mission 1's pair pin — here, frame it as "does the
   cap have a hole via an uncapped sibling leg?")
4. **Cap vs the real pool.** The default is `300_000e18 ≈ ~3% slippage ≈ ~$10k` (`:57-58`). This is an
   economic parameter, not a safety invariant — do NOT report "the cap is too high/low" as a vuln (it's
   Timelock-tunable to track pool depth). Only flag if the cap can be made non-binding through a CODE path.

## What is already proven (don't re-report as new)

`test_sellXAlpha_not_capped_by_maxSellHydx`, `test_setMaxSellHydx_owner_resizes` / `_only_owner` /
`_rejects_zero`, `_rejects_zero_maxSellHydx` (setUp). Confirming the cap binds the HYDX leg and only it goes in
your Summary. A finding must be a concrete bypass / wrong-leg / brick path the cap matrix does NOT cover — most
valuably an aggregation or uncapped-leg hole, not a restatement of "the operator can sell up to the cap" (that
is the bounded X-1 residual).
