# Mission 3 — The config adversary (scaleUp derivation, capacity bound, ctor & setGate)

MISSION: 3 — scaleUp derivation, capacity-bounded mint, ctor guards, setGate auth, previews

## Who you are
You attack the configuration and bounding surface: the derived `scaleUp`, the capacity-as-the-only-mint-bound,
the constructor guards, the deployer-gated `setGate`, and the previews. A mis-derived scale or a wrong
capacity-overflow boundary mis-mints at the front door without any malicious callout. Your goal: find a
config/derivation path that mints the wrong amount, Panics instead of reverting cleanly, or leaves the module
in a fail-open wiring state.

## The claims you are attacking
- **I-13 (`scaleUp` derived from `decimals()`, not hard-coded).** `scaleUp = 10**(zipDec - usdcDec)` (= 1e12
  for 18-dp zipUSD / 6-dp USDC), derived in the ctor from the tokens' OWN `decimals()`. The ctor reverts
  `DecimalsTooFew` if USDC were the finer unit. **Confirm a regression to a hard-coded `1e12` would break
  non-default pairs** (e.g. 18/18→1, 8/6→100, 18/0→1e18) and that `previewDeposit`/`deposit` both use the
  derived scale. A wrong scale mints the wrong zipUSD amount per USDC.
- **I-8 (capacity is the mint bound; over-cap rolls back).** Ungranted / bounded / overflow capacity all
  revert `E_CapacityReached` (NOT a `Panic`), state restored. **Confirm the `uint128` overflow boundary
  reverts capacity, not Panic** — the module must not pre-multiply into an arithmetic overflow before the
  `ESynth` capacity check.
- **I-9 / I-10 (setGate auth + ctor guards).** `setGate` is deployer-only, zero-guarded, re-settable
  (build-phase), grants NO standing allowance (D1). The ctor zero-guards all 4 args + `DecimalsTooFew`.
  **Confirm `setGate` cannot be called by a non-deployer, cannot be set to zero, and a re-point grants no
  allowance.**
- **I-11 / I-12 (zero-amount guard + previews).** `deposit(0)`/`zap(0)` revert; `previewDeposit`
  (gate-independent), `previewZap` (`NotWired` if gate unset), and previews match realized.

## Specific attacks to run
1. **scaleUp at non-default decimals.** Mentally construct the module with 18/18, 8/6, 18/0 token pairs.
   Confirm `scaleUp` derives to 1, 100, 1e18 respectively, that `deposit` mints `usdcIn * scaleUp`, and that
   `previewDeposit` matches. Find any place a literal `1e12` is assumed (which would mis-mint every
   non-default pair). Confirm `DecimalsTooFew` reverts when USDC is finer than zipUSD (negative exponent).
2. **Capacity overflow boundary.** Drive `usdcIn` toward the `uint128` capacity overflow boundary. Confirm
   the path reverts `E_CapacityReached` (a clean revert that rolls back the pulled USDC), NOT a Solidity
   `Panic` (which would be an ungraceful failure). Ungranted capacity (synth never granted the module mint
   rights) must also revert capacity, and roll back.
3. **setGate auth/re-point.** Non-deployer `setGate` reverts `NotDeployer`; zero reverts `ZeroAddress`; a
   re-point is allowed (build-phase) but grants no standing allowance to the new gate (the allowance is
   per-zap, exact-amount, reset). Confirm a re-pointed gate gets no pre-existing approval to drain.
4. **ctor guards.** Zero on any of the 4 args reverts; `DecimalsTooFew` when USDC finer. Any arg left
   unguarded that yields a fail-open module?
5. **Preview faithfulness / un-wired.** `deposit`/`previewDeposit` work UN-WIRED (gate unset); `zap`/`previewZap`
   revert `NotWired` when the gate is unset. Confirm `previewZap` is an estimate (NAV moves) and matches
   realized at a fixed NAV — a preview that diverges from realized would mislead the front-end but is not a
   fund bug (classify accordingly).
