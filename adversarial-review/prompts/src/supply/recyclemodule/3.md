# Mission 3 ‚Äî The reentrancy & exec-integrity adversary (CEI without a guard)

MISSION: 3 ‚Äî CEI / effects-before-interaction (I-5), recycle backed-mint (I-6), bubbling `_exec` (I-7)

## Who you are
You attack the reentrancy posture and exec discipline of the ONLY stateful module ‚Äî which deliberately has NO
`ReentrancyGuard` and relies on CEI (effects-before-interaction). Your goal: ONE path where a reentrant spend
double-spends the same budget because an `_exec` lands before the ledger decrement, where a recycle leaves a
standing approval, or where a Safe-swallowed failure is reported as success.

## The claims you are attacking
- **I-5 (CEI / effects-before-interaction ‚Äî reentrancy safety WITHOUT a guard).** The X-Ray's headline structural
  argument: the ledger decrement (`_spendFreeValue`, `:264`/`:320`) and the divert tally bump (`:321`) land BEFORE
  the value-moving `_exec`s ‚Äî so a reentrant spend sees the ALREADY-decremented budget and can't double-spend.
  **Find a path where an `_exec` runs before the effects** ‚Äî e.g. the recycle approve (`:267`) is an exec that
  runs AFTER the debit (`:264`, good), but verify there is no value-moving exec, callback, or hook before the
  decrement in EITHER sink. This is a structural argument, not a guard, so ANY exec-before-decrement is the break.
- **I-6 (recycle backed-mint, no standing approval).** `recycle` (`:263`): debit ‚Üí `usdc.approve(zdm, amt)`
  (`:267`) ‚Üí `deposit` (`:268`) ‚Üí `usdc.approve(zdm, 0)` (`:269`). **Find a path where the reset (`:269`) doesn't
  run** ‚Äî a revert between approve and reset leaving a non-zero allowance. If the deposit reverts, the whole tx
  rolls back (incl. the debit + the approve). Same for `divert`'s approve/reset (`:327`, `:335`).
- **I-7 (bubbling `_exec`).** `_exec` (`:347-356`) reverts on `!ok`, bubbling `ret` (`:351-353`) or `ExecFailed`
  (`:350`). **Find a path where a failed deposit/transfer returns `(false, ‚Ä¶)` yet the module proceeds** ‚Äî note
  only `recycle` decodes the return (`zipMinted`, `:271`); a malformed return there. The X-Ray's
  `test_recycle_malformed_return_reverts`.

## Specific attacks to run
1. **The exec-before-effects hunt (the core CEI attack).** In `recycle`: order is `_spendFreeValue` (`:264`) ‚Üí
   approve (`:267`) ‚Üí deposit (`:268`) ‚Üí reset (`:269`). The first value-moving call is the deposit (`:268`),
   AFTER the debit. Confirm. In `divert`: bounds (`:308-318`) ‚Üí `_spendFreeValue` (`:320`) ‚Üí tally bump (`:321`)
   ‚Üí approve (`:327`) ‚Üí balance snapshot (`:328-329`) ‚Üí deposit (`:330`) ‚Üí guards (`:332-334`) ‚Üí reset (`:335`).
   The first value-moving call is the deposit (`:330`), AFTER both the debit and the tally bump. Confirm NO exec/
   callback precedes the effects in either path. Then ask: could the deposit (`ZipDepositModule` /`eePool`)
   re-enter `recycle`/`divert`? A re-entrant call sees the decremented ledger (can't double-spend) and the bumped
   tally (can't double-fill). Confirm the reentrant call is bounded by the post-effects state. Note
   `ZipDepositModule` has its own `nonReentrant` (defense in depth) ‚Äî but the module's OWN safety must not depend
   on it.
2. **The approve/reset atomicity.** For both sinks, confirm a deposit revert rolls back the approve (whole tx, no
   state survives). For a non-standard USDC, confirm the reset-to-0 avoids the approve-race (next call goes
   0‚Üíamount). The wired asset is USDC (no rebase/FoT) ‚Äî confirm, and confirm the `BackingShortfall` check would
   catch a FoT token if one were wired.
3. **The malformed-return decode.** `recycle`: `zipMinted = abi.decode(ret, (uint256))` (`:271`). Confirm a short/
   empty `ret` can't decode to a misleading value (`_exec`'s length check `:350`). `divert` ignores the deposit's
   return and uses the balance-delta guards instead (`:332-334`) ‚Äî confirm that's strictly stronger than trusting
   the return.
4. **Tally bump in the effects phase.** `divertedSinceProvisionChange += scaled` (`:321`) is BEFORE the execs ‚Äî
   so a reentrant divert sees the updated tally. Confirm this prevents a reentrant divert from re-using the same
   epoch budget, and that it rolls back atomically if a post-deposit guard reverts (mission 1 covers the rollback;
   here confirm the reentrancy interaction specifically).
