# Mission 1 — The SEC-09 cumulative-divert adversary (can the hole be over-filled or the tally resurrected?)

MISSION: 1 — the SEC-09 cumulative bound (I-3) + the divert bounds/liveness guards (I-4)

## Who you are
You attack the subtlest property in the fleet: `divert` may fill the capital hole only up to the live
`provision()`, CUMULATIVELY across calls, per provision-epoch — and a stale-value re-mark must NOT resurrect a
spent budget. Your goal: ONE path where the total diverted exceeds the live hole within an epoch, where the
strict-`>` boundary is wrong, or where a `H→H'→H` re-mark sequence lets the old tally come back (the value-key
bug the ticket forbids).

## The claims you are attacking
- **I-3 (SEC-09 cumulative bound).** `divert` (`:307`) reads `hole = provision()` (`:309`), reverts `NoHole` if 0
  (`:310`), and on a CHANGED hole adopts it + zeroes the tally (`:312-315`, reset-on-change). Then the cumulative
  guard: `if (divertedSinceProvisionChange + usdcAmount * 1e12 > hole) revert ExceedsHole()` (`:317-318`). The
  `1e12` scales USDC 6-dp → USD 18-dp; strict `>` allows an EXACT cumulative fill. **Find a path where total
  diverts exceed `hole` within one epoch** — an arithmetic edge in `scaled = usdcAmount * 1e12` (`:317`, overflow?
  — it's checked 0.8, reverts before spend), a tally that doesn't accumulate, or a `hole` read that's stale.
- **The stale-value resurrection (the forbidden bug).** The tally is keyed by LAST-SEEN provision + a single
  counter, NOT by value. A sequence `$100 → $80 → $100`: at the second `$100`, `hole != lastSeenProvision`
  (lastSeen is `$80`), so the tally RESETS — a fresh `$100` budget. The X-Ray says this is intended (cross-epoch
  over-supply is benign — extra backing only strengthens the peg). **Confirm the reset-on-change (`:312`) keys on
  `hole != lastSeenProvision`, NOT on a value-set, so it cannot resurrect a PRIOR epoch's tally to allow a
  double-fill WITHIN what should be one logical epoch.** Find any keying that would let the old tally come back
  improperly.
- **I-4 (bounds-before-spend).** The `NoHole`/`ExceedsHole` checks + the `scaled` overflow land BEFORE
  `_spendFreeValue` (`:320`) — so an over-hole/no-hole divert records no exec and leaves the ledger untouched.
  **Confirm no spend or exec happens before the bounds pass.** Then `NoSharesMinted` (`:334`) + `BackingShortfall`
  (`:332`) are POST-deposit guards — confirm they can't be the thing that lets an over-fill commit (they revert
  the whole tx, rolling back the tally bump).

## Specific attacks to run
1. **The exact-fill boundary.** Confirm `divertedSinceProvisionChange + scaled == hole` is ALLOWED (strict `>`,
   `:318`) and `== hole + 1` reverts. Walk a multi-call sequence summing to exactly the hole, then one wei over.
2. **The reset-on-change keying.** Trace `:312-315`: a changed `hole` sets `lastSeenProvision = hole` and
   `divertedSinceProvisionChange = 0`. Construct `$100 (divert $100) → $80 (divert?) → $100 (divert $100 again)`.
   At the third step the tally resets to 0 → a fresh `$100` budget. Confirm this matches the X-Ray's intended
   behavior and is benign (extra senior backing), AND that there is no sequence where the tally FAILS to reset
   when it should (which would block legitimate diverts — grief) or resets when it should NOT (allowing an
   in-epoch over-fill). The `lastSeenProvision == 0` sentinel: confirm `hole == 0` reverts `NoHole` first (`:310`),
   so the reset block can never set `lastSeenProvision = 0`.
3. **Tally vs ledger atomicity.** `divertedSinceProvisionChange += scaled` (`:321`) lands in the effects phase,
   AFTER `_spendFreeValue` but BEFORE the execs. If a post-deposit guard (`BackingShortfall`/`NoSharesMinted`)
   reverts, the whole tx rolls back — including the tally bump. Confirm a reverted divert does NOT leave the tally
   incremented (which would shrink the remaining budget for a no-op). Trace the rollback.
4. **`provision()` read trust.** `divert` reads `provision()` fresh each call (`:309`, no memoization), from the
   wired `navOracle`. The oracle's sole writer is `DefaultCoordinator` (loss-side). Distrusting the oracle's
   value is out of scope (it's the loss subsystem's trust); attack only how THIS module USES the read — e.g. a
   read that's cached, or a `hole` that could be manipulated by an interleaved call the module doesn't expect.
