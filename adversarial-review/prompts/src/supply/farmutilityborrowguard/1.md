# Mission 1 — The borrow-gate adversary (can a non-engine account borrow the shared USDC?)

MISSION: 1 — the account-identity `OP_BORROW` gate (I-1)

## Who you are
You attack the whole reason this hook exists: a borrow is allowed ONLY when the EVK-appended on-behalf account is
the engine Safe. The borrowed USDC is shared depositor cash. Your goal: ONE path where a borrow on an on-behalf
account that is NOT `juniorTrancheEngine` passes the gate — i.e. an outsider levers the depositor cash on their
own account.

## The claims you are attacking
- **I-1 (OP_BORROW pinned to the engine Safe).** `fallback()` (`:94`) reverts `NotEngineSafe` unless `_msgSender()
  == juniorTrancheEngine` (`:95`). The hook is installed only on `OP_BORROW`, so it only ever guards borrows.
  **Find a borrow on a non-engine on-behalf that the gate admits** — a `_msgSender()` that returns the engine
  address for a non-engine borrower (spoof — mission 2), an op other than `OP_BORROW` reaching the fallback (it's
  op-agnostic; but it's only INSTALLED on OP_BORROW — confirm the install scope), or a borrow path that doesn't
  invoke the hook.
- **The gate is account-IDENTITY, not operator-auth.** Distinct from `CREGatingHook` (which gates
  `isAccountOperatorAuthorized`): here the engine Safe borrows on its OWN account (no operator), so the gate is
  `_msgSender() == juniorTrancheEngine` exactly. **Confirm there's no operator/delegation path** that would let a
  third party act "as" the engine — the gate is a literal address equality.
- **Op-agnostic, no return data.** `fallback()` reverts on mismatch, else falls through (no return). For a hook on
  `OP_BORROW`, the EVK calls it before the borrow; a revert blocks the borrow. **Confirm a pass (engine on-behalf)
  doesn't need to return anything** (the EVK treats a non-reverting hook call as allowed) and a non-engine reverts.

## Specific attacks to run
1. **The on-behalf extraction.** `_msgSender()` (`:101`) returns the appended 20-byte account when `msg.sender` is
   a factory proxy, else raw `msg.sender`. For a real borrow, the EVK (a vault = factory proxy) calls the hook
   with the on-behalf appended → `_msgSender()` returns the borrower's account. Confirm the gate compares THAT to
   `juniorTrancheEngine`. The X-Ray's `test_third_party_borrow_blocked_by_guard` proves a third party on its own
   account is rejected on the REAL market — find a case it misses (a sub-account of the engine? the engine's EVC
   sub-account 1 vs 0?).
2. **Sub-account nuance.** The engine borrows on `juniorTrancheEngine` (sub-account 0, address == Safe). EVC
   sub-accounts share the upper 19 bytes, differing in the last byte. Could a borrow on a SUB-account of the
   engine (e.g. `juniorTrancheEngine ^ 1`) be a different address that fails the `== juniorTrancheEngine` check
   (so it's blocked — safe) OR be treated as the engine (unsafe)? Confirm the gate is an EXACT address equality,
   so only sub-account 0 (the Safe address) passes — a stricter-is-safer property; confirm no sub-account
   collision admits an outsider.
3. **Install scope.** The hook is op-agnostic but installed only on `OP_BORROW` (deploy/config, X-1). Confirm the
   `fallback()` gate doesn't need to distinguish ops (it guards every call it receives) and that being installed
   on the wrong op would be a deploy error (out of scope) — the third-party test evidences the OP_BORROW install
   end-to-end.
4. **No value path.** `fallback()` is non-payable (`:94`, the EVK invokes with no value). Confirm no value/asset
   moves through the hook — it's a pure gate (revert-or-allow), no custody.

## What is already proven (don't re-report as new)
`test_third_party_borrow_blocked_by_guard` — on the REAL EVK market: the engine Safe's loop borrow passes; a third
party that deposits the same escrow on its OWN account and calls `EVC.call(borrow, …)` is rejected `NotEngineSafe`.
Confirming the account-identity gate holds goes in your Summary. A finding must be a concrete non-engine-borrow-
admitted path the live-market third-party test does NOT cover — most valuably a sub-account or spoof angle (spoof
is mission 2).
