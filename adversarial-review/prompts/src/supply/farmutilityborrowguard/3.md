# Mission 3 — The admin adversary (the raw-msg.sender onlyOwner + the borrow-allowlist re-point)

MISSION: 3 — the admin surface: `onlyOwner` (raw `msg.sender`) + `setJuniorTrancheEngine`/`setEVaultFactory`/`transferOwnership` (I-4)

## Who you are
You attack the admin surface — which is security-relevant because re-pointing `juniorTrancheEngine` literally
CHANGES WHO MAY BORROW the shared USDC, and the `onlyOwner` here uses a NON-STANDARD raw-`msg.sender` check
(deliberately bypassing the EVK `_msgSender()` decoder). Your goal: ONE path where a non-owner re-points the
borrow allowlist or the factory, or where the raw-`msg.sender` `onlyOwner` is bypassable.

## The claims you are attacking
- **I-4 (admin gate uses raw `msg.sender`).** `onlyOwner` (`:49-52`) checks `msg.sender != owner` → `NotOwner` —
  the RAW `msg.sender`, NOT the hook `_msgSender()` decoder, because OZ `Ownable`'s `Context._msgSender()` would
  collide with the EVK trailing-data decoder, and the admin is never an EVK on-behalf call. **Confirm the raw
  check is correct and not bypassable** — e.g. could an admin call arrive THROUGH a vault (so `_msgSender()` would
  decode an appended account)? The admin functions aren't EVK hooks, so they're called directly; confirm a
  factory-proxy can't call `transferOwnership`/`setX` with a spoofed trailing account that the raw `msg.sender`
  check would misread (it reads raw `msg.sender`, ignoring any trailing bytes — confirm that's robust).
- **`setJuniorTrancheEngine` = the borrow allowlist.** `setJuniorTrancheEngine` (`:79`) re-points the ONLY account
  allowed to borrow. A non-owner reaching it would let an attacker set themselves as the legal borrower. **Confirm
  it's `onlyOwner` + zero-guarded** (`:79-83`), and that the owner is the Timelock (post-`transferOwnership`).
- **`setEVaultFactory` = the anti-spoof root.** `setEVaultFactory` (`:72`) re-points the factory that `isProxy`
  (the anti-spoof, mission 2) trusts. A re-point to an attacker factory that returns `isProxy(attacker) == true`
  would let the attacker spoof the on-behalf account. **Confirm it's `onlyOwner` + zero-guarded** — a hostile
  factory is the sharpest admin consequence (it breaks mission 2's anti-spoof). Rate: owner-only (Timelock), X-1
  build-phase wiring — but the consequence (anti-spoof defeated) is worth stating.
- **`transferOwnership` handoff.** `transferOwnership` (`:64`) hands the admin to the Timelock; zero-guarded
  (`:65`); after it, the old owner loses the gate. The ctor sets `owner = msg.sender` (the deployer, `:59`).
  **Confirm the handoff is clean** — no path where the old owner retains admin, no zero-address owner bricking the
  guard.

## Specific attacks to run
1. **Raw-`msg.sender` robustness.** Confirm `onlyOwner` (`:50`) reads `msg.sender` directly, never the trailing-
   data `_msgSender()`. Since admin calls are direct (not EVK hooks), `msg.sender` is the real caller. Could a
   factory proxy be tricked into calling an admin function on the attacker's behalf with appended bytes? Even if
   so, `onlyOwner` reads raw `msg.sender` (the proxy), not the appended account — so it'd fail unless the proxy IS
   the owner (it isn't). Confirm the raw check can't be confused by the EVK trailing-data convention.
2. **The three admin functions' guards.** `transferOwnership` (`:64`), `setEVaultFactory` (`:72`),
   `setJuniorTrancheEngine` (`:79`) — confirm ALL THREE carry `onlyOwner` and a `ZeroAddress` guard. A missing
   guard on any (esp. `setJuniorTrancheEngine`, the borrow allowlist) would be the hole. The X-Ray's
   `test_guard_admin_onlyOwner_transfer_and_wiring` proves all three.
3. **The hostile-factory re-point (the sharpest consequence).** `setEVaultFactory(attackerFactory)` → mission 2's
   `isProxy` now trusts whatever the attacker factory says → the attacker can spoof the on-behalf as the engine →
   the borrow gate is defeated. Confirm this requires the OWNER (Timelock) to re-point to a hostile factory (an
   X-1 build-phase wiring act, closed by the pre-prod re-freeze). State the consequence (anti-spoof root of trust)
   but rate it owner-only, not an operator/outsider path.
4. **No CRE operator, no value.** Confirm there is NO operator role here (unlike the fleet modules) and no value
   path — the only privileged surface is the 3 owner functions. So the attack surface is exactly "can a non-owner
   reach them" + "what does a hostile owner re-point break".

gate); the zero-guards fire; `setJuniorTrancheEngine`/`setEVaultFactory` effects; `transferOwnership` hands off +
the old owner then loses the gate. Confirming the admin gate holds goes in your Summary. A finding must be a
