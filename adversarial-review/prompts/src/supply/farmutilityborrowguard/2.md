# Mission 2 — The spoof adversary (can a non-vault caller forge an authorized account?)

MISSION: 2 — the `isProxy`-guarded `isHookTarget()` + `_msgSender()` anti-spoof (I-2/I-3)

## Who you are
You attack the anti-spoof: the appended on-behalf account is trusted ONLY when the caller is a recognized factory
proxy (a real vault), and `isHookTarget()` returns the magic selector ONLY to a proxy. Your goal: ONE path where a
non-vault caller gets `_msgSender()` to return a forged authorized account (passing the gate), or where
`isHookTarget` blesses a non-proxy (letting a fake vault install/impersonate).

## The claims you are attacking
- **I-2 (`isHookTarget` anti-spoof).** `isHookTarget()` (`:87`) returns `this.isHookTarget.selector` ONLY when
  `eVaultFactory.isProxy(msg.sender)` (`:88`), else `0` (`:89`). **Find a path where a non-proxy caller gets the
  magic selector** — a wrong `isProxy` read, a re-pointed `eVaultFactory` that returns true for an attacker, or a
  caller that isn't a factory proxy but is treated as one.
- **I-3 (`_msgSender` anti-spoof).** `_msgSender()` (`:101`) returns the appended 20-byte account
  (`shr(96, calldataload(sub(calldatasize(), 20)))`, `:105`) ONLY when `isProxy(msg.sender)` (`:102`); else
  returns raw `msg.sender` (`:102`). So a NON-vault caller can't spoof an authorized account — it gets its own
  `msg.sender`, which (unless it IS the engine) fails the gate. **Find a path where a non-vault caller's appended
  bytes are trusted** — e.g. `isProxy` spuriously true, or the calldata extraction reading attacker-controlled
  bytes when `msg.sender` isn't a proxy.
- **The direct-non-vault `fallback()` edge (uncovered).** The X-Ray flags this: a DIRECT (non-vault) call to
  `fallback()` falls to `_msgSender()` returning raw `msg.sender` (`:102`), then `!= juniorTrancheEngine` →
  `NotEngineSafe` (`:95`). **Confirm this fails CLOSED** — a random caller calling the hook directly is rejected
  (its own address != engine), AND the engine Safe calling DIRECTLY (not via the vault) would PASS (`msg.sender ==
  engine`) — is that a problem? (A direct engine call to the hook does nothing — the hook has no side effect on a
  pass; it's only meaningful when the EVK invokes it pre-borrow.) Confirm the direct edge is benign.

## Specific attacks to run
1. **`isProxy` trust.** Both anti-spoof checks call `eVaultFactory.isProxy(msg.sender)` (`:88`, `:102`). The
   `eVaultFactory` is the wired EVK GenericFactory (the real factory only returns true for vaults it deployed).
   **Find a path where `isProxy` returns true for a non-vault** — a re-pointed `eVaultFactory` (mission 3's admin
   surface) to an attacker factory that returns true for the attacker. That's an admin/wiring issue (owner-only);
   confirm the anti-spoof's correctness depends on an HONEST factory (the EVK base trust) and attack only the
   guard's USE.
2. **The calldata extraction.** `_msgSender` reads the last 20 bytes of calldata (`:105`). When `msg.sender` is a
   real vault, the EVK appended the on-behalf there. When NOT a proxy, the function returns early (`:102`) WITHOUT
   reading the appended bytes — so a non-vault caller's calldata (which it controls) is NOT read as an account.
   Confirm the early-return (`:102`) happens BEFORE the assembly extraction (`:104`), so a non-proxy caller can't
   feed forged trailing bytes. A reordering (extract THEN check proxy) would be the spoof.
3. **`isHookTarget` install spoofing.** `isHookTarget` is the EVK's install-time check (the vault calls it to
   confirm the hook is willing). Returning `0` to a non-proxy means a non-vault can't get a "yes". Could an
   attacker deploy a fake vault that the real `eVaultFactory.isProxy` recognizes? No — only the real factory's
   deployed proxies return true. Confirm `isHookTarget` can't be tricked into blessing a fake install (which
   would be a deploy-time concern anyway, X-1).
4. **Replication fidelity.** `_msgSender`/`isHookTarget` are replicated VERBATIM from `BaseHookTarget`. Diff the
   inline copy (`:87-90`, `:101-107`) against the EVK convention — a transcription bug (wrong offset, wrong shift
   `shr(96, ...)` = drop the low 12 bytes... wait, `shr(96)` keeps the HIGH 160 bits = the address in the high
   bytes) — confirm the shift extracts the address correctly from the last 20 bytes.

## What is already proven (don't re-report as new)
`test_guard_isHookTarget_only_for_factory_proxy` (non-proxy caller → `0`; real factory proxy → the selector). The
`_msgSender` proxy branch is exercised through the real-vault borrow path (`test_third_party_borrow_blocked_by_
guard`). Confirming the anti-spoof holds goes in your Summary. A finding must be a concrete spoof / non-proxy-
trusted path the isHookTarget + real-vault tests do NOT cover — the direct-non-vault `fallback()` edge fails closed
(note as low-risk uncovered, not a hole, unless you show it fails open); a re-pointed-factory spoof is mission 3.
