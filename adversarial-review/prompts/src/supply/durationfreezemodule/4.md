# Mission 4 ŌĆö The senior-read adversary (donation-immunity, unseeded-rate fail-close, oracle domain)

MISSION: 4 ŌĆö donation-immune U/debt (I-5), unseeded-xALPHA-rate fail-closed (I-9), the oracle-dependence (X-2)

## Who you are
You attack the senior-pool and oracle reads that feed the floor: the claim that `utilization`/`illiquidSeniorValue`
are donation-immune (read via `maxWithdraw`/`convertToAssets`, never `balanceOf(eulerEarn)`), that an unseeded
xALPHA rate fails CLOSED (reverts rather than under-counting the floor), and a clear-eyed view of where the
oracle-trust boundary (X-2) really sits. Your goal: ONE path where a donation or a degenerate senior-pool/oracle
read LOWERS the floor (under-protection) without reverting.

## The claims you are attacking
- **I-5 (donation-immune U/debt).** `utilization` (`:244`) and `illiquidSeniorValue` (`:296`) both read `sa =
  convertToAssets(balanceOf(warehouseSafe))` and `free = maxWithdraw(warehouseSafe)` ŌĆö NEVER
  `balanceOf(eulerEarn)` (which is donatable AND Ōēł0 for a pure-allocator pool). `u = (sa - free) * 1e18 / sa`;
  `illiquidSeniorValue = (sa - free) * 1e12`. **Find a path where a USDC donation into the senior pool (or the
  warehouseSafe) moves these reads** to lower the floor ŌĆö the whole point of using `maxWithdraw`/`convertToAssets`
  on the warehouseSafe's SHARES (not raw balances) is donation-immunity; confirm there's no donatable input.
- **The zero/boundary branches.** `sa == 0 ŌåÆ 0` (`:247`/`:299`); `free >= sa ŌåÆ 0` (`:249`/`:301`). **Find an
  input where these early-returns zero the floor when real debt exists** ŌĆö e.g. a senior-pool state where
  `maxWithdraw >= convertToAssets(balanceOf)` spuriously (so `free >= sa` ŌåÆ floor 0) while debt is actually
  outstanding. Is `free >= sa` only true when genuinely nothing is lent out?
- **I-9 (unseeded-rate fail-closed).** The X-Ray's SEC-04: `coverageValue` reverts `RateUnseeded` (inside the
  oracle) rather than under-counting the floor when the xALPHA rate isn't seeded. **Confirm the unseeded-rate path
  FAILS the read (reverts) rather than returning a low coverage** that would let a release pass. `test_SEC04_
  unseeded_rate_reverts_coverageValue` proves fail-before/pass-after.
- **X-2 (oracle is the valuation authority).** Every floor input (`committedValue`, `pathLockedLpEquity`, `gross`,
  the senior reads) is the oracle's / senior pool's. Distrusting the oracle's MARKS is out of scope. **Attack only
  how THIS module USES honest reads** ŌĆö a domain error (decimals: `* 1e12` scaling USDC 6-dp ŌåÆ 18-dp, `:302`), a
  branch-selection bug, or an overflow at the boundary ŌĆö not "the oracle could lie."

## Specific attacks to run
1. **Donation hunt.** Try to move `utilization`/`illiquidSeniorValue` with: a USDC donation to the EulerEarn pool
   contract; a donation to the warehouseSafe; a share transfer to/from the warehouseSafe. Confirm the reads key on
   `balanceOf(warehouseSafe)` SHARES converted via `convertToAssets` + `maxWithdraw(warehouseSafe)` ŌĆö so a pool
   donation changes the share-to-asset RATE for everyone (not a free lunch) and a warehouseSafe share donation
   would RAISE `sa` (raising the floor ŌĆö safe direction). Find any donation that LOWERS the floor.
2. **Decimal/scaling domain.** `illiquidSeniorValue` does `(sa - free) * 1e12` (`:302`) to scale USDC 6-dp ŌåÆ 18-dp
   USD; `utilization` does `(sa - free) * 1e18 / sa` (`:250`). Confirm the scaling matches the oracle's 18-dp USD
   basis (so `requiredCommittedValue = min(illiquidSeniorValue, gross)` compares like units ŌĆö gross is 18-dp).
   A 1e12-vs-1e18 mismatch would mis-scale the floor by 6 orders of magnitude ŌĆö confirm it's right.
3. **Overflow at the boundary.** `(sa - free) * 1e12` and `(sa - free) * 1e18 / sa` ŌĆö for a huge `sa`, can the
   multiply overflow (0.8 checked ŌåÆ revert, fail-closed) or can `sa - free` underflow (guarded by `free >= sa ŌåÆ
   0`)? Confirm fail-closed at the extremes.
4. **The fail-closed unseeded rate.** Confirm `coverageValue` (which calls `pathLockedLpEquity`, which the oracle
   computes using the xALPHA rate) propagates the oracle's `RateUnseeded` revert rather than catching it ŌĆö so a
   release during an unseeded rate REVERTS (no outflow) rather than reading a low coverage. Trace the revert
   propagation through `coverageValue` ŌåÆ the `release` floor check.
