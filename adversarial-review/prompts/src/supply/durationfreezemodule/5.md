# Mission 5 ‚Äî The privilege & 12-setter wiring adversary

MISSION: 5 ‚Äî access gates, the clone init-lock, and the build-phase wiring blast radius (X-1/X-3)

## Who you are
You attack the access-control and configuration surface: the operator gate, the SEC-14 mastercopy lock, the
SEC-15 owner-recheck, the `setUp` guards, and the **12** `onlyOwner` wiring setters (the largest such surface in
the fleet ‚Äî including the 5 leg tokens that define the whitelist). Your goal: ONE path where a non-operator
rotates value, the mastercopy is initializable, a leg-token re-point desyncs the whitelist from what the oracle
prices (an under-freeze vector), or a re-point DRAINS.

## The claims you are attacking
- **Operator gate.** `commit`/`release` check `msg.sender == operator` (`:354`/`:378`) AFTER the `nonReentrant` +
  `onlyValued` modifiers. **Find a non-operator path to a rotation.** Confirm the operator CANNOT call any
  `onlyOwner` setter or the inherited `setAvatar`/`setTarget`.
- **SEC-14 / SEC-15 / `setUp` guards.** Mastercopy locked (`MastercopyInitLock`); clone `setUp` once
  (`initializer`, `:99`). Seven addresses non-zero (`:110-113`), `owner != operator` (`:114`), `juniorTrancheSafe
  != juniorTrancheSidecar` (`:116`, load-bearing ‚Äî equal Safes make a self-transfer trivially pass the floor),
  THEN the 5 legs read LIVE off the oracle (`:131-136`), THEN `_transferOwnership` (`:139`). **Find a decode/
  order bug** or a path that makes the two Safes equal post-setUp.
- **X-3 / the 12-setter blast radius ‚Äî the leg-token drift is the key one.** 12 `onlyOwner` setters: the 2 Safes,
  operator, navOracle, eulerEarn, warehouseSafe, and the 5 leg tokens (`setZipUSD`/`setUsdc`/`setXAlpha`/`setHydx`/
  `setOHydx`, `:187-219`). **Pressure-test:** `setUp` reads the 5 legs LIVE off the oracle to guarantee the
  whitelist == exactly what the oracle prices (no drift). The 5 leg setters RE-INTRODUCE drift: a `setUsdc(X)`
  makes `X` whitelisted-movable while the oracle still prices the REAL usdc. **Construct the under-freeze:** if a
  leg token is re-pointed to an asset the oracle does NOT price, a `release` of it leaves the sidecar without
  moving `committedValue()` (the oracle doesn't see it leave) ‚Üí the floor passes while real value exits. Is that
  reachable, and is it a DRAIN or an under-freeze? It's `onlyOwner` (Timelock, X-3) ‚Äî but it's the sharpest
  wiring consequence in the fleet.
- **The avatar/target setters are INERT for rotation.** The X-Ray (`:233-237`): rotation uses explicit
  `ISafe(juniorTrancheSafe)`/`ISafe(juniorTrancheSidecar)` calls, NOT the inherited avatar-bound exec. So
  `setAvatar`/`setTarget` (onlyOwner) do NOT redirect rotation. **Confirm this** ‚Äî that re-pointing avatar/target
  can't change where commit/release move value (which reads the explicit Safe slots, re-pointable only via
  `setJuniorTrancheSafe`/`setJuniorTrancheSidecar`).

## Specific attacks to run
1. **The leg-drift under-freeze (the headline wiring question).** A `setUsdc(attackerToken)` (`:194`) makes
   `attackerToken` pass `onlyValued`. Now `release(attackerToken, amount)` moves it sidecar‚Üímain, but the oracle's
   `committedValue` (which reads the REAL usdc the oracle prices) doesn't drop ‚Üí the floor sees no change ‚Üí the
   release passes while `attackerToken` value exits the sidecar. BUT: is `attackerToken` value the oracle even
   counts? If the oracle never priced it, it was never in `committedValue`/coverage, so moving it out doesn't
   lower real coverage either ‚Äî reason carefully about whether the drift actually under-freezes or is benign
   (moving an un-priced asset doesn't change the priced coverage). Then the dangerous version: re-point a leg to a
   DIFFERENT real priced asset, desyncing the whitelist from the oracle's set. Work whether the Timelock can
   create a genuine under-freeze, and rate it (X-3, owner-only, closed by the pre-prod re-freeze).
2. **Safe re-point to equal.** `setJuniorTrancheSafe`/`setJuniorTrancheSidecar` (`:144`/`:151`) each zero-guard
   but do NOT re-check distinctness (unlike `setUp`'s `:116`). **Can the Timelock set them equal** (re-introducing
   the self-transfer degeneracy that trivially passes the floor)? If so, note it: a missing distinctness re-check
   on the setters vs `setUp`. Rate severity (owner-only, but a real guard asymmetry).
3. **navOracle/eulerEarn/warehouseSafe re-point.** Re-pointing `navOracle` (`:166`) changes the entire floor
   valuation source; `eulerEarn`/`warehouseSafe` (`:173`/`:180`) change the senior-debt read. A re-point to an
   attacker oracle/pool could report a zero floor (under-freeze) or a low debt. Drain or under-freeze? All
   owner-only (X-3) ‚Äî frame the consequence and whether the pre-prod re-freeze closes it.
4. **Operator ‚Üí owner escalation.** Confirm the operator cannot reach any setter or become owner. SEC-15 (`:160`)
   keeps operator != owner.
