# Mission 1 — The under-freeze adversary (can `release` open the run hatch below the floor?)

MISSION: 1 — the autonomous floor: un-breachable + debt-pinned (I-1/I-2)

## Who you are
You attack the single most important property in the szipUSD subsystem: no successful `release` ever leaves
coverage below the live liability floor. The drill's #1 question. Your goal: ONE `release` path where post-move
`coverageValue()` ends below `requiredCommittedValue()` and the call still succeeds — an under-freeze that opens
the run hatch while senior debt is outstanding.

## The claims you are attacking
- **I-1 (the floor holds).** `release` (`:377`) moves the asset SIDECAR→MAIN, then reads `floor =
  requiredCommittedValue()` (`:390`) and `c = coverageValue()` (`:391`) and reverts `FreezeFloorBreach` if `c <
  floor` (`:392`). The read is POST-move; the revert atomically rolls the transfer back. **Find a path where the
  post-move coverage is below the floor but `c < floor` is false** — a stale read, a coverage that doesn't reflect
  the just-moved asset, an ordering where the floor is read before the move's effect lands, or a rounding that
  makes `c >= floor` spuriously.
- **I-2 (debt-pinned, un-drainable).** `requiredCommittedValue = min(illiquidSeniorValue(), grossBasketValue())`
  (`:312-315`). `illiquidSeniorValue` (`:296`) is `(sa - free) * 1e12` — ABSOLUTE lent-out senior dollars, NOT
  divided by `sa`, so shrinking the junior basket can't move it (that's what makes the floor un-drainable). **Find
  an input where shrinking the basket, or manipulating the senior pool reads, LOWERS the floor** so a release that
  should breach passes — e.g. a path that makes `illiquidSeniorValue` read low, or the `min(...)` selecting
  `gross` improperly.
- **The coverage numerator.** `coverageValue = committedValue() + pathLockedLpEquity()` (`:287`). After a release
  of a plain leg, `committedValue` (sidecar-only) drops by the moved value. **Confirm the floor check sees that
  drop** — that `committedValue()` re-reads live from the oracle post-move (it does, `:262`) and isn't cached.

## Specific attacks to run
1. **The post-move read ordering.** Walk `release`: transfer via `ISafe(sidecar).execTransactionFromModule`
   (`:382`) → `TransferShortfall` delta check (`:386`) → read floor (`:390`) + coverage (`:391`) → compare
   (`:392`). The coverage read happens AFTER the asset has physically left the sidecar. Confirm `committedValue()`
   (the oracle's sidecar-only sum) reflects the moved asset at read time — i.e. the oracle reads live Safe
   balances, so the post-transfer balance is what's summed. Find any caching/snapshot that would let the floor see
   the PRE-move (higher) coverage.
2. **Debt-pin manipulation.** `illiquidSeniorValue` reads `sa = convertToAssets(balanceOf(warehouseSafe))` and
   `free = maxWithdraw(warehouseSafe)` (`:298-301`). Can an attacker move `sa`/`free` between the move and the
   read to lower the floor? These are senior-pool reads (EulerEarn); donation-immune (mission 4). Within a single
   `release` tx, can the operator (or a reentrant call — but `nonReentrant`) change the senior pool state? Trace
   whether the floor is manipulable mid-tx.
3. **The `min` selection.** `requiredCommittedValue = debt < gross ? debt : gross` (`:315`). The cap-at-gross
   handles the insolvent edge (`debt > gross` → freeze everything). **Find a state where `gross` reads low** (so
   the floor caps below the real debt) and a release passes that shouldn't — is `grossBasketValue` manipulable
   (donation into a Safe? mission 2 covers gross-parity), and does a low gross actually under-protect (it caps the
   floor — but if gross is genuinely low, the basket can't cover the debt anyway; reason about whether the cap is
   safe).
4. **The zero-floor drain.** `test_release_floor_zero_allows_full_drain` — when the floor is 0 (no senior debt),
   full drain is allowed. Confirm a floor of 0 ONLY arises when `illiquidSeniorValue == 0` (no lent-out senior
   dollars) or `gross == 0`, and that a non-zero debt can never read as a zero floor (which would allow a full
   drain with debt outstanding — the catastrophic case).
5. **Debt-rise flips legal→breach.** `test_release_debt_rise_flips_legal_to_breach` proves a release legal at low
   debt reverts when debt rises. Confirm there's no caching of the floor across calls that would let a release use
   a stale (lower) debt.
