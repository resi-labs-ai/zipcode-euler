# Mission 3 — The value-movement adversary (third destination / unvalued asset / FoT)

MISSION: 3 — value moves only between the two Safes (I-3), whitelist (I-6), FoT/false-return defense (I-7)

## Who you are
You attack the transfer mechanics: the claim that value can move ONLY main↔sidecar, only for the 5 oracle-valued
legs (the LP fenced in place, never rotated), with `value==0` and no recipient parameter, and that a fee-on-
transfer or false-return token can't slip the balance-delta check. Your goal: ONE path where value reaches a
third destination, an unvalued asset is moved (the non-basket-asset leak), or a transfer records success without
moving exactly `amount`.

## The claims you are attacking
- **I-3 (value moves only between the two Safes, no recipient param, value==0).** `commit` (`:353`) transfers
  `juniorTrancheSafe → juniorTrancheSidecar` (`:358-359`); `release` (`:377`) transfers `sidecar →
  juniorTrancheSafe` (`:382-383`). Both via `ISafe(src).execTransactionFromModule(asset, 0, transfer(dest,
  amount), 0)` — the `dest` is the literal set-once OTHER Safe, `value` is the literal `0`, operation `0` (Call).
  **Find any path where `dest` becomes operator-influenced, a third destination is reachable, `value != 0`, or a
  delegatecall/generic-exec leaks.** The operator's only inputs are `(asset, amount)` — confirm neither reaches
  `dest`/`value`/`operation`. NOTE: rotation uses explicit `ISafe(src)` calls, NOT the inherited avatar-bound
  exec (which is inert here) — so the avatar/target setters don't redirect rotation (relevant to mission 5).
- **I-6 (only the 5 oracle-valued legs are movable; LP fenced).** `onlyValued(asset)` (`:226`) reverts
  `UnvaluedAsset` unless `asset` is one of `zipUSD/usdc/xAlpha/hydx/oHydx` (`:227`). The ICHI LP share is NOT
  whitelisted (fenced in place, counted via `pathLockedLpEquity`). **Find a path to move an unvalued asset** — the
  X-Ray's security-#6 note: a release of an unvalued asset would leave the sidecar without moving
  `committedValue()`, so the floor would pass while real value exits the freeze. Confirm the whitelist closes
  this for BOTH commit and release.
- **I-7 (FoT / false-return defense).** Both mutators read `beforeBal` on the DEST, exec the transfer, check `!ok`
  → `ExecFailed` (`:361`/`:385`), then assert `dest balance delta == amount` → `TransferShortfall` (`:362`/`:386`).
  **Find a path where a fee-on-transfer token (delta < amount) or a false-return token slips through** — does the
  delta check catch both? Does it read the dest balance correctly (the asset arriving at `dest`)?

## Specific attacks to run
1. **Trace `(asset, amount)` to calldata.** `asset` → the `IERC20.transfer` target + the balance reads; `amount`
   → the transfer amount + the delta check. Confirm NEITHER reaches `dest` (the literal other Safe), `value` (0),
   or `operation` (0). The transfer uses `abi.encodeCall(IERC20.transfer, (dest, amount))` — typed; confirm `dest`
   is the set-once `juniorTrancheSidecar`/`juniorTrancheSafe`, not derived from input.
2. **The unvalued-asset leak (release side).** A `release(unvaluedAsset, amount)` is barred by `onlyValued`
   (`:226`). But confirm: if an unvalued asset somehow held in the sidecar could be released, the floor check
   (`committedValue` reads only the 5 valued legs via the oracle) wouldn't see it leave → the floor passes while
   value exits. Confirm the whitelist is the ONLY gate needed and that the oracle's `committedValue` would indeed
   miss an unvalued asset (so the whitelist is load-bearing, not redundant).
3. **The delta check vs FoT.** `IERC20(asset).balanceOf(dest) - beforeBal != amount` (`:362`/`:386`). For a FoT
   token, the dest receives `amount - fee` → delta `!= amount` → revert. Confirm. For a false-return token
   (transfer returns false but the Safe's `execTransactionFromModule` returns ok), the delta check catches a
   no-move (delta 0). Confirm both branches. Can a token with a transfer HOOK re-enter? `nonReentrant` guards the
   mutator — confirm the guard wraps the whole transfer+check.
4. **No custody.** Confirm the module never holds the asset (it execs THROUGH the Safes; the tokens never touch
   the module). A balance accruing to the module would be a leak; confirm none does.
5. **Self-transfer degeneracy.** `setUp` rejects `juniorTrancheSafe == juniorTrancheSidecar` (`:116`, `BadParams`)
   — because equal Safes make a rotation a self-transfer that trivially passes the floor. Confirm no setter
   (mission 5) can later make them equal (re-introducing the degeneracy).
