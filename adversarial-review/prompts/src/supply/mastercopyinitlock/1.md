# Mission 1 — The clone-safety adversary (does the lock break clone init?)

MISSION: 1 — mastercopy locked (I-1), clones still init exactly once (I-2), the empty-initializer trick (I-3)

## Who you are
You attack the one consequential property of a 4-line defense-in-depth mixin: the lock must NOT break clone init.
The mastercopy-lock direction is benign (a bare mastercopy is never enabled on a Safe); the direction that matters
is that a real EIP-1167 clone still `setUp`s EXACTLY ONCE — not zero times (bricked), not twice (re-init). Your
goal: ONE path where the lock makes a clone uninitializable or re-initializable, or where the empty-`initializer`
trick mis-fires.

## The claims you are attacking
- **I-2 (clones still init exactly once — the consequential direction).** EIP-1167 clones NEVER run the
  implementation's constructor, so a clone's storage `_initialized` is fresh-zero (the ctor's `_lockMastercopy()`
  ran only on the mastercopy, mutating only the MASTERCOPY's storage). **Confirm a clone starts `_initialized ==
  0` and `setUp`s once**, then a second `setUp` reverts `AlreadyInitialized`. Find any path where the lock's ctor
  could affect a clone's storage (it can't — proxies don't run impl ctors — but confirm there's no storage-layout
  collision or shared storage that the lock writes which a clone would inherit).
- **I-1 (mastercopy locked — the benign direction).** After construction, `setUp` on the bare mastercopy reverts
  `AlreadyInitialized` (the ctor's `_lockMastercopy()` consumed the one-shot `initializer`). **Confirm the lock
  fires** — but recall the X-Ray/NatSpec: even if it DIDN'T, the impact is nil (a bare mastercopy is never enabled
  on a Safe, holds no custody). So a lock-doesn't-fire finding is INFO/benign, NOT high-severity.
- **I-3 (the empty-`initializer` trick).** `_lockMastercopy()` (`:28`) is `private initializer` with an EMPTY
  body — the `initializer` modifier (zodiac-core `Initializable`) flips `_initialized` WITHOUT running `setUp`'s
  zero/owner validation. **Confirm the trick consumes the SAME one-shot flag `setUp` uses** — so after
  `_lockMastercopy()`, `setUp` (also `initializer`) sees `_initialized == true` and reverts. Find any way the two
  use DIFFERENT flags (so the lock flips one flag but `setUp` checks another → the mastercopy isn't actually
  locked, OR the clone's `setUp` is wrongly blocked).

## Specific attacks to run
1. **Storage isolation mastercopy vs clone.** The ctor runs ONLY at mastercopy deploy, mutating the MASTERCOPY's
   `_initialized`. A clone is a fresh proxy with its own (zero) storage. Confirm there is NO shared/constant
   storage and no immutable that a clone would read from the mastercopy (an immutable `_initialized` would be the
   footgun — but `_initialized` is a storage flag in zodiac-core `Initializable`, not immutable). Confirm the
   deploy path (`ModuleProxyFactory.deployModule`) gives each clone fresh-zero storage.
2. **The one-shot flag identity.** `_lockMastercopy()` is `initializer`; `setUp` (in each module) is `initializer`.
   Both modifiers act on the SAME inherited `_initialized` (zodiac-core `Initializable`). Confirm flipping it in
   the ctor blocks a later mastercopy `setUp` (the lock) AND that a fresh clone (flag zero) can run `setUp` once.
   A finding would be: the trick flips a flag `setUp` doesn't check, so the lock is cosmetic (benign) — OR it
   flips a flag that a clone shares (impossible — separate storage), bricking clones.
3. **Re-init via the lock.** Could `_lockMastercopy()` be callable again (it's `private` — only the ctor calls it)
   or could a module expose a path to re-run it? It's `private` and called only from the ctor (`:23-24`). Confirm
   no external/public re-entry to it.
4. **Inheritance order.** `MastercopyInitLock is Module` (`:22`); each engine module `is MastercopyInitLock`. The
   ctor chain runs `Module`'s ctor (if any) then `_lockMastercopy()`. Confirm the lock runs AFTER any base-ctor
   that might set `_initialized`, and that the module's own ctor (if any) doesn't un-set it. (The modules have no
   ctor of their own beyond the inherited chain — confirm.)

## What is already proven (don't re-report as new)
9× `test_SEC14_mastercopy_setUp_reverts` (bare mastercopy `setUp` → `AlreadyInitialized`) + 9×
`test_mastercopy_inert`/`test_setUp_initializer_once` (fresh clone reads zero wiring, `setUp`s once, a second
reverts) — across all 9 engine modules (18 tests, coverage-complete). Confirming the lock holds AND clones init
exactly once goes in your Summary. A finding must be a concrete clone-init break (bricked or doubled) or a flag-
identity mis-fire the 18 consumer tests do NOT cover — the benign mastercopy-lock direction is INFO at most.
