# Mission 2 — The identity-gate adversary (workflow-id/author bypass + the assembly decode)

MISSION: 2 — the optional workflow-identity gate (I-2) + the hand-rolled `_decodeMetadata` (the assembly replica)

## Who you are
You attack the second layer of the report gate: when configured, a mismatched workflow-id or author must be
rejected — and that check depends on a hand-rolled assembly `_decodeMetadata` that reads fixed calldata offsets
with NO length validation. Your goal: ONE path where a configured identity check is bypassed, or where a
malformed/short `metadata` blob makes `_decodeMetadata` read zero/garbage that spuriously MATCHES the expected
id/author (silently weakening the gate).

## The claims you are attacking
- **I-2 (optional workflow-identity gate).** `onReport` (`:66`): if `expectedWorkflowId != 0 || expectedAuthor !=
  0`, it decodes the metadata (`:67`) and reverts `InvalidWorkflowId` (`:69`) on an id mismatch and `InvalidAuthor`
  (`:72`) on an author mismatch. When both are unset, the forwarder gate alone applies. **Find a path where a
  configured id/author is bypassed** — a decode that returns the EXPECTED value regardless of input, an OR/AND
  logic error in the two checks, or a configured-but-skipped branch.
- **The `_decodeMetadata` assembly (replicated verbatim, NO length check).** `_decodeMetadata` (`:116`) reads
  `workflowId` at offset 32, `workflowName` at 64, `workflowOwner` at 74 (via `shr(96, ...)`) — fixed offsets,
  no `metadata.length` validation. The layout is `abi.encodePacked(workflowId bytes32, workflowName bytes10,
  workflowOwner address)`. **Find a malformed/short `metadata` that reads garbage** — a blob shorter than 74+20
  bytes reads zero/uninitialized memory; could that garbage `workflowOwner`/`workflowId` spuriously equal a
  configured `expectedAuthor`/`expectedWorkflowId`? (The X-Ray flags this as the uncovered edge — low severity
  since a wrong decode usually MISMATCHES → revert, but a transcription/offset bug or a zero-reads-as-unset could
  weaken it.)
- **Diff against `ReceiverTemplate._decodeMetadata`.** The X-Ray says it's replicated VERBATIM (`:107-115`). **Diff
  the assembly offsets byte-for-byte** against `ReceiverTemplate`'s — a transcription error (a wrong offset, a
  wrong shift) would silently mis-extract the identity fields, breaking the gate while passing the happy-path test.

## Specific attacks to run
1. **The decode happy path vs the offsets.** Confirm `workflowId := mload(add(metadata, 32))` (`:122`),
   `workflowName := mload(add(metadata, 64))` (`:123`), `workflowOwner := shr(mul(12,8), mload(add(metadata,
   74)))` (`:124`). For `bytes memory metadata`, offset 32 is the first data word (after the length word at 0).
   So workflowId is the first 32 bytes of data, workflowName the next 10 (at 64? — that's data offset 32, i.e.
   the 33rd–42nd data bytes), workflowOwner the 20 bytes at data offset 42 (mload at 74 = 32 length + 42, shr 96
   takes the high 20). Walk these offsets against the packed layout `bytes32 ++ bytes10 ++ address` and confirm
   they line up. A 1-byte slip mis-reads the author.
2. **Short-metadata garbage.** If `metadata.length < 62` (32+10+20), the `mload(add(metadata, 74))` reads beyond
   the blob into adjacent memory (zero or leftover). Construct a short blob where the garbage `workflowOwner`
   happens to equal a configured `expectedAuthor` (e.g. `expectedAuthor` is an address that aligns with zero-
   padding). Could a forwarder (the only caller — it's forwarder-gated) supply such metadata? The forwarder is
   trusted (Keystone), but confirm a malformed metadata from even the trusted forwarder fails closed (mismatch →
   revert) rather than spuriously matching.
3. **The OR/AND logic.** `if (expectedWorkflowId != 0 || expectedAuthor != 0)` (`:66`) decodes; then `if
   (expectedWorkflowId != 0 && workflowId != expectedWorkflowId)` (`:68`) and `if (expectedAuthor != 0 &&
   workflowOwner != expectedAuthor)` (`:71`). Confirm: if only id is set, only id is checked; if only author, only
   author; if both, both. Find a config where one is set but its check is skipped (e.g. a `||` that should be
   `&&` or vice-versa).
4. **Zero-as-unset collision.** `expectedAuthor == 0` means "check off". Could a metadata with `workflowOwner ==
   0` (garbage/short) interact with this so a check that SHOULD run is treated as off? No — the gate keys on
   `expectedAuthor != 0` (the configured value), not the decoded one. Confirm the decoded zero can't disable a
   configured check.

## What is already proven (don't re-report as new)
`test_CTR01_workflow_id_mismatch_reverts_match_passes` (id mismatch reverts, match passes),
`test_CTR01_workflow_author_mismatch_reverts_match_passes` (author mismatch → `InvalidAuthor`, match passes),
`test_CTR01_report_postBid_*` (the both-unset all-zero-meta path). Confirming the identity gate holds when
configured goes in your Summary. A finding must be a concrete identity-bypass / malformed-metadata-spurious-match /
offset-transcription path the id+author happy/mismatch tests do NOT cover — the short-metadata garbage decode is
the one most worth working (the X-Ray flags it uncovered).
