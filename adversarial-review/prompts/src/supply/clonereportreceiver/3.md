# Mission 3 — The dispatch & shared-owner adversary (two-doors equivalence, MRO, setter auth)

MISSION: 3 — `_processReport` dispatch (I-3), the single-owner MRO (X-1), the setter `onlyOwner`

## Who you are
You attack the dispatch hook and the inheritance/ownership structure: the claim that the report path reaches the
SAME `_processReport` internals as the operator path (identical effect), that C3/MRO merges the zodiac `Ownable`
to ONE owner (no second `Ownable` from the receiver), and that the three wiring setters are genuinely `onlyOwner`.
Your goal: ONE path where the report door does something the operator door can't, where a second owner exists, or
where a setter is reachable by a non-owner.

## The claims you are attacking
- **I-3 (two doors, one guard set).** `onReport` dispatches to `_processReport(report)` (`:76`), the abstract hook
  (`:82`) the concrete module implements (e.g. `SzipBuyBurnModule._processReport` routes the §8.0 envelope to the
  SAME `_postBid`/`_cancelBid` the operator path uses). **Confirm this base imposes NO dispatch logic of its own
  that could diverge** — it just calls `_processReport(report)` after the gates. The base can't make the two doors
  differ (that's the concrete module's job); confirm the base passes the report through unmodified.
- **X-1 (single-owner MRO).** This is `is Ownable, IReceiver` where `Ownable` is zodiac-core's
  `reference/zodiac-core/contracts/factory/Ownable.sol` — the SAME base `Module` derives from. C3 linearization
  merges it to ONE `owner`/`onlyOwner`. **Confirm there is no SECOND `Ownable` (no OZ `Ownable`)** — the whole
  reason it doesn't inherit `ReceiverTemplate` (which brings OZ `Ownable`) is to avoid two owners. A second owner
  would split authority (the setters' `onlyOwner` vs the module's). Diff the import (`:4`) — it's the zodiac-core
  `Ownable`, not OZ.
- **The setter `onlyOwner`.** `setForwarder` (`:87`), `setExpectedAuthor` (`:93`), `setExpectedWorkflowId` (`:99`)
  are `onlyOwner`. The X-Ray notes this is proven via the SHARED zodiac `Ownable` on sibling setters, NOT directly.
  **Confirm all three carry the `onlyOwner` modifier** (read `:87`, `:93`, `:99`) and that the modifier is the
  module's single owner (the Timelock), so the CRE operator (hot key) cannot re-point the forwarder/identity.

## Specific attacks to run
1. **The dispatch is a pass-through.** Confirm `onReport`'s only post-gate action is `_processReport(report)`
   (`:76`) — no decode, no transform of `report` in the base (the concrete module decodes the §8.0 envelope).
   So the base can't make the report door diverge from the operator door; the equivalence is the concrete
   module's `_processReport` reaching the same internals. (The buy-burn two-doors equivalence is its own mission;
   here confirm the BASE doesn't inject a divergence.)
2. **MRO / single owner.** Read the inheritance (`:29` `is Ownable, IReceiver`) and the import (`:4`
   `@gnosis-guild/zodiac-core/factory/Ownable.sol`). Confirm the concrete module (`is MastercopyInitLock,
   CloneReportReceiver` → `MastercopyInitLock is Module is ... Ownable`) linearizes to ONE `Ownable`. A diamond
   that instantiated `Ownable` twice would give two `owner` slots. Confirm `owner`/`onlyOwner` resolve to a single
   slot shared with the module.
3. **Setter reachability.** Confirm `setForwarder`/`setExpectedAuthor`/`setExpectedWorkflowId` are `onlyOwner`
   (Timelock) and the operator can't reach them. A forwarder re-point by a non-owner would let an attacker point
   the socket at their own forwarder (opening the door). The `setForwarder` has NO zero-guard BY DESIGN (zero =
   inert) — confirm that's the only intentional guard omission and it's safe (zero fails closed).
4. **`UnsupportedReportType`.** The base declares `UnsupportedReportType` (`:43`) but the concrete module's
   `_processReport` reverts it for unknown types. Confirm the base doesn't dispatch unknown types itself (it
   delegates to `_processReport`). An unknown type must revert, not fall through.
