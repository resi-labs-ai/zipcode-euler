# Mission 1 — The world-callable-clone adversary (does the inversion fail closed?)

MISSION: 1 — the fail-closed forwarder gate / clone inversion (I-1/I-4)

## Who you are
You attack the entire reason this contract exists: a fresh EIP-1167 clone must be INERT (zero forwarder ⇒
fail-closed), the opposite of `ReceiverTemplate` (zero ⇒ open ⇒ world-callable clone). Your goal: ONE path where a
fresh/unwired clone's `onReport` is callable by anyone, or where the forwarder gate admits a non-forwarder caller.

## The claims you are attacking
- **I-1 (fail-closed forwarder gate).** `onReport` reverts `InvalidForwarder(msg.sender, forwarder)` if `forwarder
  == address(0) || msg.sender != forwarder` (`:62`). A fresh clone has `forwarder == 0` (no constructor runs), so
  the FIRST clause fails closed. **Find a path where a zero-forwarder clone is callable** — e.g. a default/sentinel
  that isn't zero, a `msg.sender == 0` edge (would need a zero caller — impossible), or any branch that reaches
  `_processReport` (`:76`) without passing the gate.
- **I-4 (clone-safe, no constructor).** The base has NO constructor (`:24-25` of the NatSpec; confirm in code) so
  all three wiring slots (`forwarder`/`expectedWorkflowId`/`expectedAuthor`) start zero on a fresh clone — inert
  until the Timelock wires them. **Confirm there is no constructor and no immutable that a clone would miss** (an
  immutable forwarder baked into the mastercopy would be the `ReceiverTemplate` footgun; confirm `forwarder` is
  plain storage, `:34`).
- **The delta from `ReceiverTemplate`.** Diff `onReport`'s gate (`:62`) against `ReceiverTemplate`'s. The X-Ray
  says `ReceiverTemplate`'s gate is "zero ⇒ open"; this base's is "zero ⇒ fail-closed". **Confirm the inversion is
  exactly the `forwarder == address(0)` clause added BEFORE the caller check** — and that there's no path where
  the inversion is incomplete (e.g. the identity-check block running before the forwarder gate).

## Specific attacks to run
1. **The gate ordering.** `onReport` (`:60`): the FIRST statement is the forwarder gate (`:62`), BEFORE the
   workflow-identity block (`:66`) and `_processReport` (`:76`). Confirm nothing runs before the forwarder check —
   no decode, no state change, no callback. A fresh clone hits `:62` and reverts.
2. **The zero-forwarder clause.** `forwarder == address(0) || msg.sender != forwarder` (`:62`). Confirm BOTH
   clauses: a zero forwarder always reverts (even if `msg.sender` were somehow zero, `0 == 0` would pass the
   second clause but the first clause `forwarder == address(0)` reverts first). And a non-zero forwarder requires
   `msg.sender == forwarder`. Find any way to satisfy neither-revert with a zero forwarder.
3. **Constructor / immutable audit.** Read the contract top-to-bottom: confirm NO `constructor`, NO `immutable`,
   NO initializer that a clone skips. The three wiring slots (`:34`, `:37`, `:40`) are plain storage. A fresh
   clone reads them as zero. (Contrast: if `forwarder` were immutable set in a ctor, a clone would read the
   mastercopy's value — the footgun. Confirm it's storage.)
4. **`supportsInterface` doesn't leak.** `supportsInterface` (`:131`) returns true only for `IReceiver` + `IERC165`
   (`:132`). Confirm it advertises nothing spurious (a wrong interface id could mislead a caller, though it can't
   bypass the gate). `0xffffffff` → false.

## What is already proven (don't re-report as new)
`test_CTR01_unwired_clone_onReport_reverts` (zero forwarder → `InvalidForwarder(caller, 0)`),
`test_CTR01_wrong_forwarder_caller_reverts` (wrong caller → `InvalidForwarder`). Confirming a fresh clone fails
closed goes in your Summary. A finding must be a concrete world-callable-clone / gate-bypass path the unwired-clone
+ wrong-caller tests do NOT cover — most valuably an immutable/constructor a clone would mishandle, or an ordering
where something runs before the gate.
