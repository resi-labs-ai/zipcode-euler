# Mission 3 — The fail-closed / wiring adversary (do unwired reads abort, does an unknown silo never touch address(0), and does a non-Euler venue plug in?)

MISSION: 3 — fail-closed reads, venue-agnostic plug-in, and wiring

## Who you are
You attack the BOUNDARY: the unwired-slot guards, the unknown-silo fail-closed, the venue-agnostic plug-in,
and the two Timelock setters. The design claims: every aggregate read reverts `RegistryUnset` until
`registry` is wired; the live ratio reverts `ZipUsdUnset` until `zipUsd` is wired; a per-silo getter returns
0 (never calls into `address(0)`) for an unknown silo; the per-silo read is against the venue-neutral
`ISeniorPool` surface so a non-Euler venue plugs in; and both setters are `onlyOwner` + `ZeroAddress`-guarded
while the ctor accepts zero for both. Your goal: find a read that proceeds on an unwired slot, a getter that
dereferences `address(0)`, a venue coupling that breaks the plug-in, or a setter auth/zero-guard gap.

## The claims you are attacking
- **I-8 (RegistryUnset until wired).** Every aggregate / per-silo read guards
  `if (address(registry) == address(0)) revert RegistryUnset()` at the TOP: `seniorBacking:75`,
  `activeSeniorBacking:86`, `illiquidSeniorValue:96`, `seniorBackingOf:124`, `illiquidSeniorValueOf:133`.
  **Find any read that loops the registry (`registry.allSiloIds()` / `registry.getSilo()`) BEFORE the guard**
  — that would call into `address(0)` (revert with no/wrong error) or, worse, proceed on a stale registry.
  Confirm the guard precedes the first `registry.` call on every path.
- **I-9 (per-silo getters: unknown silo → 0, never calls into address(0)).** `seniorBackingOf` (`:123-128`)
  and `illiquidSeniorValueOf` (`:132-137`) call `registry.getSilo(siloId)` then check
  `if (s.eePool == address(0)) return 0` (`:126,:135`) BEFORE `_seniorValue`/`_illiquidValue`. An unknown
  silo's `getSilo` returns a zero-filled struct (`eePool == 0`); the guard must short-circuit so
  `_seniorValue(address(0), ...)` is never called (which would `convertToAssets`/`balanceOf` on
  `address(0)`). **Find a path where an unknown/zero `eePool` reaches `ISeniorPool(address(0))`.**
- **I-10 (CTR-10b venue-agnostic).** The per-silo read is against `ISeniorPool` (the
  `{balanceOf, convertToAssets, maxWithdraw}` surface, `ISeniorPool:18-28`), NOT an Euler-specific type.
  Confirm `_seniorValue`/`_illiquidValue` (`:53,:62`) construct `ISeniorPool(eePool)` and call ONLY those
  three selectors — no `eulerEarn()`, no `expectedSupplyAssets`, no Euler-named getter. A coupling to an
  Euler-only selector would break a non-Euler venue's plug-in. (The registry admits a non-Euler adapter
  behind a thin wrapper — that surface is the registry's; here confirm the AGGREGATOR reads only the neutral
  three.)
- **I-11 (wiring setters + ctor flexibility).** `setRegistry` (`:142-146`) and `setZipUsd` (`:149-153`): each
  is `onlyOwner`, rejects zero (`:143,:150` → `ZeroAddress`), writes the slot, and emits
  `WiringSet(slot, value)` (`:145,:152`). The ctor (`:43-46`) accepts zero for BOTH (deploy-order
  flexibility). Confirm the setters cannot be called by a non-owner, cannot zero a slot, and that the
  `WiringSet` event tags the correct slot name.

## Specific attacks to run
1. **Guard-before-call ordering.** For each of the five reads (`:75,:86,:96,:124,:133`), confirm the
   `RegistryUnset` guard is the FIRST statement, before any `registry.` dereference. A guard placed after the
   `allSiloIds()` call would call into `address(0)`.
2. **Unknown-silo into address(0).** Call `seniorBackingOf(ghostId)` / `illiquidSeniorValueOf(ghostId)` for a
   never-admitted id. `getSilo` returns a zero struct; confirm `:126`/`:135` returns 0 BEFORE
   `_seniorValue(address(0), address(0))`. Trace whether a partially-zero struct (e.g. `eePool != 0` but
   `warehouseSafe == 0`) could slip past the `eePool == 0` check and read `balanceOf(address(0))` — is
   `eePool == 0` the right and sufficient sentinel given how the registry admits silos?
3. **ZipUsdUnset only on the live form.** `systemCollateralization` (`:115`) reverts `ZipUsdUnset` on an
   unwired `zipUsd`, but the arg form `collateralization(supply)` (`:107`) must STILL work (it never reads
   `zipUsd`). Confirm the unset-zipUsd guard is scoped to the live form only — and that `seniorBacking()`
   etc. do not gate on `zipUsd` at all (zipUsd is irrelevant to the backing numerator).
4. **Venue coupling.** Grep `_seniorValue`/`_illiquidValue` for any selector beyond
   `balanceOf`/`convertToAssets`/`maxWithdraw`. Any Euler-specific call would couple the aggregator to Euler
   and break CTR-10b. Confirm `ISeniorPool` is the only senior-surface import (`:6`) and the only type
   constructed against `eePool`.
5. **Setter auth + zero + event.** `vm.prank(stranger)` each setter → `OwnableUnauthorizedAccount`; zero arg
   → `ZeroAddress`; happy path → slot updated + `WiringSet` with the right `bytes32` slot tag (`"registry"` /
   `"zipUsd"`). Confirm the ctor path accepts zero without reverting (so deploy can precede wiring).

## Pressure-test (don't over-rate)
The build-phase mutable wiring (`registry`/`zipUsd`, both `onlyOwner` + zero-guarded, frozen pre-prod by the
§17 immutable re-freeze) is the documented subsystem residual. A bare "the Timelock can re-point a wired
address" restatement is **INFO** — the owner is the Timelock (a ratified trust anchor) and a re-point cannot
drain (the contract holds no funds). A finding is only actionable if it shows a setter that skips the
`onlyOwner`/`ZeroAddress` guard, zeroes a slot, or emits the wrong slot tag — not the existence of the setter.
