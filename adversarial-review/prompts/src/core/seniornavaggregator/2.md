# Mission 2 — The aggregate-semantics adversary (do retired vs active count right, and can a breaker false-trip?)

MISSION: 2 — aggregate semantics & breaker-safety math

## Who you are
You attack the AGGREGATE semantics: which silos count toward which view, and the collateralization ratio a
circuit-breaker reads. Two design claims. First: `seniorBacking()`/`illiquidSeniorValue()` sum ALL silos
(retired silos STILL back the zipUSD their still-open lines minted — intentional), while
`activeSeniorBacking()` is the routable-capacity view that filters on `active`. Second: the load-bearing
`collateralization(0) → type(uint256).max` — with NO zipUSD outstanding the system is not insolvent, so a
breaker reading `ratio < threshold` must NOT trip. Your goal: find a retired/active mis-count, a
collateralization rounding/overflow error, or a `supply==0` case that lets a breaker false-trip.

## The claims you are attacking
- **I-3 (retired counted in backing, dropped from active).** Trace the THREE summing loops:
  `seniorBacking()` (`:74-81`) and `illiquidSeniorValue()` (`:95-102`) iterate ALL `allSiloIds()` with NO
  `active` check; `activeSeniorBacking()` (`:85-92`) gates each silo on `if (s.active)` (`:90`). Confirm the
  `active` filter belongs ONLY to `activeSeniorBacking`, and that a retired silo (`active == false`, still
  holding senior backing) is INCLUDED in `seniorBacking`/`illiquidSeniorValue` but EXCLUDED from
  `activeSeniorBacking`. A mis-placed filter that drops retired silos from `seniorBacking` would
  UNDER-COUNT the solvency numerator (false insolvency); one that includes them in `activeSeniorBacking`
  would over-state routable capacity.
- **I-6 (collateralization math + zero-supply → max).** `collateralization(supply)` (`:107-110`):
  `supply == 0 → type(uint256).max` (`:108`), else `seniorBacking() * 1e18 / supply` (`:109`). The
  `supply==0 → max` is LOAD-BEARING: a breaker comparing `ratio < threshold` must see `max` (not 0, not
  revert) when no zipUSD is outstanding, so it does NOT trip. **Find a `supply==0` path that returns 0 / a
  small number / reverts** (any of which a `< threshold` breaker reads as insolvency → false-trip). Also
  check the live ordering: does `:108` short-circuit BEFORE the `* 1e18 /` so a zero supply never divides?
- **I-7 (systemCollateralization wired).** `systemCollateralization()` (`:114-117`) reverts `ZipUsdUnset` if
  `zipUsd == address(0)` (`:115`), else calls `collateralization(IERC20(zipUsd).totalSupply())` (`:116`).
  Confirm it routes through the SAME `collateralization` (so the `supply==0 → max` safety applies to the live
  form too — a fully-redeemed zipUSD supply of 0 must yield `max`, not a false-trip).

## Specific attacks to run
1. **Retired-silo accounting.** Construct two silos, retire one (`active=false`) that still holds backing.
   Assert `seniorBacking` STILL includes it (`:74-81` has no `active` gate) and `activeSeniorBacking`
   DROPS it (`:90`). Find any path where the gate leaks into `seniorBacking`/`illiquidSeniorValue` or is
   missing from `activeSeniorBacking`.
2. **Zero-supply false-trip.** Call `collateralization(0)` and `systemCollateralization()` with a 0
   `totalSupply`. Both must return `type(uint256).max`. If either returns 0 (e.g. `seniorBacking()*1e18/0`
   reverting, or a guard checking the wrong variable), a breaker `< threshold` false-trips at the worst
   moment (genesis, full redemption). Confirm the `:108` short-circuit fires first.
3. **Ratio rounding / overflow.** `seniorBacking() * 1e18 / supply` (`:109`). `seniorBacking()` is 18-dp USD;
   could `seniorBacking() * 1e18` overflow uint256 for a plausible backing? (uint256 max ≈ 1.15e77; backing
   in 18-dp would need ~1e59 USD — not plausible, but confirm the multiply-then-divide ordering is integer-safe
   and the truncation direction is acceptable for a telemetry ratio.) Is `1e18 == exactly 100%` the documented
   unit, and does a partially-backed silo round DOWN (conservative for a breaker)?
4. **Stale `seniorBacking` re-read.** `collateralization` calls `seniorBacking()` (`:109`) which re-loops the
   registry. Confirm there is no cached/stale numerator — the ratio is computed against a FRESH sum each call
   (a pure view, no storage of the aggregate). A cached aggregate would be a staleness bug; confirm there is
   no cache.
5. **Active filter on the wrong field.** Confirm `:90` gates on `s.active` (the registry-managed bool,
   `SiloRegistry.Silo.active`), not on `eePool != 0` or some other field — a wrong-field filter would
   mis-classify silos.

## What is already proven (don't re-report as new)
`test_retired_stillCounted_droppedFromActive` (retire one of two silos; asserts it stays in `seniorBacking`
and leaves `activeSeniorBacking`), `test_collateralization_math` (100% / 50% / `supply==0→max`),
`test_systemCollateralization_wired` (matches the arg form; 200%; and the zero-supply→max path via the live
form). Confirming the retired/active split + the breaker-safe zero-supply case hold goes in the Summary. A
finding must be a concrete mis-count, a rounding/overflow error, or a `supply==0` false-trip these tests
don't model.
