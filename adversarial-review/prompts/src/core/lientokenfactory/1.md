# Mission 1 — The squatter adversary (can a foreign caller occupy the controller's slot?)

MISSION: 1 — the caller-bound CREATE2 derivation: squat-proofing + keying (I-1/I-2/I-3)

## Who you are
You attack the entire reason this factory's `create` can be permissionless: the deployed address must be keyed on
BOTH `lienId` AND the caller, so a griefer who front-runs origination by calling `create(lienId)` first lands at a
DIFFERENT address and cannot pre-occupy the controller's predicted slot. Your goal: ONE path where a FOREIGN caller
can land at the controller's `(lienId, controller)` address, or where the pure precompute `computeAddress` DISAGREES
with the address `create` actually lands at (which would brick the controller's `PrecomputeMismatch` assert at
`ZipcodeController._origination:234`, or — worse — let a squatter occupy the predicted slot).

## The claims you are attacking
- **I-1 (precompute matches deploy).** `computeAddress(lienId, caller)` (`:37-42`) must equal the address
  `create(lienId)` from `caller` (`:24-30`) lands at. Both must assemble the SAME salt and the SAME init-code.
  **Find a divergence:** `create` salt is `keccak256(abi.encode(lienId))` (`:25`); `computeAddress` salt is
  `keccak256(abi.encode(lienId))` (`:38`) — confirm identical (same `abi.encode`, not one `encode`/one
  `encodePacked`). `create` init-code is `abi.encodePacked(creationCode, abi.encode(msg.sender))` (`:26-27`);
  `computeAddress` init-code is `abi.encodePacked(creationCode, abi.encode(controller))` (`:39-40`) — confirm the
  caller arg (`msg.sender`) and the precompute arg (`controller`) are encoded the SAME way (both `abi.encode`,
  same constructor arg layout). A mismatch (one packs the address, one word-encodes it) would make precompute lie.
- **I-2 (keyed on BOTH `lienId` and `controller`).** A different `lienId` OR a different caller must produce a
  different address. The salt carries `lienId`; the init-code (hence `keccak256(initCode)`, which feeds the CREATE2
  address) carries the caller. **Find a collision:** could two distinct callers, or a caller and a `lienId`, hash to
  the same CREATE2 address? (The address is `keccak256(0xff ++ factory ++ salt ++ keccak256(initCode))[12:]` — a
  collision needs a keccak256 collision, the trusted base. Confirm the factory feeds BOTH inputs in.)
- **I-3 (squat-proof / caller-binding IS the authorization).** Because the init-code embeds `msg.sender` (`:27`),
  an attacker calling `create(lienId)` deploys to the ATTACKER-keyed address, not the controller's. The controller's
  `(lienId, controller)` slot stays empty and free. **Find a path where the binding fails** — e.g. the init-code does
  NOT actually embed the caller (a hardcoded address, a constant, a mis-passed arg), so every caller's token lands at
  the SAME address and the first caller squats it; or the constructor arg the token reads (`LienCollateralToken`
  ctor `:18`, sets `controller = controller_`) is decoded from a DIFFERENT slice than the one the factory packed.

## Specific attacks to run
1. **Byte-for-byte init-code parity.** Diff `create:26-27` against `computeAddress:39-40`. Same `type(...)
   .creationCode`, same `abi.encodePacked(creationCode, abi.encode(<addr>))`, only the address source differs
   (`msg.sender` vs `controller`). Confirm there is no extra byte, no different encoding, no salt drift between the
   two. The `PrecomputeMatchesDeploy` test pins the happy path — your job is a config/edge where they could diverge.
2. **The caller actually reaches the token's `controller`.** The factory packs `abi.encode(msg.sender)` as the
   constructor calldata; `LienCollateralToken`'s ctor (`:18`) takes `address controller_` and sets the immutable
   `controller` (`:21`) + mints 1e18 to it. Confirm the address the factory packs is exactly the address the token
   ends up authorizing — i.e. an attacker's `create` produces a token whose `controller()` is the attacker, NOT the
   controller (so it's worthless for occupying the controller's line). Trace the abi layout: `abi.encode(address)`
   = one left-padded 32-byte word; the ctor reads it as `controller_`.
3. **Front-run / squat construction.** Walk the real flow: `ZipcodeController._origination` (`:232-234`) computes
   `predicted = computeAddress(lienId, address(this))` then `create(lienId)` (caller = the controller) and asserts
   equality. Now have an attacker call `factory.create(lienId)` FIRST, in the same block, before origination. Show
   the attacker's deploy lands at `computeAddress(lienId, attacker)` (a different address), so the controller's
   `predicted` slot is untouched and its `create` still succeeds. Find any way the attacker's deploy could instead
   land at — or block — `computeAddress(lienId, controller)`.
4. **Salt-collision / `encode` vs `encodePacked` on the salt.** `salt = keccak256(abi.encode(lienId))` for a single
   `bytes32` arg — `abi.encode` and `abi.encodePacked` coincide for one fixed-size value, but confirm BOTH sites use
   the SAME one (`:25` and `:38` both `abi.encode`). A mismatch only matters for multi-arg encodes (not the case
   here, one arg) — confirm it's a non-issue rather than assuming it.

## What is already proven (don't re-report as new)
`test_PrecomputeMatchesDeploy` (precompute == deploy, predicted slot empty before, has code after),
`test_PrecomputeKeyedOnLienIdAndController` (distinct lienId → distinct address AND distinct controller → distinct
address), `test_CreateCallerBoundSquatProof` (attacker's token at the attacker slot with attacker authority, the
controller's canonical slot untouched, controller still deploys there). Confirming the derivation is squat-proof goes
in your Summary. A finding must be a concrete foreign-caller-occupies-controller-slot or precompute-disagrees-with-
deploy path the precompute/keying/squat tests do NOT cover — most valuably an init-code/salt assembly divergence
between the two functions, or a hardcoded/mis-passed caller that breaks the binding.
