# Mission 2 ‚Äî The re-mint adversary (single-use-forever + the cross-contract constants)

MISSION: 2 ‚Äî single-use-forever + the cross-contract constants (I-4/I-5/I-6/I-7)

## Who you are
You attack the lien-identity model's other half: a `(lienId, caller)` must be one-shot FOREVER. The same pair must
revert on a re-`create`, and crucially burning the token's entire supply must NOT free the slot ‚Äî the token never
self-destructs, so its address stays occupied. You also attack the two off-chain/cross-contract links the factory is
responsible for: the `LienCreated` event (the only on-chain `lienId -> lien` record, since the factory stores no
mapping) and `LIEN_DECIMALS == 18` (the constant the `ZipcodeOracleRegistry` validates every key's `decimals()`
against). Your goal: ONE path where a retired slot is RE-MINTED, a double-create does NOT revert, or the decimals
constant is INCOHERENT with the token or registry.

## The claims you are attacking
- **I-4 (single-use dedup).** A second `create` with the same `(lienId, caller)` must revert `FailedDeployment`
  (`:28`, via OZ `Create2.deploy` ‚Äî CREATE2 to an address with existing code reverts). **Find a non-reverting
  re-create:** a path where the salt or init-code differs on the second call for the SAME `(lienId, caller)` (it
  must NOT ‚Äî both are pure functions of the args), or where `Create2.deploy` is invoked with a different value/salt
  that dodges the collision. (Note the controller ALSO guards with `LienExists` at `_origination:229` ahead of the
  factory ‚Äî but the factory's `FailedDeployment` is the load-bearing dedup for any direct caller.)
- **I-5 (burn does NOT free the slot).** `LienCollateralToken.burn` (`:32-35`) calls `_burn` (ERC20 supply
  reduction) ‚Äî it does NOT `selfdestruct`; the contract code stays at the address. **Confirm there is no
  self-destruct anywhere in the token** (no `selfdestruct`, no `SELFDESTRUCT` opcode, no delegatecall to a
  destructor) ‚Äî if there were, a burned-then-destructed token would free its CREATE2 slot and let the caller
  re-mint the same `lienId`, breaking single-use-forever. The address occupancy, not the supply, is what retires
  the `lienId`.
- **I-6 (`LienCreated` links `lienId -> lien`).** `event LienCreated(bytes32 indexed lienId, address indexed lien)`
  (`:18`), emitted in `create` (`:29`). Both args indexed. Because the factory stores NO mapping, this event is the
  ONLY on-chain recovery link from a `lienId` to its deployed address (off-chain indexers / the CRE rely on it).
  **Confirm the event emits the ACTUAL deployed address** (`lien`, the `Create2.deploy` return `:28`), not a
  precompute or a stale value, and that it cannot be emitted without a real deploy (it's after `deploy` `:28-29`).
- **I-7 (`LIEN_DECIMALS == 18`, cross-contract pin).** `LIEN_DECIMALS = 18` (`:15`). The token pins `decimals()`
  `pure` to `18` (`LienCollateralToken:26-28`), and the `ZipcodeOracleRegistry` validates every priced key's
  `decimals()` against this constant before caching. **Find an incoherence:** the factory's constant disagreeing
  with the token's `decimals()` override, or with the value the registry expects ‚Äî a mismatch would let the oracle
  infer a wrong scale for a lien.

## Specific attacks to run
1. **Determinism of the re-create.** Both `salt` (`:25`) and `initCode` (`:26-27`) are pure functions of `(lienId,
   msg.sender)`. The second `create(lienId)` from the same caller recomputes the IDENTICAL salt + init-code, so
   `Create2.deploy` targets the SAME address, where code now exists ‚Üí `FailedDeployment`. Confirm there is no
   nonce/counter/timestamp/blockhash in the salt or init-code that would shift the address on a retry and dodge the
   collision. (There is none ‚Äî the inputs are only `lienId` and the caller ‚Äî confirm rather than assume.)
2. **Self-destruct sweep on the token.** Read `LienCollateralToken.sol` top-to-bottom. Confirm: no `selfdestruct`,
   no `delegatecall`, no proxy/upgradeable base (it's a plain immutable `ERC20`), no fallback that could destroy.
   The `burn` (`:32-35`) only reduces supply via `_burn`. So even `totalSupply() == 0` leaves the contract ‚Äî and its
   CREATE2 slot ‚Äî permanently occupied. Construct the re-mint attempt: deploy, burn 1e18 to zero, re-`create` ‚Üí
   still `FailedDeployment` (this is `test_BurnThenRecreateStillReverts`; your job is to find any destruct path that
   would make it NOT revert).
3. **Event integrity.** `LienCreated(lienId, lien)` (`:29`) emits the `lien` returned by `Create2.deploy` (`:28`).
   Confirm an indexer reading the event recovers the true deployed address (it equals `computeAddress(lienId,
   emitter-of-create's-caller)`). Confirm the event cannot fire on a failed deploy (a revert in `deploy` aborts the
   tx before `:29`). A wrong/forgeable address in the event would corrupt off-chain recovery since there's no
   mapping fallback.
4. **Decimals coherence across the three contracts.** `factory.LIEN_DECIMALS()` = `18` (`:15`). `token.decimals()`
   = `18` (pinned `pure`, `LienCollateralToken:26-28`). Confirm both are literally `18` and the registry's
   validation constant is the same `18` (the registry rejects a key whose `decimals()` != the pin). A drift between
   any two of the three would either reject a valid lien or admit a wrong-scaled one. Confirm all three agree.

## What is already proven (don't re-report as new)

`test_BurnThenRecreateStillReverts` (burn to 0 supply, re-create STILL reverts ‚Äî slot retired),
`test_LienCreatedEvent` (the indexed `lienId -> lien` link, emitted address == returned address),
`test_DecimalsPin` (`factory.LIEN_DECIMALS() == 18`; the registry side is in `ZipcodeOracleRegistry.t.sol`).
Confirming a `(lienId, caller)` is single-use-forever and the constants are coherent goes in your Summary. A finding
must be a concrete re-mint-of-a-retired-slot, a double-create-that-doesn't-revert, a forgeable/stale `LienCreated`
address, or a decimals mismatch ‚Äî most valuably a self-destruct/upgrade path on the token that would free the slot,
or a `LIEN_DECIMALS` value that disagrees with the token's `decimals()` or the registry's pin.
