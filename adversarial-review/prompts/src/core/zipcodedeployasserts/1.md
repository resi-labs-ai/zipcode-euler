# Mission 1 — The fail-closed-gate adversary (can a partially-wired fleet slip through?)

MISSION: 1 — per-receiver author+name gate (I-1), the fleet loop + registry seed (I-2/I-3), happy-path freeze
(I-4), and the defended dormancy vuln (I-5/I-6)

## Who you are
You attack the one consequential property of a 26-nSLOC deploy-time fail-closed gate: it must REVERT the deploy
unless EVERY receiver is fully wired (author + workflowName both set) AND the registry's set-once controller is
seeded. The gate runs at `DeployZipcode.s.sol:602`, immediately before the irreversible `transferOwnership`
hand-off (`:604-614`). Your goal: ONE path where a PARTIALLY-wired fleet (a receiver with an unset name, an
unsealed receiver the loop misses, a `controller()==0` registry) PASSES the gate — letting the deploy seal a
dormant receiver live — OR where a fully-wired fleet falsely REVERTS (a false positive bricking the deploy).
The "gate exists to prevent X" framing is NOT a finding; the dormancy hazard is the thing the gate defends.

## The claims you are attacking
- **I-1 (per-receiver gate — author AND name both load-bearing).** `requireReceiverIdentityWired` (`:49-54`)
  reverts `ReceiverIdentityNotWired` unless BOTH `getExpectedAuthor() != address(0)` (`:51`) AND
  `getExpectedWorkflowName() != bytes10(0)` (`:52`). The `||` means EITHER unset trips it. **Confirm both are
  required** — author alone (name unset) and name alone (author unset) BOTH revert. The reason both matter:
  `ReceiverTemplate.onReport` (`:107-115`) enforces the name only when the author is also set, and author alone
  cannot separate two co-tenant daemons sharing the deploy wallet (the K5 reason). Find any way ONE set slot
  alone passes the gate (it can't — `||` — but confirm there is no short-circuit or type-coercion that lets a
  zero slot read non-zero).
- **I-2 (fleet loop — EACH receiver individually).** `requireIdentityWired` (`:63-70`) loops `receivers`
  (`:64-66`) and asserts EACH. CTR-16 DROPPED the old "one representative `workflowId` ⇒ all wired" inference;
  under the separate-daemon model each receiver carries its OWN name, so a missing name on ANY one must fail
  closed. **Confirm the loop visits every index** (`k < receivers.length`, `:64`) and reverts on the FIRST
  unwired receiver — a partially-wired fleet (one unset name among many) cannot pass. Find any path where the
  loop skips an element, an empty array vacuously passes the receiver leg, or a representative-only inference
  survives.
- **I-3 (registry-controller gate).** After the loop, `:67` asserts
  `IOracleRegistryController(registry).controller() != address(0)`, else `RegistryControllerUnset` (`:68`).
  **Confirm a `controller()==0` registry reverts** even when every receiver is sealed (the F7 brick). Find any
  way the check is skipped (e.g. the loop reverting first masks it — that's fine, fail-closed either way — but
  confirm a sealed-fleet + unseeded-registry genuinely reverts at `:67`).
- **I-4 (happy path then freeze).** The gate PASSES (no revert) only when every receiver is sealed AND the
  registry is seeded; then the deploy renounces/transfers and the fleet is correctly frozen. **Confirm the
  happy path passes and is not a false-negative** (it doesn't pass a fleet it should reject) — and that a
  fully-wired fleet does NOT falsely revert (no off-by-one in the loop bound, no spurious zero-read).
- **I-5/I-6 (the defended vuln is real + K5 separation).** A DORMANT (unsealed) receiver ACCEPTS a wrong
  identity (`ReceiverTemplate.onReport:88` not entered ⇒ any Forwarder-cleared report writes the cache); a
  SEALED one REJECTS a wrong name (`InvalidWorkflowName`, `:113`). `workflowName` separates two same-author
  daemons. **Confirm the gate guards a PROVEN hazard, not a hypothetical** — this confirms soundness; it is not
  itself a finding against the gate.

## Specific attacks to run
1. **One-set-slot bypass.** Can a receiver with author set but name unset (or vice-versa) PASS
   `requireReceiverIdentityWired`? The `||` at `:50-52` reverts if EITHER is zero — confirm there's no
   short-circuit bug, and that `bytes10(0)` / `address(0)` comparisons are exact (a workflowName is `bytes10`,
   not `bytes32` — confirm the getter type at `IReceiverIdentity:10` matches `ReceiverTemplate`'s
   `getExpectedWorkflowName` `:66`, so a non-empty name reads non-zero and an unset one reads `bytes10(0)`).
2. **Loop completeness / partial fleet.** Does the loop (`:64-66`) visit EVERY receiver? Construct a fleet
   where exactly ONE receiver has an unset name and the rest are sealed — confirm it reverts pointing at that
   one (the CTR-16 win over the representative-id inference). Check the bound (`k < receivers.length`) and that
   an empty array doesn't vacuously pass BOTH legs (an empty array skips the receiver checks but still hits the
   `:67` registry check — confirm the registry leg is unconditional, not inside the loop).
3. **Registry leg skipped or wrong target.** Is `:67` reached and checked against the RIGHT `registry` arg?
   Confirm the registry is checked even when it is ALSO one of the `receivers` (the NatSpec says the registry
   SHOULD appear in the array too — `:60-61` — so it gets BOTH an identity check as a receiver AND the
   set-once `controller()` check). Confirm checking it twice is harmless (no double-spend, no state — it's a
   view) and that omitting it from the array still leaves the `controller()` check intact.
4. **False-positive revert on a wired fleet.** Could a FULLY-wired fleet falsely revert? E.g. a getter
   returning a non-zero value the gate mis-reads as zero, an off-by-one truncating the loop, or the inline
   `IReceiverIdentity`/`IOracleRegistryController` selectors not matching the real `ReceiverTemplate`/registry
   getters (a selector mismatch would make EVERY call revert at the ABI boundary — confirm the interface
   signatures match `ReceiverTemplate.sol:60/66` and `ZipcodeOracleRegistry.sol:48`).
5. **Process invariant (NOT a code finding here).** The gate only fires when CALLED. Confirm the deploy script
   actually calls it before renounce (`DeployZipcode.s.sol:602`, before the `:604-614` transfer) — this is the
   one residual and it IS covered (the call site + the `DeployZipcode.t.sol:132` echo). Note it as the residual,
   not a finding against the library.

## What is already proven (don't re-report as new)
`test_PreGate_RevertsWhenAuthorUnset` / `_RevertsWhenNameUnset` / `_PassesWhenSealed` (author-OR-name gate),
`test_Negative_ReceiverUnsealed_GateReverts` / `_NameUnset_GateReverts` / `_RegistryControllerUnset_GateReverts`
(the three combined-fleet reverts), `test_Positive_GatePasses_ThenRenounce_ThenFrozen` (happy path + freeze),
the four `test_Behavioral_*` (dormant accepts wrong identity / sealed rejects wrong name / sealed accepts
correct / identity-sealed-after-seal), and `test_K5_PrivilegeSeparation_NameSeparatesSameAuthorDaemons` (same
author, different names) — 12 tests, coverage-complete. Confirming the gate fail-closes on every unwired/unseeded
path, passes only a fully-wired fleet, and defends a real hazard goes in your Summary. A finding must be a
concrete bypass (a partially-wired fleet that PASSES) or a false-positive (a wired fleet that REVERTS) the 12
tests do NOT cover — the "gate exists to prevent the dormancy vuln" framing is soundness, not a finding.
