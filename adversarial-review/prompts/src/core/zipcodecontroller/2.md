# Mission 2 — The draw + close lifecycle adversary (re-anchor, reclaim-before-burn, double-close)

MISSION: 2 — the draw + close lifecycle (I-2/I-3)

## Who you are
You attack the post-origination lifecycle: the additional-draw path (re-anchor → fund → draw on an open line)
and the close path (zero-debt guard → closeLine → burn, with the RECLAIM-BEFORE-BURN sequencing). Your goal:
ONE path where a draw or close corrupts the line — a close with outstanding debt, a burn that runs before the
lien is reclaimed (or a burn that succeeds on a never-held balance), a double-close, a close of a never-opened
lien, or a redraw on a closed line.

## The claims you are attacking
- **I-2 (draw re-anchors + re-funds; fail-closed on unknown/closed).** `_draw` (`:264-283`): decode `(lienId,
  proofRef, equityMark, drawAmount)` → load `r = liens[lienId]` storage → `if (!r.open) revert UnknownLien`
  (`:269`) → re-resolve the SAME venue from the STORED siloId (`_venueFor(r.siloId)`, `:273`) → re-anchor a
  fresh mark `seedPrice(r.lien, equityMark)` (`:276`) → `fund` (`:278`) → `draw` to `erebor` (`:279`). **Find a
  draw on a closed or never-opened line that does NOT revert `UnknownLien`** (the `!r.open` gate is the only
  one — `_close` flips `open=false` but keeps `r.lien`, so a closed line reads `!open` → reverts). Or a
  re-anchor seed that lands but the draw later reverts WITHOUT rolling the seed back (impossible — same frame;
  `test_Draw_ReAnchorBelowLTV_RollsBack` proves a too-low mark reverts `E_AccountLiquidity` and the prior mark
  is intact). Note: `proofRef` is decoded but only referenced (`:282`) for off-chain indexing — confirm it's
  not used in any on-chain decision.
- **I-3 (close: zero-debt guard → closeLine → burn; reclaim-before-burn; single-use).** `_close` (`:286-309`):
  decode `lienId` → load `r` storage → `if (!r.open) revert UnknownLien` (`:290`) → re-resolve venue (`:294`) →
  `if (observeDebt(r.lineRef) != 0) revert DebtOutstanding` (`:296`) → `closeLine` reclaims the 1e18 lien to
  the controller (`:300`) → `burn(FULL_LIEN)` (`:301`) → flip `r.open = false`, keep `r.lien` (`:303-304`) →
  `decrementLineCount` (`:308`). **The reclaim-before-burn is load-bearing:** `closeLine` must reclaim the lien
  to the controller BEFORE `burn`, else `burn` reverts `ERC20InsufficientBalance` (the controller holds 0 — the
  lien is in the escrow vault). Find an ordering where `burn` runs first, or where the burn succeeds without a
  prior reclaim. (`test_Close_BurnAfterReclaim_Sequencing` mocks `closeLine` to no-op and proves `burn` then
  reverts — pinning the dependency.)

## Specific attacks to run
1. **The zero-debt guard.** `observeDebt(r.lineRef) != 0 → DebtOutstanding` (`:296`). The debt is zeroed by a
   PERMISSIONLESS repay (any EOA — `_repay` in the suite uses `makeAddr("permissionlessRepayer")`; borrow is
   hook-gated to the adapter, so an attacker can repay but not add debt — `test_Close_RepayCannotAddDebt`).
   **Find a close that passes the guard while debt is actually outstanding** — a stale `observeDebt` read, a
   reentrancy that adds debt between the read and `closeLine` (no — same frame, the venue is trusted no-callback),
   or an `observeDebt` that reads a different line than the one closed (it reads `r.lineRef`, the stored handle —
   confirm). (`test_Close_DebtOutstanding_StateUnchanged` proves a close with debt reverts and leaves
   `totalSupply == 1e18` + `open == true`.)
2. **Double-close + never-opened + redraw-on-closed.** After a close, `r.open == false` but `r.lien` stays set.
   Confirm: a second close reverts `UnknownLien` (`:290`, `test_Close_DoubleClose_Reverts`); a close of a
   never-opened lien reverts `UnknownLien` (`:290`, `test_Close_NeverOpened_Reverts`); a draw on a closed line
   reverts `UnknownLien` (`:269`, `test_Draw_OnClosedLine_Reverts`). **Find a state where `!r.open` is false but
   the line is actually closed** (a path that flips `open` back to true post-close — there is none; only
   `_origination` sets `open=true`, and it's dup-guarded on `r.lien != 0`, which close keeps set → a closed
   lienId can NEVER be re-originated or re-opened). Reason whether that permanent strand of a closed lienId is
   intended (it is — single-use lienId forever, F-12) or a liveness hole.
3. **The re-resolve-from-stored-siloId on draw/close.** Both `_draw` (`:273`) and `_close` (`:294`) call
   `_venueFor(r.siloId)`, NOT the global `venue` slot and NOT `currentSilo`. **Confirm a draw/close routes to
   the line's ORIGINAL venue even if the silo was re-pointed or retired** (mission 4 owns the routing depth;
   here just confirm the lifecycle paths use `r.siloId`, not a mutable pointer). A draw/close that resolved a
   DIFFERENT venue than origination would fund/draw/close the wrong market — find any path where `r.siloId` is
   mutated post-origination (it's written once in `_origination:254` and never re-written).
4. **The burn target + amount.** `burn(FULL_LIEN)` burns exactly 1e18 from the controller's balance
   (`LienCollateralToken.burn:32` → `_burn(controller, amount)`, controller-only). After the reclaim the
   controller holds exactly 1e18. **Find a partial-burn or over-burn path** — `FULL_LIEN` is a constant 1e18,
   the token's total supply is exactly 1e18 minted once; confirm the burn zeroes the supply
   (`test_Close_L7L8_RepayThenRelease` asserts `totalSupply == 0` + escrow drained post-close).
