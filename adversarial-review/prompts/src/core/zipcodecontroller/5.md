# Mission 5 — The identity-gate, reentrancy & build-phase wiring adversary

MISSION: 5 — identity gates, reentrancy, and build-phase wiring (I-8/I-9/I-11)

## Who you are
You attack the access-control, reentrancy, and configuration surface: the CRE Forwarder + workflow-identity
gate inherited from `ReceiverTemplate`, the structural reentrancy-impossibility (last-write ordering + a
trusted no-callback registry), and the 5 `onlyOwner` wiring setters + the 4 ctor zero-guards. Your goal: ONE
path where a non-Forwarder drives a report, a malicious venue re-enters to corrupt state, or a wiring re-point
DRAINS (rather than merely redirects/griefs).

## The claims you are attacking
- **I-9 (identity gates).** The inbound `onReport` (`ReceiverTemplate:78`) gates on the Forwarder
  (`s_forwarderAddress != 0 && msg.sender != s_forwarderAddress → revert InvalidSender`, `:83`) THEN the
  optional workflow identity (id/author/name, `:88-117`) BEFORE calling `_processReport` (`:119`). The
  controller does NOT override `onReport` — it inherits the base's gate and only overrides `_processReport`.
  **Find a non-Forwarder path into `_processReport`** (there is none — `_processReport` is `internal`, reachable
  only through the gated `onReport`; `test_Authority_NonForwarder_Reverts` proves a non-Forwarder reverts
  `InvalidSender`). The workflow-identity gate is DORMANT by default (all expectations zero → any report passes
  the identity check; `test_DormantGate_Demonstration`): confirm the dormant gate is a documented posture (the
  Forwarder alone is the live gate; the workflow pins are opt-in, set by the Timelock). Post-renounce, the
  parent `onlyOwner` setters revert (`test_PostRenounce_SettersRevert`).
- **I-8 (reentrancy structurally impossible).** The `liens` write is the LAST state mutation (`:254`) before
  the trusted no-callback registry increment (`:260`); the registry makes NO callback into the controller
  (`SiloRegistry.incrementLineCount` is a pure storage bump, `:240-244`). A malicious venue's `openLine` could
  try to re-enter `onReport` — but the Forwarder gate rejects it (the venue is not the Forwarder).
  **Find a reentrancy that corrupts state** — there is no `nonReentrant` modifier (it's structural, not a
  guard): the only inbound is Forwarder-gated, and a re-entrant call from a (trusted) venue would itself need to
  be the Forwarder. `test_Reentrancy_Impossible` stands up a `ReentrantVenue` whose `openLine` tries to
  re-enter `onReport` and confirms the Forwarder gate rejects it (`reentered == false`). This is
  defense-in-depth — the venue is trusted infra. Reason about whether ANY ordering (e.g. a venue callback
  between the `liens` write and the increment) could corrupt state given the venue is trusted and the registry
  is no-callback.
- **I-11 (build-phase wiring setters + ctor guards).** The 5 setters (`setVenue`/`setLienFactory`/
  `setOracleRegistry`/`setErebor`/`setRegistry`, `:141-174`) are each `onlyOwner` + `ZeroAddress`-guarded +
  emit `WiringSet`; pure store-and-emit (no external call). The ctor (`:121-136`) `require`-zero-guards the 4
  non-forwarder args (`:128-131`); the Forwarder zero-guard is the parent's (`ReceiverTemplate:45`); `registry`
  is NOT in the ctor (starts zero, wired post-deploy via `setRegistry` — fail-closed `RegistryUnset` until
  wired). **Find a setter that escapes the owner gate, accepts zero, or whose re-point DRAINS.** Pressure-test
  the re-point blast radius (below).

## Specific attacks to run
1. **Operator/sender escalation.** Confirm there is no non-Forwarder, non-owner path that mutates state. The
   only state-mutating entrypoints are `onReport` (Forwarder-gated) and the 5 setters + the 4 parent setters
   (all `onlyOwner`). A `getLien` is view. **Find a missing gate** on any mutator (there is none).
2. **The wiring re-point blast radius (the headline INFO question).** Each setter re-points a trusted dependency
   (the venue adapter, the factory, the oracle registry, the off-ramp, the silo registry). Pressure-test each:
   - `setVenue` (`:141`) — the `venue` slot is DEAD for report routing (mission 4: routes route via the
     registry). A re-point of `venue` has no on-chain effect on a report. INFO (dead-state re-point).
   - `setLienFactory` (`:148`) — a future origination would `create`/`computeAddress` against the new factory;
     it can't drain an EXISTING lien (the lien token's authority is the controller, immutable). A factory that
     returns a mismatched address trips `PrecomputeMismatch`. INFO unless you show a drain of a live lien.
   - `setOracleRegistry` (`:155`) — a re-point changes where future seeds land; an existing line's mark lives in
     the OLD registry. A re-point to an attacker registry could mis-seed a NEW line's mark (under-collateralize)
     — but that's a fresh-line valuation issue gated by the LTV check at draw, and the registry is trusted /
     Timelock-set (X-3). Reason whether a re-point DRAINS an existing line (it can't — existing debt/marks are
     in the old registry; the new one only feeds new seeds).
   - `setErebor` (`:162`) — the draw receiver. A re-point sends FUTURE draws to a new off-ramp; existing debt is
     unaffected. The venue backstops `receiver == erebor` (F2) — so the controller can only draw to the wired
     `erebor`. A Timelock re-point to an attacker off-ramp would route future draws there — but draws are
     CRE-authorized (a draw report is needed) and the off-ramp is Timelock-set (X-3). Frame whether this is a
     redirect (INFO, governed) or a drain (would need an un-authorized draw — there is none).
   - `setRegistry` (`:170`) — re-points the routing/slot registry. A re-point to an attacker registry could
     return an arbitrary `venueOf` for a NEW origination (routing a new line to an attacker venue) — but the new
     venue is trusted infra by assumption, and the re-point is Timelock-only (X-3). For EXISTING lines,
     draw/close re-resolve `venueOf(r.siloId)` against the NEW registry — so a registry re-point COULD change
     where an open line draws/closes. **This is the sharpest wiring consequence — work it:** does re-pointing
     the registry to one that returns a different (attacker) adapter for an open line's `r.siloId` let a
     draw/close hit an attacker venue? Rate it: it's `onlyOwner`/Timelock (X-3), closed by the pre-prod
     re-freeze, and the attacker venue would still need the per-line EVC operator grant (which it lacks) to move
     real value — so frame whether it's a DRAIN or a governed redirect/grief (INFO unless a concrete drain).
3. **The ctor zero-guards.** `test_I11_Ctor_ZeroGuards` zeroes each non-forwarder arg in turn (4 distinct
   `require` strings). The Forwarder zero-guard is the parent's `InvalidForwarderAddress` (`ReceiverTemplate:45`).
   **Find an arg that can be zero post-construction** (the setters all `ZeroAddress`-guard; `registry` starts
   zero but fails closed). Confirm a zero `registry` can never route (it reverts `RegistryUnset`, not a
   zero-venue call).
4. **The dormant identity gate.** Confirm the dormant workflow gate (all expectations zero) is the intended
   posture: the Forwarder alone gates inbound; the workflow id/author/name pins are opt-in hardening the Timelock
   may set later. A finding that "the workflow gate is dormant" is INFO (documented posture) — NOT a vuln,
   because the Forwarder gate is the live access control and the CRE Forwarder is a trusted, identity-gated
   dependency.
