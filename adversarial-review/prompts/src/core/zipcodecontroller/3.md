# Mission 3 — The report-type allow-list adversary (can reportType 3 — or a marker — do more than it should?)

MISSION: 3 — the report-type allow-list + dispatch (I-4)

## Who you are
You attack the dispatcher: the claim that `_processReport` accepts EXACTLY origination/draw/close as effectful,
treats default/liquidation as status-marker-only, and rejects everything else — notably reportType 3
(revaluation), which is delivered DIRECT to the registry (§4.1) and must NOT be processed here. Your goal: ONE
path where reportType 3 (or 0/7/255 or a truncated payload) is processed instead of reverting, or where a
default/liquidation marker does more than emit an event.

## The claims you are attacking
- **I-4 (report-type allow-list).** `_processReport` (`:192-208`) decodes `(uint8 reportType, bytes payload)`
  (`:193`) then dispatches: `RT_ORIGINATION`=1 → `_origination` (`:195-196`); `RT_DRAW`=2 → `_draw` (`:197-198`);
  `RT_CLOSE`=4 → `_close` (`:199-200`); `RT_DEFAULT`=5 OR `RT_LIQUIDATION`=6 → decode `(bytes32 lienId, uint8
  status)` and `emit LienStatusUpdated` ONLY (`:201-204`); EVERYTHING else → `revert UnsupportedReportType`
  (`:206`). **Find a reportType that is neither 1/2/4/5/6 yet reaches an effectful branch** — there is no
  fall-through (the chain is `if/else if/.../else revert`), so 0, 3, 7, 255, and every other `uint8` hit the
  terminal revert. Confirm reportType 3 (the revaluation type the registry handles via ITS `_processReport`,
  `ZipcodeOracleRegistry:128-138`) reverts here (`test_Dispatch_ReportType3_Rejected`).
- **The marker branch does ONLY emit.** RT=5/6 decode `(lienId, status)` and `emit LienStatusUpdated(lienId,
  status)` (`:203-204`) — M1 status-marker only, NO markdown / escrow / `venue.liquidate` (the comment `:202`
  notes DefaultCoordinator is M2). **Find a marker path that touches the venue, the registry, or the lien
  record** (it doesn't — the branch has no external call beyond the event; `test_Markers_DefaultAndLiquidation_
  StatusOnly` mocks `liquidate` to REVERT and proves it's never reached, and asserts `open`/`debt` unchanged).
  Can a marker for a non-existent lienId corrupt state? No — it only emits; reason whether emitting a status for
  an unknown/closed lienId is a problem (it's an indexing event, no on-chain effect).
- **The truncated / malformed payload.** A type-1 report with an empty (or short) payload: the inner
  `abi.decode` of `_origination` (`:213-222`, decoding 8 fields) bounds-checks and reverts (fails closed — no
  zero-filled origination; `test_Dispatch_TruncatedPayload_Reverts`). **Find a malformed payload that decodes to
  a partially-valid origination/draw/close** instead of reverting — e.g. a payload long enough to decode but
  with garbage that skips a guard. Reason about `abi.decode`'s bounds semantics (it reverts on under-length, and
  over-length trailing bytes are ignored — confirm trailing garbage can't inject a second action).

## Specific attacks to run
1. **The full reportType sweep.** Enumerate the dispatch: 1/2/4 are effectful, 5/6 are markers, ALL others
   revert `UnsupportedReportType(reportType)` carrying the rejected type. Confirm there is no numeric gap that
   falls through (the `else` is terminal). Pay attention to reportType 3 specifically — it's a REAL type in the
   §4.4 ABI but is routed to the registry, not the controller; a controller that processed it would be a
   cross-component confusion. (`test_Dispatch_ReportType0_Rejected`, `_ReportType7And255_Rejected`,
   `_ReportType3_Rejected`.)
2. **The decode discriminant.** The outer decode is `(uint8 reportType, bytes payload)` (`:193`). **Find a
   report whose outer encoding lets `reportType` be mis-read** — e.g. an ABI encoding where the `uint8` is not
   what the dispatcher branches on. `abi.decode(report, (uint8, bytes))` is strict standard-ABI; a CRE report is
   `abi.encode(uint8, bytes)`. Confirm there's no packed-vs-standard ambiguity (the producer uses `abi.encode`,
   not `encodePacked` — the test helpers `_origReport`/`_drawReport`/`_closeReport` all use `abi.encode`).
3. **The marker no-op proof.** Walk RT=5 and RT=6: both land in the same branch (`:201`), decode `(bytes32, uint8)`
   and emit. **Find any state mutation in that branch** — there is none. Then reason: is the marker's lack of
   on-chain effect a GAP (a default/liquidation should freeze/mark-down the line)? No — that's deferred to M2's
   DefaultCoordinator BY DESIGN (`:202` comment); the controller's job in M1 is to NOT act on it. A finding that
   the marker "should do more" is a design observation (INFO), not a vuln.
4. **Gate ordering vs the identity gate.** `_processReport` runs AFTER the `ReceiverTemplate.onReport` Forwarder
   + identity gate (`ReceiverTemplate:78-119`). Confirm an unsupported type still requires the Forwarder gate
   first (`test_DormantGate_Demonstration` uses a type-3 report to prove a report got PAST the identity gate by
   observing the `UnsupportedReportType(3)` revert — i.e. the identity gate is upstream of dispatch). A finding
   that the allow-list is reachable by a non-Forwarder is mission 5's territory; here just confirm dispatch is
   downstream of the gate.
