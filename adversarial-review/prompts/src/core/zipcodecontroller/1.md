# Mission 1 — The orphan-batch adversary (can a mid-batch revert leave a lien/market/seed/count behind?)

MISSION: 1 — atomic origination + dup-no-double-mint (I-1/I-10)

## Who you are
You attack the single most important property of an orchestrator: an origination is `create → openLine → seed →
setLineLimits → fund → draw → store → incrementLineCount`, and ANY revert must roll back the WHOLE batch —
including the CREATE2 deploys (the lien token AND the per-line LineAccount/market). Your goal: ONE path where a
mid-batch revert leaves a residue — an orphan lien token, an orphan market, an orphan oracle seed, or a phantom
line count — OR a double-mint on a re-used lienId.

## The claims you are attacking
- **I-1 (atomic origination).** `_origination` (`:212-261`) is the 9-step batch. Walk it in order: `_venueFor`
  (`:226`) → `LienExists` dup-guard (`:229`) → `computeAddress` (`:232`) + `create` (`:233`) + `PrecomputeMismatch`
  assert (`:234`) → approve exactly `FULL_LIEN` (`:237`) → `openLine` (`:240`) → `seedPrice` on the
  openLine-returned `oracleKey` (`:243`) → `setLineLimits` (`:246`) → `fund` (`:250`) + `draw` to `erebor`
  (`:251`) → store record, LAST write (`:254`) → `incrementLineCount`, FINAL (`:260`). Because all of this runs
  in ONE call frame, a revert at any step reverts the whole transaction (incl. the CREATE2 deploys, which are
  state changes in the same frame). **Find a step whose effect persists across a later revert** — a deploy that
  isn't rolled back, a seed that survives, a count that leaks. (`test_Atomicity_MidBatchRevert_ZeroMark_
  RollsBackDeploys` drives a zero-mark seed revert AFTER both CREATE2 deploys and proves both roll back; find
  an incident IT does not model.)
- **I-10 (dup origination no double-mint).** The `LienExists` guard (`:229`) reverts if `liens[lienId].lien !=
  address(0)`, and the factory ALSO reverts `FailedDeployment` on a re-used (lienId, caller) CREATE2 slot
  (`LienTokenFactory.create:24-30`). **Find a path that mints a second token for a live lienId, or that
  corrupts the first record on a dup attempt.** Note the guard reads `liens[lienId].lien`, which `_close` keeps
  set forever (`:303-304`, single-use lienId) — so a CLOSED lienId can never be re-originated either; confirm
  that's intentional and not a strand.
- **The draw bound (no controller pre-check).** Step 7 (`:248-251`): the controller does NOT pre-check LTV/cap.
  The draw's on-chain LTV/cap bound (the EVK account-status check, behind the seam) is the ONLY gate. **Confirm
  a draw that escapes its on-chain bound is impossible** — i.e. an over-LTV or over-cap draw must REVERT inside
  the venue (rolling back the batch), never silently succeed. (`test_Atomicity_LateRevert_OverLTV` asserts the
  EXACT `E_AccountLiquidity` selector — proving the mock pre-funded cash and the failure is the LTV check, not
  a cash shortfall; `test_Atomicity_CapOnlyBound` asserts `E_BorrowCapExceeded`, mark-independent.)

## Specific attacks to run
1. **The orphan hunt.** For each revert point (dup, precompute mismatch, zero-mark seed, over-LTV draw, over-cap
   draw, `SiloFull`), confirm the post-state has NO residue: `predictedLien.code.length == 0` (no orphan token),
   `liens[lienId].lien == 0` (no record), the registry has no seed for the predicted lien, and the line count
   is unbumped. `_assertNoOrphan` (`:590`) does this for the late/mid/cap reverts — reason about whether a
   revert at a DIFFERENT step (e.g. `setLineLimits` reverting, or `fund` reverting) could leave a residue the
   helper doesn't check (it can't if the whole frame reverts — confirm there's no `try/catch` or low-level call
   that swallows a sub-call revert; there is none — every venue/factory/oracle call is a direct external call
   that propagates).
2. **The precompute assert.** `predicted = computeAddress(lienId, address(this))` (`:232`); `lien = create(lienId)`
   (`:233`); `if (lien != predicted) revert PrecomputeMismatch()` (`:234`). Both derive from `(lienId, this)` so
   they're equal by construction. **Find a path where `create` returns an address != `predicted`** — e.g. a
   factory re-point (`setLienFactory`) between the compute and create (impossible — same frame, no reentrancy),
   or a salt/init-code divergence between `computeAddress` and `create` (compare the two: both use `salt =
   keccak256(abi.encode(lienId))` and init-code `creationCode + abi.encode(controller/msg.sender)` — confirm
   `controller == msg.sender == this`).
3. **The approve scope.** Step 3 approves exactly `FULL_LIEN`=1e18 to `venue_` (`:237`). `openLine` pulls the
   escrow deposit. **Confirm no standing allowance is left** (the F-7 obligation; `test_Origination_L4_
   FullTranscript` asserts `allowance(controller, adapter) == 0` post-batch — the venue's deposit consumed it).
   Find a path where the approve is set but NOT consumed (a venue that pulls < 1e18) leaving a residual
   allowance — and reason whether that's reachable given the venue is trusted to pull exactly the full lien.
4. **The dup + double-mint.** Re-originate a live lienId: confirm `LienExists` fires BEFORE `create`, so no
   second deploy is even attempted, and the first record/token are untouched (`test_Dispatch_DuplicateOrigination_
   NoDoubleMint` asserts `totalSupply == 1e18` + first record intact). Then probe: can two DIFFERENT lienIds
   collide to the same CREATE2 address (salt = `keccak256(abi.encode(lienId))` — a 256-bit space; collision is
   infeasible). Can the dup-guard be bypassed by a record whose `.lien` is zero but `.open` is true (impossible
   — `create` always sets `.lien` non-zero in the same write)?
5. **The store-then-increment ordering.** The record write is LAST before the trusted registry increment
   (`:254` then `:260`). The increment is the FINAL statement so a `SiloFull` revert rolls back the whole batch
   (`test_CTR03_SiloFull_RollsBackOrigination` + `_assertNoOrphan`). **Confirm the ordering can't strand:** a
   `SiloFull` after the record write still reverts the record write (same frame). Reason whether ANY ordering of
   store-vs-increment could leave a counted-but-unrecorded or recorded-but-uncounted line (it can't — both are
   in the atomic frame; a revert undoes both).

## What is already proven (don't re-report as new)
`test_Origination_L4_FullTranscript`, `test_Origination_EmitsExpectedEvents`, `test_Atomicity_LateRevert_OverLTV`
(exact `E_AccountLiquidity`), `test_Atomicity_MidBatchRevert_ZeroMark_RollsBackDeploys` (proves both CREATE2
deploys roll back + re-origination of the same lienId then succeeds), `test_Atomicity_CapOnlyBound` (exact
`E_BorrowCapExceeded`), `test_Dispatch_DuplicateOrigination_NoDoubleMint`. Confirming origination is atomic +
no double-mint goes in your Summary. A finding must be a concrete orphan / double-mint / escaped-draw-bound the
atomicity cluster does NOT cover — most valuably a revert at a step the `_assertNoOrphan` helper doesn't drive,
or a residue that survives a frame revert.
