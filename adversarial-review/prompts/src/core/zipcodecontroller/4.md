# Mission 4 — The CTR-03 routing & slot-accounting adversary (can a re-pointed silo strand or misroute a line?)

MISSION: 4 — CTR-03 fail-closed routing + slot accounting + revolving (I-5/I-6/I-7)

## Who you are
You attack the multi-silo routing layer: the claim that draws/closes re-resolve the venue from the line's
STORED `siloId` via the `SiloRegistry` (never a mutable global pointer), that the unwired registry fails closed
rather than falling back to the `venue` slot, and that the line count increments on open / decrements on close
with `SiloFull` rolling back the whole origination. Your goal: ONE path where a line is stranded or misrouted
by a silo re-point/retire, a silent fallback to the `venue` slot, or a phantom/leaked line count.

## The claims you are attacking
- **I-5 (CTR-03 fail-closed routing).** Every report path resolves the venue via `_venueFor` (`:178-182`):
  `if (registry == address(0)) revert RegistryUnset` (`:179`) → `v = venueOf(siloId)` (`:180`) → `if (v ==
  address(0)) revert SiloUnrouted(siloId)` (`:181`). Origination resolves from the report's `siloId` (`:226`);
  draw/close re-resolve from the line's STORED `r.siloId` (`:273`/`:294`), NEVER the global `venue` slot. The
  `venue` slot exists (set in the ctor / `setVenue`) but the report paths route EXCLUSIVELY via the registry —
  the comment `:224-225` is explicit ("do NOT shadow the `venue` state var"). **Find a path where a report
  reaches the `venue` slot as a fallback** (it doesn't — `_venueFor` reverts `RegistryUnset` on a zero registry
  rather than falling back; `test_CTR03_RegistryUnset_Reverts`). Or a draw/close that resolves a DIFFERENT venue
  than origination because the silo was re-pointed (it can't — `venueOf(r.siloId)` returns the silo's
  CURRENTLY-registered adapter, but the line's `r.siloId` is immutable; if the registry's `silos[siloId].adapter`
  is re-pointed, BOTH origination and later draws/closes resolve to the new adapter — reason carefully about
  whether re-pointing a silo's adapter mid-life strands an open line, and rate it: the adapter is trusted infra,
  re-pointing is `onlyOwner`/Timelock, X-3, INFO unless it drains).
- **The retire-then-close path.** `venueOf` (`SiloRegistry:257`) returns `silos[siloId].adapter` and IGNORES
  `active` — so a retired silo's lines still resolve their adapter and CAN close (`test_CTR03_OpenLine_
  ClosesAfterSiloRetired`: retire clears `currentSilo`, but `venueOf` survives, and the open line closes +
  decrements). **Find a line that can't close after its silo is retired** (it always can — `venueOf` is
  active-agnostic; the only thing retire blocks is NEW originations via `currentSilo`, which the controller
  doesn't even read).
- **I-6 (slot accounting).** `incrementLineCount` on open (FINAL statement, `:260`); `decrementLineCount` on
  close (FINAL, `:308`). The registry's `incrementLineCount` (`SiloRegistry:240`) reverts `SiloFull` at
  `MAX_LINES_PER_SILO=28`; `decrementLineCount` (`:248`) reverts `NoLinesToDecrement` on a zero count. **Find a
  phantom count** (an increment that lands but the origination later reverts — impossible, increment is the LAST
  statement so any later revert is moot, and a `SiloFull` revert at the increment rolls the whole batch back:
  `test_CTR03_SiloFull_RollsBackOrigination` + `_assertNoOrphan`) **or a leaked count** (a close that
  decrements twice, or an origination that increments twice). Confirm one open == one increment, one close ==
  one decrement (`test_CTR03_LineCount_IncrementsAndDecrements`: 0→1→2→1→0).
- **I-7 (revolving / coexistence).** A revolving line opens ONCE then borrow→permissionless-repay→redraw cycles
  on the SAME open line / oracle key / EE slot (no new market minted) — the CRE simply never files RT_CLOSE.
  Per-revolution the only gate is LTV × the mark the CRE supplies; a STALE mark does NOT block a draw (`_draw`
  re-seeds first). **Find a redraw that mints a new market/slot, that uses a different oracle key, or that
  escapes the LTV backstop.** Confirm a repo line (closes + frees its slot) and a revolving line (persists)
  coexist on one pool.

## Specific attacks to run
1. **The fallback-to-venue-slot hunt.** Grep every `IZipcodeVenue(...)` call site in `_origination`/`_draw`/
   `_close`: they all use the local `venue_` from `_venueFor` (`:226`/`:273`/`:294`), never the state `venue`.
   The `venue` slot is set in the ctor and `setVenue` but is NEVER read by a report path. **Confirm the `venue`
   slot is dead for routing** (it's a legacy/build-phase artifact; the report paths are registry-exclusive). A
   finding that the `venue` slot is unused dead state is INFO; a finding that a report reaches it is a real bug
   — find it or confirm it can't.
2. **The silo re-point mid-life.** If `addSilo` registered SILO_A → adapter_1 and a later (impossible without a
   delete — `addSilo` reverts `DuplicateSilo`; there's no `setAdapter`) re-point changed it to adapter_2, would
   an open line on SILO_A draw/close against adapter_2? Trace: the registry has NO adapter-re-point setter (only
   `setController`, `setActive`, `retireSilo`, `setCurrentSilo`, `addSilo`). So a silo's adapter is immutable
   once admitted — a line CANNOT be misrouted by a re-point because there's no re-point. Confirm this closes the
   strand vector structurally.
3. **The `SiloUnrouted` / `RegistryUnset` fail-close.** A zero-resolving siloId reverts `SiloUnrouted`
   (`test_CTR03_UnknownSilo_RevertsSiloUnrouted`); an unwired registry reverts `RegistryUnset`
   (`test_CTR03_RegistryUnset_Reverts`). **Find a zero/unknown siloId that resolves to a usable venue anyway**
   (it can't — `venueOf` returns `silos[siloId].adapter` which is zero for an unknown silo → `SiloUnrouted`).
4. **The count-leak via a partial close.** `_close` decrements at `:308` AFTER the burn + `open=false`. A
   double-close reverts `UnknownLien` BEFORE the decrement (`:290`), so the count can't be decremented twice for
   one line. **Find a path that decrements without a matching increment** (a close of a line whose origination's
   increment reverted — impossible, the line wouldn't exist). The `NoLinesToDecrement` guard
   (`SiloRegistry:250`) is a backstop against an underflow. Reason whether the CTR-04 lifetime-vs-concurrent
   caveat (`ZipcodeController:71-75`) is a controller bug (it's not — it's an adapter `closeLine` limitation,
   noted as a cross-ticket dependency; the controller's decrement is correct REGISTRY accounting).
5. **The revolving redraw.** Trace `test_Revolving_BorrowRepayRedraw_SameSlot`: same `lineRef`, same `lien`
   (oracle key), same `borrowAccount`, supply+withdraw queues unchanged across cycles. **Find a redraw that
   perturbs the slot** (it can't — `_draw` re-uses `r.lineRef`/`r.lien`, opens nothing). The LTV backstop:
   `test_Revolving_LtvBackstopOnRedraw` proves a too-low-mark redraw reverts `E_AccountLiquidity` and rolls back.
