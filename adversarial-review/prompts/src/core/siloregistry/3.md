# Mission 3 — The lifecycle adversary (can a silo lose its record or strand a close path?)

MISSION: 3 — silo lifecycle + the UnknownSilo branch (I-7, I-8, I-12, I-13)

## Who you are
You attack the silo lifecycle: the claim that retiring a silo stops new routing but KEEPS the record (so its
open lines can still close), that `setCurrentSilo` is a governed rollover that never lands on an unknown or
inactive silo, that the `currentSilo` sentinel can never strand at a retired/inactive silo, and that the
`UnknownSilo` guard is present on EVERY by-id function. Your goal: ONE path where a retired silo loses its
record (stranding its open lines' close path), where `currentSilo` ends up pointing at a retired/inactive
silo, or where an `UnknownSilo` branch is missing on one function.

## The claims you are attacking
- **I-7 (retire stops routing, keeps record, clears sentinel if current).** `retireSilo` (`:209`, onlyOwner):
  reverts `UnknownSilo` (`:210`), sets `active = false` (`:211`, NEVER deletes the record), and — IF it was
  `currentSilo` — clears the sentinel to `bytes32(0)` (`:212-215`) so the next origination forces an explicit
  `setCurrentSilo`. **Confirm the record survives** (no `delete silos[siloId]`, the `siloIds` array keeps the
  id) so an open line's close path can still read `venueOf`/`getSilo`. **Confirm a non-current retire leaves
  `currentSilo` untouched** (`:212` is conditional).
- **I-8 (`setCurrentSilo` governed rollover).** `setCurrentSilo` (`:228`, onlyOwner): reverts `UnknownSilo`
  (`:229`) and `SiloInactive` (`:230`) — the registry **never auto-rollovers**; the rollover is explicit +
  governed. **Find a path where `currentSilo` is set to an unknown or inactive silo**, or where it
  auto-advances without an owner call.
- **I-12 (`UnknownSilo` on ALL state-mutating by-id functions).** The guard `silos[siloId].adapter ==
  address(0)` must be on all five: `retireSilo` (`:210`), `setActive` (`:221`), `setCurrentSilo` (`:229`),
  `incrementLineCount` (`:241`), `decrementLineCount` (`:249`). **Find a by-id function MISSING the guard** —
  one that would silently operate on (or initialize) a phantom record.
- **I-13 (`setActive` effect).** `setActive` (`:220`, onlyOwner): reverts `UnknownSilo` (`:221`), flips the flag
  BOTH ways (`:222`, `active = active_` — not a one-way latch), emits `SiloActiveSet` (`:223`). **Confirm the
  flag is readable after the flip** (it is the real storage write, not just the gate) and that re-activating a
  retired silo is possible (false→true).

## The sentinel-stranding scenario (the sharpest lifecycle hazard)
The reserved sentinel `bytes32(0)` means "no current silo" (`:115`, `:165` rejects it as a siloId). Trace the
interaction: `retireSilo` clears the sentinel IF the retired silo was current (`:212-215`), but `setActive(id,
false)` (`:220`) does NOT clear `currentSilo`. **So can `currentSilo` end up pointing at an INACTIVE silo via
`setActive(currentSilo, false)`?** Read whether that's a reachable strand — and whether the NEXT origination
re-checks active before routing (the registry itself never reads `currentSilo` to route; the controller does —
trace whether the controller re-validates `active` at origination, or whether a stranded-inactive `currentSilo`
would route a new line into an inactive silo). Frame: is this a registry hole (the registry should clear the
sentinel on any deactivation of the current silo) or a controller-side validation responsibility? `retireSilo`
clears it but `setActive(_, false)` does not — that asymmetry is the thing to pressure-test.

## Specific attacks to run
1. **Record survival on retire.** Confirm `retireSilo` (`:209-217`) only flips `active` + conditionally clears
   the sentinel — NO `delete`, NO `siloIds` pop. `getSilo`/`venueOf`/`allSiloIds` (`:257-275`) still return the
   retired silo. An open line's close path (`ZipcodeController.closeLine` → `decrementLineCount`) must still
   resolve the venue — confirm a retired silo's lines can close.
2. **The inactive-currentSilo strand.** Set a silo as `currentSilo`, then `setActive(it, false)` — is
   `currentSilo` now pointing at an inactive silo? `setCurrentSilo` rejects inactive at SET time (`:230`) but
   `setActive` can deactivate AFTER it's already current. Determine if anything downstream guards against
   routing into it.
3. **The five `UnknownSilo` guards.** Grep each by-id function's first line: `:210`, `:221`, `:229`, `:241`,
   `:249`. Confirm every one checks `adapter == address(0)`. A missing guard on, e.g., `setActive` would let an
   owner flip `active` on a never-admitted id, writing a phantom record (`active=true`, `adapter=0`) that
   `addSilo`'s `DuplicateSilo` check (`:166`, keyed on `adapter != 0`) would NOT catch — so it could later be
   re-admitted. Confirm the guard closes this.
4. **`setActive` two-way flip.** `:222` is a plain assignment, not `active = false` — confirm a deactivated silo
   can be re-activated (false→true→false) and the count/record are preserved across flips.

`test_setCurrentSilo_unknown_reverts`, `_inactive_reverts`, `_rollover`,

the sentinel clears on a current retire, and the `UnknownSilo` guard is on all five goes in your Summary. A

— most valuably the `setActive(currentSilo, false)` inactive-strand asymmetry (does it route a new line into
an inactive silo, or is that a controller-side validation?).
