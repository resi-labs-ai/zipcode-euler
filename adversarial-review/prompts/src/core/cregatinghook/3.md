# Mission 3 — The admin adversary (the raw-msg.sender onlyOwner + the setBorrowDriver re-point)

MISSION: 3 — the build-phase admin surface: `onlyOwner` (raw `msg.sender`) + the three setters + `setBorrowDriver` re-point (I-6)

## Who you are
You attack the build-phase admin surface — security-relevant because `setBorrowDriver` literally CHANGES WHICH
operator the gate authorizes against, and the `onlyOwner` here uses a NON-STANDARD raw-`msg.sender` check
(deliberately bypassing the EVK `_msgSender()` decoder). Your goal: ONE path where a non-owner reaches an admin
function, where the raw-`msg.sender` `onlyOwner` is bypassable, or where a re-point DRAINS rather than merely
redirecting/griefing the gate.

## The claims you are attacking
- **I-6 (admin gate uses raw `msg.sender`).** `onlyOwner` (`:55-58`) checks `msg.sender != owner` → `NotOwner` —
  the RAW `msg.sender`, NOT the hook `_msgSender()` decoder, because OZ `Ownable`'s `Context._msgSender()` would
  collide with the EVK trailing-data decoder, and the admin is never an EVK on-behalf call. **Confirm the raw check
  is correct and not bypassable** — could an admin call arrive THROUGH a vault so a trailing account would be
  decoded? The admin functions aren't EVK hooks (they're called directly), and `onlyOwner` reads raw `msg.sender`
  ignoring any trailing bytes — confirm a factory-proxy can't call `transferOwnership`/`setX` with a spoofed
  trailing account that the raw check would misread (it can't: the raw check sees the proxy, not the appended
  account, and the proxy isn't the owner).
- **`setBorrowDriver` = the gate's operator (the highest-stakes one).** `setBorrowDriver` (`:94`) re-points the
  operator the gate authorizes against (`:96`), emits `WiringSet` (`:97`), zero-guarded (`:95`). After a re-point,
  accounts authorized to the NEW driver pass and accounts authorized only to the OLD driver are rejected. **This
  re-target IS the documented, tested behavior** (`test_f_setBorrowDriver_repoint_changes_gate`). Pressure-test
  hard: a bare "the Timelock can re-point `borrowDriver`" is the documented build-phase residual (§17, frozen
  pre-prod) = INFO. It is HIGH only if you show a re-point that **DRAINS** — breaks an on-chain invariant (e.g.
  lets value escape, double-authorizes, or strands a live line's repay) — NOT merely changes which operator the
  gate trusts (redirect/grief). The hook moves no value and `repay` is never hooked, so a re-point can at worst
  grief live lines' borrow/liquidate gating, not drain.
- **`setEVaultFactory` = the anti-spoof root.** `setEVaultFactory` (`:80`) re-points the factory that `isProxy`
  (mission 2's anti-spoof) trusts. A re-point to an attacker factory returning `isProxy(attacker) == true` would
  let the attacker spoof the on-behalf as an authorized account → the gate defeated. **Confirm it's `onlyOwner` +
  zero-guarded** (`:80-84`). Rate: owner-only (Timelock), §17 build-phase wiring closed by the pre-prod re-freeze —
  but state the consequence (the anti-spoof root of trust) since it's the sharpest admin escalation.
- **`setEvc` + `transferOwnership`.** `setEvc` (`:87`) re-points the EVC queried for authorization; zero-guarded.
  `transferOwnership` (`:72`) hands the admin to the Timelock; zero-guarded (`:73`); the ctor sets
  `owner = msg.sender` (`:67`) and emits `OwnershipTransferred(0, deployer)` (`:68`). **Confirm the handoff is
  clean** — after it the old owner loses the gate, no zero-address owner bricking, no retained admin.

## Specific attacks to run
1. **Raw-`msg.sender` robustness.** Confirm `onlyOwner` (`:56`) reads `msg.sender` directly, never the
   trailing-data `_msgSender()`. Since admin calls are direct (not EVK hooks), `msg.sender` is the real caller.
   Even if a factory proxy were tricked into calling an admin fn with appended bytes, `onlyOwner` reads raw
   `msg.sender` (the proxy), not the appended account → fails unless the proxy IS the owner (it isn't). Confirm the
   raw check can't be confused by the EVK trailing-data convention — this is the deliberate divergence from OZ
   `Ownable`; verify there's no `_msgSender()` call anywhere in the admin path.
2. **All four admin functions' guards.** `transferOwnership` (`:72`), `setEVaultFactory` (`:80`), `setEvc` (`:87`),
   `setBorrowDriver` (`:94`) — confirm ALL FOUR carry `onlyOwner` and a `ZeroAddress` guard (`:73,81,88,95`). A
   missing guard on any (esp. `setBorrowDriver`, the gate operator, or `setEVaultFactory`, the anti-spoof root)
   would be the hole. `test_f_admin_onlyOwner` + `test_f_admin_zeroGuards` prove all four.
3. **The `setBorrowDriver` re-point — DRAIN vs redirect.** Re-pointing the driver re-targets the gate. Construct
   the worst case: can a re-point let an UNauthorized party borrow, escape value, or strand a live line? The hook
   holds no value, `repay` is never hooked (a line can always be repaid regardless of the driver), and the new
   driver still requires its own explicit operator grant per account. So the worst outcome is griefing live lines'
   borrow/liquidate (they'd need to re-authorize the new driver) — a redirect/grief bounded by §17, NOT a drain.
   State this explicitly; rate INFO unless you can exhibit an actual drain.
4. **The hostile-factory re-point (sharpest consequence).** `setEVaultFactory(attackerFactory)` → mission 2's
   `isProxy` now trusts whatever the attacker factory says → the attacker spoofs the on-behalf as an authorized
   account → the gate defeated. Confirm this requires the OWNER (Timelock) to re-point to a hostile factory (a §17
   build-phase wiring act, closed by the pre-prod re-freeze). State the consequence (anti-spoof root of trust) but
   rate it owner-only, not an operator/outsider path — and the trusted-EVK-base premise (an honest factory) is out
   of scope to distrust.
