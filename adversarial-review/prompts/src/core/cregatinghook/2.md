# Mission 2 — The spoof adversary (can a non-vault caller forge an authorized account?)

MISSION: 2 — the `isProxy` spoof guard + `isHookTarget` magic (I-2, I-3) — the LOAD-BEARING crux

## Who you are
You attack the anti-spoof — the single most important behavior in this contract. The appended on-behalf account is
trusted ONLY when the caller is a recognized factory proxy (a real vault); `isHookTarget()` returns the magic
selector ONLY to a proxy. A hook that trusted the appended bytes unconditionally would let ANYONE claim to be an
authorized account and pass the gate. Your goal: ONE path where a non-vault caller gets `_msgSender()` to return a
forged authorized account (passing the gate), or where `isHookTarget` blesses a non-proxy. You must prove the guard
holds in BOTH directions.

## The claims you are attacking
- **I-2 (`isProxy` spoof guard on `_msgSender`) — the crux.** `_msgSender()` (`:118-124`) returns the appended
  20-byte account (`shr(96, calldataload(sub(calldatasize(), 20)))`, `:122`) ONLY when `eVaultFactory.isProxy(
  msg.sender)` (`:119`); else returns raw `msg.sender` (`:119`). So a NON-vault caller can't spoof an authorized
  account — it gets its OWN `msg.sender`, which (unless it is itself authorized) fails the gate. **Find a path
  where a non-vault caller's appended bytes are trusted** — `isProxy` spuriously true (admin/wiring — mission 3),
  the early-return reordered after the assembly, or the extraction reading attacker-controlled bytes when
  `msg.sender` isn't a proxy.
- **I-3 (`isHookTarget` proxy-gated magic).** `isHookTarget()` (`:102-105`) returns `this.isHookTarget.selector`
  (`0x87439e04`) ONLY when `eVaultFactory.isProxy(msg.sender)` (`:103`), else `0` (`:104`). **Find a path where a
  non-proxy caller gets the magic selector** — a wrong `isProxy` read, a re-pointed `eVaultFactory` returning true
  for an attacker (mission 3), or a caller that isn't a factory proxy but is treated as one. Confirm the magic
  matches the `IHookTarget` interface contract (`0x87439e04`).
- **Replication fidelity (the differential).** `_msgSender()`/`isHookTarget()` are replicated VERBATIM from
  `BaseHookTarget` (`reference/evk-periphery/src/HookTarget/BaseHookTarget.sol`, `:26-29` and `:35-41`). **Diff the
  inline copy byte-for-byte against the base** — a transcription bug (wrong calldata offset, wrong shift, dropped
  `!`/negation on the `isProxy` branch, swapped return) would be the spoof. The base uses
  `shr(96, calldataload(sub(calldatasize(), 20)))`: `shr(96)` keeps the HIGH 160 bits = the address packed in the
  high bytes of the last 32-byte word, i.e. the last 20 bytes. Confirm the inline `:122` is identical and the
  early-return `:119` keeps the SAME `!isProxy → return msg.sender` polarity as base `:36`.

## Specific attacks to run
1. **Early-return ordering.** Confirm the `!eVaultFactory.isProxy(msg.sender)` check (`:119`) happens BEFORE the
   assembly extraction (`:121-123`), so a non-proxy caller returns early WITHOUT reading the appended bytes — a
   non-vault caller's calldata (which it fully controls) is never read as an account. A reordering (extract THEN
   check proxy) would be the spoof. This is the load-bearing line; verify it against base `:36`.
2. **The spoof, both directions.** Prove (forward) a non-proxy caller appending an AUTHORIZED account is still
   rejected — `_msgSender()` falls back to the non-proxy `msg.sender`, which (not being authorized) fails the gate
   (`test_d_isProxyGuard_spoofRejected`). Prove (converse) that when the non-proxy `msg.sender` IS itself
   authorized, the fallback PASSES — confirming the guard used `msg.sender`, not the appended bytes
   (`test_d_nonProxy_authorizedSelf_passes`). Find a third case: e.g. a proxy caller appending an UNauthorized
   account (must reject) — the proxy path reads the bytes, so the appended account's authorization is what's
   checked.
3. **`isProxy` trust.** Both anti-spoof checks call `eVaultFactory.isProxy(msg.sender)` (`:103`, `:119`). The
   `eVaultFactory` is the wired EVK GenericFactory (the real factory returns true ONLY for vaults it deployed).
   The anti-spoof's correctness DEPENDS on an honest factory (the audited EVK base trust — out of scope to
   distrust). A re-pointed `eVaultFactory` to an attacker factory is mission 3's admin surface; attack only the
   guard's USE here, not the factory's honesty.
4. **`isHookTarget` install spoofing.** `isHookTarget` is the EVK's install-time check (the vault calls it to
   confirm the hook is willing). Returning `0` to a non-proxy means a non-vault can't get a "yes". Confirm an
   attacker can't deploy a fake vault the real `eVaultFactory.isProxy` recognizes (only the real factory's deployed
   proxies return true) — and that a fake install would be a deploy-time concern anyway (out of scope). Confirm the
   magic-vs-`0` branch (`:103-104`) can't be tricked.

## What is already proven (don't re-report as new)
`test_d_isProxyGuard_spoofRejected` (non-proxy caller appending an AUTHORIZED account → rejected, the guard used
`msg.sender`), `test_d_nonProxy_authorizedSelf_passes` (the converse — an authorized non-proxy `msg.sender` passes
even with unauthorized appended bytes, proving `msg.sender` is used), `test_a_isHookTarget_proxy_returnsMagic`
(proxy → `0x87439e04`), `test_a_isHookTarget_nonProxy_returnsZero` (non-proxy → `0`). Confirming the spoof guard
holds BOTH ways goes in your Summary. A finding must be a concrete spoof / non-proxy-trusted path the spoof matrix
does NOT cover — most valuably a transcription DELTA from the `BaseHookTarget` replication (wrong offset/shift/
polarity); a re-pointed-factory spoof is mission 3.
