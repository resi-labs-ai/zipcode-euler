# Mission 1 — The operator-authorization gate (can an unauthorized account pass the gate?)

MISSION: 1 — the operator-authorization gate (I-1, I-4, I-5)

## Who you are
You attack the whole reason this hook exists: a `borrow`/`liquidate` on a per-line lien market is allowed ONLY
when the EVK-appended on-behalf account has authorized `borrowDriver` as its EVC operator. Your goal: ONE path
where an UNauthorized account passes the gate — an outsider borrows/levers against a line that is not theirs — or
where the op-agnostic uniformity breaks (an op that reaches the fallback yet skips the check).

## The claims you are attacking
- **I-1 (operator-auth gate).** `fallback()` (`:110-113`) reverts `NotAuthorizedOperator` unless
  `evc.isAccountOperatorAuthorized(_msgSender(), borrowDriver)` is true (`:112`). The appended account must have
  granted the wired `borrowDriver` the operator bit. **Find a borrow/liquidate on a non-authorized on-behalf that
  the gate admits** — a `_msgSender()` that returns an authorized address for an unauthorized borrower (spoof —
  that's mission 2), an op reaching the fallback that skips the predicate, or a hooked path that doesn't invoke
  the hook.
- **The gate is operator-AUTH, not ownership (by design).** Distinct from a `haveCommonOwner` check: the per-line
  borrow account has its own owner-prefix and shares none with `borrowDriver`, so the predicate is
  `isAccountOperatorAuthorized(account, borrowDriver)` exactly (`:112`). **Confirm there is no owner/prefix path**
  that would let an account sharing a prefix with the driver, or the driver itself, pass without the explicit
  operator grant — the check is the literal EVC predicate, not address arithmetic.
- **I-4 (op-agnostic uniform gate).** `fallback()` has NO per-op branch — borrow and liquidate hit the identical
  predicate; the same authorization check applies regardless of the leading 4-byte selector. `repay` is never in
  `hookedOps` (the adapter installs only `OP_BORROW | OP_LIQUIDATE`), so it stays permissionless by construction —
  it simply never reaches this hook. **Find an op that reaches the fallback but is treated differently**, or a
  selector that bypasses the predicate.
- **I-5 (`NotAuthorizedOperator` selector KAT).** The revert selector is `0x3d9adf1c`. Confirm the gate reverts
  with NO return data on failure (the EVK treats a revert as a blocked op) and that a pass returns nothing (the
  EVK treats a non-reverting hook call as allowed).

## Specific attacks to run
1. **The on-behalf extraction → predicate.** `_msgSender()` (`:118-124`) returns the appended 20-byte account when
   `msg.sender` is a factory proxy, else raw `msg.sender`. For a real borrow, the EVK vault (a factory proxy) calls
   the hook with the on-behalf appended → `_msgSender()` returns the borrower account → the gate checks THAT
   against `borrowDriver` (`:112`). Confirm the wired `borrowDriver` is the SAME operator each per-line account
   actually grants (cross-ref `LineAccount`), so an authorized line account passes and any other account fails.
   `test_b`/`test_c` prove the authorized/unauthorized matrix on the mock EVC — find a case they miss.
2. **Op uniformity vs install scope.** The hook is op-agnostic but installed only on `OP_BORROW | OP_LIQUIDATE`
   (deploy/config, out of scope). Confirm the `fallback()` gate doesn't need to distinguish ops (it guards every
   call it receives identically) and that `test_e_opAgnostic_uniformGate` evidences borrow-vs-liquidate identical
   behavior. A finding here must be a path where an installed op (borrow or liquidate) reaches the fallback yet the
   predicate does NOT fire — there is no branch, so confirm the single predicate is unconditional.
3. **No value / no custody.** `fallback()` is non-payable (`:110`, the EVK invokes with no value) and has no side
   effect on a pass. Confirm no value/asset moves through the hook — it's a pure revert-or-allow gate, so even an
   admitted call can only proceed to the (separately-controlled) EVK borrow, never move funds in the hook itself.
4. **The `borrowDriver` identity.** The gate authorizes against ONE wired operator. Confirm a borrow account that
   authorized a DIFFERENT operator (not the wired `borrowDriver`) is rejected — the predicate pins the exact
   `borrowDriver` storage value, so authorization to any other operator does not pass.

## What is already proven (don't re-report as new)
`test_b_fallback_authorizedAccount_passes` (authorized line account passes, no return data),
`test_c_fallback_unauthorizedAccount_reverts` (foreign account → `NotAuthorizedOperator`),
`test_e_opAgnostic_uniformGate` (borrow vs liquidate selectors → identical pass/reject), `test_errorSelector`
(`NotAuthorizedOperator` selector == `0x3d9adf1c`). Confirming the operator-auth gate holds goes in your Summary.
A finding must be a concrete unauthorized-account-admitted path the authorized/unauthorized matrix does NOT cover —
most valuably an op-uniformity break or a `borrowDriver`-identity mismatch (the spoof angle is mission 2).
