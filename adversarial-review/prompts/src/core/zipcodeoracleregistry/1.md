# Mission 1 — The malformed-push adversary (two write paths, all-or-nothing batch, value/staleness guards)

MISSION: 1 — write paths: controller seed + Forwarder reportType-3 batch, all-or-nothing, value/staleness guards

## Who you are
You attack the WRITE side: the two mutators of the cache. Either path landing a malformed, partial, or stale
mark prices a borrow later. The controller and the Forwarder are trusted writers (ratified base) — but a wrong
reportType, a length-mismatched batch, a zero/overflow price, a future/backdated/equal-ts mark, or a partial
revaluation must all be rejected fail-closed. Your goal: find ONE malformed or hostile push that lands in
`cache` (or a PARTIAL revaluation that lands some keys and skips others).

## The claims you are attacking
- **I-2 (seedPrice controller-gated).** `seedPrice` (`:113-117`) must reject any caller `!= controller`
  (`msg.sender != controller → NotController` `:114`) BEFORE writing. Confirm there is no second seed path and
  the gate precedes `_writePrice`. The seed writes `ts = uint48(block.timestamp)` (`:115`).
- **I-3 (revaluation all-or-nothing batch).** `_processReport` (`:128-138`) must: (a) be reachable only via the
  Forwarder-gated `onReport` in `ReceiverTemplate:78-119` (`InvalidSender` for any non-Forwarder caller); (b)
  reject `reportType != REVALUATION(3)` (`InvalidReportType` `:130`) BEFORE decoding the payload; (c) reject
  `liens.length != prices.length` (`LengthMismatch` `:133`); (d) loop `_writePrice` so that a SINGLE poison key
  reverts the WHOLE report (`:134-137`). **The all-or-nothing is the intentional WOOF-02 fail-closed design**
  (`:120-126`): a per-key try/catch is deliberately omitted. Confirm there is no partial-landing path — that a
  revert on key `i` rolls back keys `0..i-1`.
- **I-4 (write-value guards, both paths).** `_writePrice` (`:141-148`) must reject: `price == 0 →
  PriceOracle_InvalidAnswer` (`:142`); `price > type(uint208).max → PriceOracle_Overflow` (`:143`); `ts >
  block.timestamp → FutureTimestamp` (`:144`). **Walk each bound for exactness** — `uint208.max` (not `uint256`),
  a future-ts one second ahead rejected, the `uint208.max` value itself ALLOWED.
- **I-6 (SEC-01 strictly-newer).** `ts <= cache[lien].timestamp → StaleReport` (`:145`). This blocks replay,
  out-of-order reportType-3 delivery, AND a seed-clobber — and a same-`price` replay the value guards CANNOT
  catch. **Pin the comparison: strictly-greater required** (`ts <= cached` reverts), and the first write
  (`cache.timestamp == 0`) is allowed.

## Specific attacks to run
1. **Partial revaluation landing.** This is the headline: trace the loop (`:134-137`). If `_writePrice` on key
   `i` reverts (zero price, overflow, future/stale ts, off-decimal), does the WHOLE call revert and roll back the
   already-written keys `0..i-1`, or can some keys land while others are skipped? Confirm there is NO per-key
   try/catch and NO continue-on-error. Then the duplicate-key case: two entries for the same lien with the same
   `ts` in one batch — the second write hits `ts <= cache.timestamp` (just written by the first) → `StaleReport`
   → whole batch reverts (this is the documented mechanism).
2. **Gate / type ordering.** Confirm `reportType != 3` reverts BEFORE the `abi.decode` of the payload (`:130`
   before `:131`). Can a payload of the wrong shape `abi.decode` into `(address[], uint256[], uint32)` without
   reverting (decode tolerance), landing garbage liens/prices? Push a `reportType == 3` with a truncated /
   mis-shaped payload and reason about whether `abi.decode` reverts or silently mis-parses.
3. **Value-guard bounds.** `price == 0`, `price == uint208.max` (boundary — allowed?), `price == uint208.max+1`
   (must overflow-revert), `ts == now` (allowed on both paths?), `ts == now+1` (future, rejected). Find any
   boundary that admits a value the read path then mis-uses. Confirm the guards fire on BOTH `seedPrice` and the
   reval loop (they share `_writePrice`).
4. **Strictly-newer off-by-one + cross-path clobber.** Is the guard `ts <= cache.timestamp` (correct) or
   `ts < cache.timestamp` (would allow an equal-ts replay/clobber over-crediting)? Construct: a seed at `ts=T`,
   then a reportType-3 mark at `ts=T` for the same lien — must revert `StaleReport` (the seed-clobber defense).
   Then a backdated reval (`ts < T`) — must revert. A `+1` mark succeeds.
5. **Second write path.** Grep the full surface for any function that writes `cache` other than
   `seedPrice → _writePrice` and `onReport → _processReport → _writePrice`. Confirm no inherited
   `ReceiverTemplate` / `BaseAdapter` mutator is exposed, and no public/external setter touches `cache`.
6. **No-value-band classification.** There is NO on-chain plausibility band (I-8): a `_writePrice` that passes
   the value/ts/decimals guards lands regardless of how far the mark moved from the prior cached value. Confirm
   that's the deliberate posture — a "big drop is rejected" expectation would be WRONG. Classify any
   "a fresh-but-wrong mark over-values within the window" as the accepted CRE-trust residual, not a vuln.
