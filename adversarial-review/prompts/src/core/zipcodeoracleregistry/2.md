# Mission 2 — The mis-scale / fail-open-read adversary (shared scale, strict 18-dp key, the read path)

MISSION: 2 — the load-bearing shared-scale + strict-18-dp key invariant + the fail-closed forward-only read

## Who you are
You attack the load-bearing design choice: there is exactly ONE shared `scale` (derived once with
`baseDecimals = 18`), so a non-18-dp lien would be SILENTLY mis-scaled — there is no per-key scale to save it.
The strict-18-dp key guard is what makes a non-18-dp lien UNREACHABLE; the read path is what must fail the
borrow closed on an unset / stale / wrong-pair quote. Your goal: find ONE path where a non-18-dp (or code-less)
key reaches `cache`, where the scale rounds/overflows wrong, where a stale mark reads as fresh, or where a
reverse-pair quote succeeds.

## The claims you are attacking
- **I-1 (scale: value identity + rounds DOWN + jumbo-no-overflow).** The ONE `scale` is derived in the ctor
  (`:82-83`) and re-derived in `setQuote` (`:99-100`) as `ScaleUtils.calcScale(LIEN_DECIMALS, quoteDecimals,
  quoteDecimals)` with `LIEN_DECIMALS = 18` (`:25`). The read returns `ScaleUtils.calcOutAmount(inAmount,
  c.price, scale, false)` (`:179`) — **the `false` flag must round DOWN (against the borrower).** Confirm the
  `scale` derivation matches `ScaleUtils.calcScale`/`from` (`ScaleUtils:53-55`, `:25-30`) and that
  `calcOutAmount(.., false)` is the round-down arm (`ScaleUtils:73-76`, `fullMulDiv` truncates). A round-UP or a
  wrong base-decimals would over-value collateral; a jumbo `price * inAmount` must not overflow (it is
  `fullMulDiv`, 512-bit).
- **I-5 (strict 18-dp key guard — LOAD-BEARING).** `_writePrice` calls `_strictDecimals(lien) != LIEN_DECIMALS
  → InvalidLienDecimals` (`:146`) on BOTH write paths. `_strictDecimals` (`:152-156`) does a RAW staticcall to
  `decimals()` and reverts if `!ok || d.length != 32` (`:154`) — a FAILED/short call reverts, NOT silent-18.
  **This is the deliberate divergence from `BaseAdapter._getDecimals`** (`BaseAdapter:37-41`), which returns 18
  on a failed call and on any address `<= 0xffffffff`. Confirm a 6-dp lien, a code-less EOA, and a contract
  without `decimals()` all REVERT here, on both `seedPrice` and the reval loop. The strongest finding is a path
  where a non-18-dp key reaches `cache`.
- **I-7 (read fail-closed).** `_getQuote` (`:166-180`) must: (a) revert `PriceOracle_NotSupported(base,
  quoteAsset)` if `quoteAsset != quote` (`:172`) — forward-only, a reverse pair fails closed; (b) revert
  `NotSupported` if `c.timestamp == 0` (unset cache) (`:174`); (c) revert `PriceOracle_TooStale(s,
  validityWindow)` if `block.timestamp - c.timestamp > validityWindow` (`:175-178`). **Trace every read branch
  and confirm there is no path that returns a price without all three passing.** Pin the staleness boundary:
  `s == validityWindow` must be FRESH (`s > window` is stale, `:177`), and the same-block / future-ts read
  (`block.timestamp <= c.timestamp`, `:175`) skips the staleness check and serves the fresh mark.
- **I-8 (no on-chain value band).** A big mark drop is accepted on the read (no plausibility band) — the read
  fails closed on staleness/unset instead. Confirm that's the deliberate posture.

## Specific attacks to run
1. **Non-18-dp key reaching the cache.** This is the headline. Can ANY lien with `decimals() != 18` (6-dp
   USDC-like, 8-dp WBTC-like, 0-dp, 36-dp) reach `cache` through EITHER write path? Walk `_strictDecimals`
   (`:152-156`): a 6-dp lien returns 6 → `6 != 18` → `InvalidLienDecimals`. A code-less EOA: the staticcall
   returns `ok=true, d.length=0` (no code → empty return) → `d.length != 32` → revert. A contract returning a
   non-32-byte blob → revert. Contrast with `_getDecimals` (`:154` vs `BaseAdapter:37-41`) — confirm the strict
   path does NOT inherit the silent-18 fallback. Any key type that slips a non-18-dp value past this is CRITICAL
   (silent mis-scale of every quote for that key).
2. **Scale round-down / direction.** Walk `calcOutAmount(inAmount, c.price, scale, false)` (`:179` →
   `ScaleUtils:63-77`). Confirm `false` is the non-inverse (round-DOWN) arm. Confirm `scale` packs
   `priceExponent = quoteDecimals` and `feedExponent = feedDecimals + baseDecimals = quoteDecimals + 18`
   (`calcScale`/`from`, `ScaleUtils:53-55,25-30`) — so the result is `inAmount * 10^qDec * price / 10^(qDec+18)
   = inAmount * price / 1e18`, which is the price-per-1e18-lien semantics. A wrong base-decimals (not 18) or a
   round-UP would over-value. Check `bid == ask == mid` (getQuotes returns `(out, out)`, `BaseAdapter:24-27`)
   leaks no rounded-up arm.
3. **Unset-cache fail-open.** Deploy mentally with no seed/push for a lien: does `_getQuote` revert at the
   `c.timestamp == 0` guard (`:174`), or does a zeroed cache (`price==0, timestamp==0`) flow into `calcOutAmount`
   and return 0 (or a usable value)? Confirm the `timestamp == 0` guard precedes any arithmetic.
4. **Staleness boundary + same-block.** Pin `:175-177`: `now - ts == validityWindow` (must be fresh) vs `>
   validityWindow` (stale). An off-by-one bricks a fresh mark (grief) or serves a one-second-too-old one
   (unsafe). Then the `block.timestamp <= c.timestamp` branch (`:175`): a same-block read (`now == ts`) or a
   future-cached-ts (impossible past the write guard, but reason it through) skips the `s` computation and
   serves the mark — confirm that's safe (it's strictly fresher than now).
5. **Wrong-pair / reverse-pair read.** `_getQuote(in, X, Y)` for any `Y != quote` reverts `NotSupported`
   (`:172`). Confirm the reverse direction `(base == quote, quoteAsset == lien)` fails closed at this guard
   (this contract is INTENTIONALLY forward-only, `:158-162` — no `getDirectionOrRevert`/inverse support unlike a
   generic adapter). Does swapping base/quote sneak through and return an inverted price? It must not.
6. **Diff vs the single-asset sibling.** `SzipFarmUtilityLpOracle` has ONE fixed key fixed at deploy; this
   registry strict-guards EVERY key at write time. Confirm the per-key strict guard is the correct multi-asset
   generalization of the sibling's fixed-key posture — and that no key-validation gap opened in the
   generalization (e.g. the read path trusting a key the write path strict-guarded).
