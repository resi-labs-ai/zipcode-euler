# Mission 3 — The identity/renounce + wiring adversary (CRE identity surface, three Timelock setters, ctor)

MISSION: 3 — the identity/renounce surface + setController/setQuote/setValidityWindow + the build-phase wiring residual

## Who you are
You attack the CRE identity/renounce surface and the build-phase mutable wiring: the three `onlyOwner`
(Timelock) setters and the ctor. The Forwarder/identity is the ratified base and the wiring is settable (not
immutable) by design until the pre-prod freeze — but a renounce that bricks a writer, a setter that silently
mis-scales every quote, or a ctor that admits a degenerate config breaks the registry without any malicious
mark. Your goal: find a renounce/identity hazard or a setter/ctor path that corrupts the scale/key/window such
that subsequent quotes are wrong (most dangerously, over-valued or fail-open) — NOT a bare "the Timelock is
trusted" restatement.

## The claims you are attacking
- **I-9 (identity / renounce surface).** The CRE identity gate lives in `ReceiverTemplate.onReport`
  (`:78-119`): a wrong `workflowId` (or author/name, once configured) reverts PRE-renounce; renouncing ownership
  FREEZES the `onlyOwner` setters (`setController`/`setQuote`/`setValidityWindow` here + the `ReceiverTemplate`
  identity setters) but the identity gate STAYS LIVE (the configured expectations are storage, not owner-gated at
  read time); a renounce-WITHOUT-a-controller bricks `seedPrice` FOREVER (`:114` reverts and the controller can
  never be set again); the Forwarder is IMMUTABLE after renounce (`setForwarderAddress` is `onlyOwner`,
  `ReceiverTemplate:127-139`). **The dormant-identity hazard** (a never-configured identity gate) is what
  `ZipcodeDeployAsserts` guards at deploy. Confirm renounce freezes the setters, the identity gate survives it,
  and the seed-brick is the documented consequence (not a new bug).
- **I-10 (`setController`).** `setController(c)` (`:89-93`): `onlyOwner` + `c == address(0) → ZeroAddress`
  (`:90`) + sets `controller` + emits `ControllerSet`. Confirm the gate and zero-guard, and that a re-point
  changes the seed authority (the old controller's `seedPrice` then reverts `NotController`).
- **I-11 (`setQuote` + `setValidityWindow`).** `setQuote(quote_)` (`:96-102`): `onlyOwner` + `quote_ ==
  address(0) → ZeroAddress` (`:97`) + RE-DERIVES `scale` from the NEW token's `_getDecimals` (`:99-100`) + emits
  `WiringSet("quote", ..)`. **`scale` is numerically decimals-invariant by construction** — `calcScale(18, qDec,
  qDec)` has `feedDecimals == quoteDecimals == qDec`, which collapses `calcOutAmount` to `* price / 1e18`
  regardless of `qDec` (the `10^qDec` cancels). So the re-derive is proven by "the NEW quote prices without
  bricking/overflow", NOT a value delta — confirm that algebra holds and the re-derive reads the LIVE new token's
  decimals (not a stale cached value). `setValidityWindow(w)` (`:105-108`): `onlyOwner`, no zero-guard
  (intentional — `0` ⇒ every read past `ts` fails closed) — confirm TIGHTENING it can flip a previously-fresh
  mark to `TooStale` (`:175-177`), and that LOOSENING it (toward near-infinite) is the only "dangerous"
  direction, bounded by Timelock trust.
- **X (build-phase mutable wiring residual).** The Timelock can re-point `controller`/`quote`/`validityWindow`;
  that's the documented subsystem residual, closed by the pre-prod immutable re-freeze. **A bare re-point
  restatement is INFO** — the sharp finding is a re-point that DRAINS or breaks the scale/liveness invariant
  without reverting, not "the Timelock is trusted".

## Specific attacks to run
1. **Renounce / seed-brick.** Walk renounce (OZ `Ownable.renounceOwnership` → owner `address(0)`). After
   renounce, `setController` reverts (`onlyOwner`) — so if no controller was wired, `seedPrice` reverts
   `NotController` (`:114`) forever. Confirm this is the documented brick (the Forwarder reval path still works;
   only the origination seed is dead). Confirm the identity gate (`ReceiverTemplate:88-117`) still rejects a
   wrong-workflowId report post-renounce (it reads `s_expected*` storage, not the owner). Is there a renounce
   ordering that leaves a FAIL-OPEN identity gate (never-configured expectations + renounce)? Classify against
   the `ZipcodeDeployAsserts` deploy-time guard.
2. **`setQuote` scale re-derive.** Walk `:96-102`: does it re-read `_getDecimals(quote_)` for the NEW token
   (`:99`) and recompute `scale` (`:100`), or could it reuse a stale value? Construct: original quote = USDC
   (6dp), re-point to an 18dp quote. Confirm `scale` changes but the QUOTE result is numerically identical
   (the `feedDecimals == quoteDecimals` collapse to `/1e18`) — so a "mis-scale on re-point to different
   decimals" is NOT exploitable here (unlike a generic adapter where `feedDecimals != quoteDecimals`). State why
   the decimals-invariance holds, then probe the ONE residual: a `quote_` whose `decimals()` reverts/returns a
   non-standard shape — `_getDecimals` silently returns 18 (`BaseAdapter:37-41`), so `scale` derives at qDec=18;
   does that mis-price? (Reason: it still collapses to `/1e18`, so no — but verify.)
3. **`setValidityWindow` direction.** Confirm `setValidityWindow(0)` is a safe fail-closed state (every read
   past `ts` → `TooStale`), not fail-open. Confirm tightening flips a fresh mark to stale (grief, recoverable)
   and only loosening toward near-infinite is the dangerous direction — bounded by Timelock trust, INFO.
4. **`setController` re-point old-authority leak.** After `setController(new)`, confirm the OLD controller's
   `seedPrice` reverts `NotController` (`:114`) — the authority actually moves, no dual-authority window.
5. **ctor zero / scale.** `constructor(forwarder, quote_, validityWindow_)` (`:79-84`): zero `forwarder` reverts
   `InvalidForwarderAddress` in `ReceiverTemplate:45-47`; `quote_` is NOT zero-guarded in the ctor (only
   `setQuote` is) — reason about whether a zero `quote_` at deploy yields a fail-open oracle or just a
   `_getDecimals(0)`→18 + every read `NotSupported` (quoteAsset can't be 0 in a real EVK call). Is any ctor arg
   left in a fail-open default? Confirm `scale` derives correctly at construction.
6. **Diff vs the sibling's setter discipline.** Compare the three setters against
   `SzipFarmUtilityLpOracle`'s wiring setters (`setForwarder`/`setQuote`/`setLpToken`/`setValidityWindow`) — same
   `onlyOwner` + zero-guard + re-derive shape? Note this registry has NO `setLpToken` (keys are dynamic, per-write
   strict-guarded) but ADDS `setController` (the seed authority the sibling lacks). Any guard the sibling has that
   this one dropped, or vice versa (the recurring subsystem setter-gap class)?
