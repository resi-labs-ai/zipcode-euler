# Mission 2 — The burn authority & ctor zero-guard (can a non-controller burn, or a third party's tokens?)

MISSION: 2 — controller-only `burn` (I-3), burn edge behavior (I-4), and the ctor zero-guard (I-6)

## Who you are
You attack WHO can destroy supply and HOW. `burn` must be callable ONLY by the immutable `controller`, and it
must burn ONLY the controller's own balance — not an arbitrary holder's, and not via a third-party allowance.
The constructor must reject a zero controller. Your goal: ONE path where a non-controller burns, where a burn
reaches a third party's tokens, where the edge cases (zero / over-balance) misbehave, or where a controller-less
token can be deployed.

## The claims you are attacking
- **I-3 (burn is controller-only).** `burn(amount)` (`:32-35`) checks `if (msg.sender != controller) revert
  NotController()` (`:33`) before `_burn` (`:34`). **Find a non-controller path to `_burn`** — another function
  that calls it, an inherited OZ function exposed (OZ `ERC20` has no public burn — there is no `ERC20Burnable`
  here), or a `controller` that could be re-pointed to an attacker (there is no setter — confirm it). Any
  caller other than the wired controller must revert `NotController`.
- **I-4 (burn edge behavior).** `burn` calls `_burn(controller, amount)` (`:34`). **`burn(0)` is a no-op**
  (OZ `_burn` of 0 emits a zero-value `Transfer`, mutates nothing) and an **over-balance burn reverts** (OZ
  `_burn`'s balance check — `ERC20InsufficientBalance`). Confirm neither edge lets supply go negative or wrap.
- **I-6 (the ctor zero-guards the controller).** `constructor(address controller_)` (`:18`) reverts on a zero
  `controller_` via `require(controller_ != address(0), "LienCollateralToken: zero controller")` (`:19`).
  **Confirm a controller-less token can't be deployed** — a zero controller would mint the `1e18` supply to the
  zero address (an immediate, unrecoverable loss) AND leave `burn` permanently uncallable (`msg.sender !=
  address(0)` always reverts `NotController`). The guard prevents both.
- **The controller is immutable.** `address public immutable controller` (`:12`), set once in the ctor (`:20`).
  **Confirm there is genuinely NO setter, no second authority, no owner, no re-point path** — the only authority
  this token has is `burn`, and it can only destroy the controller's own balance. A burn cannot move value to a
  third party (it only destroys tokens). A hidden admin/setter/upgrade would be a finding.

## Specific attacks to run
1. **The `NotController` check robustness.** `if (msg.sender != controller) revert NotController()` (`:33`).
   Confirm `controller` is the immutable address set in the ctor, with no path to change it. A non-controller
   `msg.sender` always reverts. No operator/admin role exists on this token (confirm — the X-Ray says no admin).
2. **Burn targets the controller's own balance, not a third party.** `_burn(controller, amount)` (`:34`) burns
   from `controller`, NOT from `msg.sender` and NOT from an arbitrary `from` (there is no `burnFrom`, no
   allowance path). **Confirm a non-controller cannot burn anyone's tokens, and the controller cannot burn a
   third party's tokens** (it only ever destroys its own balance). A burn-of-a-third-party reachable by anyone
   would be theft.
3. **Edge cases.** `burn(0)` no-op (supply unchanged) and over-balance burn reverts (`ERC20InsufficientBalance`
   from OZ `_burn`). Confirm there is no unchecked arithmetic — `_burn` underflow is caught by OZ's balance check.
4. **The ctor zero-guard + immutability.** `require(controller_ != address(0))` (`:19`) and `immutable
   controller` (`:12`, set `:20`). Confirm a controller-less token can't be deployed, and that once set the
   controller can never be re-pointed (no setter exists). The attack surface is exactly: the ctor
   guard, the `NotController` gate, and the OZ `_burn` balance check. A hidden privilege would be a finding.

## What is already proven (don't re-report as new)
`test_BurnByNonControllerReverts` (`NotController`), `test_BurnZeroNoOp` (supply unchanged after `burn(0)`),
`test_BurnOverBalanceReverts` (`ERC20InsufficientBalance` on `1e18 + 1`), `test_ConstructorRevertsOnZeroController`
(ctor rejects `address(0)`), and `test_BurnByControllerDropsSupplyAndEmits` (the happy burn). Confirming burn is
controller-only, burns only the controller's own balance, the edges hold, and the ctor zero-guards the immutable
controller goes in your Summary. A finding must be a concrete non-controller-burn / third-party-burn / zero-or-
wrapping-edge / controller-less-deploy / hidden-setter path the dedicated tests do NOT cover — a restatement of
"the controller is the sole authority" is INFO (that's the design, and it's immutable).
