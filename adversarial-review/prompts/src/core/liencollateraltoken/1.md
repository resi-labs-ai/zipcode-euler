# Mission 1 — The supply primitive (can supply grow, drift, or move via a hidden hook?)

MISSION: 1 — fixed once-only `1e18` (I-1, I-2), the `decimals()` pin (I-5), the vanilla-ERC20 claim (I-7), and the venue's load-bearing `1e18` reliance (I-8)

## Who you are
You attack the only thing this token promises consumers: **one whole token, ever, per lien**. The supply must be
exactly `1e18`, minted ONCE at construction, and can only SHRINK thereafter (controller burn — mission 2). The
token must add NOTHING to the OZ ERC-20 surface that could move balances outside that (no hook, fee, rebase,
pause, blocklist), and `decimals()` must stay pinned to 18. Your goal: ONE path that adds supply post-
construction, a hidden modification to the OZ surface, a decimals drift, or any way the venue's `1e18` reliance
could be defeated.

## The claims you are attacking
- **I-1 (fixed `1e18` minted once to the controller).** The constructor (`:18-22`) mints exactly `1e18` to
  `controller_` (`:21`) and sets the `immutable controller` (`:20`). **Find any OTHER mint path** — a second
  `_mint` call, an inherited OZ function that mints (OZ `ERC20` exposes NO public mint), an `_update` override
  that could mint. There is exactly one `_mint`, in the ctor. Confirm it.
- **I-2 (supply can only shrink — no post-construction mint).** After construction the ONLY supply mutator is
  `burn` (mission 2). **Confirm there is structurally no path that increases `totalSupply`.** A second mint, or
  a mint reachable by anyone, breaks the 1/1 primitive.
- **I-5 (`decimals()` pinned to 18).** `decimals()` (`:26-28`) is a constant `pure` override returning `18`
  (legal: overriding a `view` base with `pure`). **This is a deliberate pin** so a base-contract bump can't
  silently shift the scale the oracle infers. Confirm it always returns 18, with no path to a different value.
- **I-7 (freely transferable, unmodified OZ).** Everything but the ctor / `decimals` / `burn` is stock OZ.
  **Find ANY hidden override** — a `_update`/`_beforeTokenTransfer`/`_afterTokenTransfer` hook, fee-on-transfer,
  rebase, pause, blocklist, transfer hook — that the "vanilla" claim denies. Read the whole contract: it should
  be exactly `ERC20` + the 3 additions, nothing else.
- **I-8 (the `1e18` supply is what the venue relies on — cross-contract).** `EulerVenueAdapter.openLine`
  requires `collateralAmount == 1e18` (`:316`); `closeLine` redeems exactly the `1e18` escrow shares back to
  `controller` (`:534-540`) before `ZipcodeController.closeLine` burns the lien (`:301`). **A token that minted
  more, or started at a different supply, would break the line lifecycle** (a partial deposit opens a line that
  can never cleanly close — the reclaim underflows). Confirm the token makes that impossible by construction.

## Specific attacks to run
1. **Enumerate every state-changing path.** The only functions are the ctor, `decimals` (pure, read-only),
   `burn` (mission 2), + inherited OZ transfer/approve. Confirm there is NO other path to `_mint` — no inherited
   OZ public mint, no `_update` override that mints/rebases. The single `_mint` is the ctor's (`:21`).
2. **The decimals pin.** Confirm `decimals()` (`:26-28`) is `public pure override returns (uint8)` and returns
   the constant `18`, with no storage read and no branch. A drift here would silently re-scale the oracle's read
   of the collateral.
3. **The vanilla claim — read for hidden hooks.** Confirm the contract is `contract LienCollateralToken is ERC20`
   (`:10`) with NO `_update` override, no `_before`/`_afterTokenTransfer`, no fee logic, no rebase, no pause, no
   blocklist. The ONLY overrides/additions are the ctor, `decimals`, and `burn`. A hidden transfer hook would
   mean balances move differently than stock OZ — confirm there is none.
4. **The venue reliance.** Confirm the `1e18` mint matches what `openLine` requires (`:316`) and what the close
   path reclaims (`:534-540` redeem to `controller`, then `ZipcodeController.closeLine` `:301` burns
   `FULL_LIEN`). The token starting at exactly `1e18` is what lets the line open and close cleanly. A token whose
   supply could deviate from `1e18` would break this — confirm the supply is structurally fixed at `1e18`.
