# Mission 3 — The reallocate adversary (donation-immune sizing, fund/defund, farm-utility two-key)

MISSION: 3 — donation-immune reallocate sizing (I-5/SEC-11), defund-to-base on close (I-7/SEC-07),
farm-utility JIT fund/defund + two-key (I-15)

## Who you are
You attack the liquidity-movement path: the claim that EVERY `reallocate` (fund, the close-time defund, and the
farm-utility JIT moves) sizes its absolute targets off the EE's **TRACKED** supplied position
(`previewRedeem(config.balance)`), NOT the live balance — so a direct EVK-share donation into the pool can't
skew the targets into an `InconsistentReallocation` revert (grief). And that the farm-utility fund/defund is
gated by `onlyFarmUtilityAllocator`, a key DISTINCT from the loop operator. Your goal: ONE reallocate whose
sizing reads the LIVE balance (donation-skewable), one that leaves USDC stranded, or one the wrong key can
drive.

## The claims you are attacking
- **I-5 (SEC-11 — donation-immune sizing).** `_eeSupplyAssets(market)` (`:413-415`) returns
  `IEVault(market).previewRedeem(eulerEarn.config(market).balance)` — the EE's TRACKED share balance, byte-
  matching what `reallocate` measures internally (`EulerEarn.sol:392-393`). `_eeMove(from, to, amount)`
  (`:425-432`) sizes BOTH absolute targets off `_eeSupplyAssets` (`:426-427`): `from → fromBalance - amount`,
  `to → toBalance + amount`. **Confirm NEITHER target is read off `convertToAssets(balanceOf(EE))` (the LIVE
  balance) nor `maxWithdraw`** — both are skewable/under-reading. Find any reallocate (fund, defund, farm-utility)
  whose sizing reads a donation-skewable measure.
- **I-7 (SEC-07 — defund-to-base on close).** `closeLine` (`:527`) defunds the line's USDC back to the base
  reservoir BEFORE pruning the queue (`:558-565`): if `lineBalance = _eeSupplyAssets(lineRef) != 0`, reallocate
  `{lineRef → 0, usdcReservoir → baseBalance + lineBalance}` (the inverse of fund). **Confirm:** (a) the line
  leg's absolute target is `0` (full redeem sweeps even donated shares, `EulerEarn.sol:397-402`); (b) the base
  target adds the line's tracked assets; (c) a never-funded line (`lineBalance == 0`) NO-OPS the defund (`:559`)
  — a `{x:0}` withdrawal-only leg can't balance. Find a strand: USDC left in a closed line's vault that later
  underflows a `fund`'s `baseBalance - amount` (`:429`).
- **I-15 (farm-utility two-key JIT).** `fundFarmUtility(amount)` (`:635`) = `_eeMove(usdcReservoir,
  farmUtilityVault, amount)`; `defundFarmUtility(amount)` (`:645`) = `_eeMove(farmUtilityVault, usdcReservoir,
  amount)` — the EXACT `fund` shape, donation-immune. Both `onlyFarmUtilityAllocator` (`:153-156`,
  `NotFarmUtilityAllocator`). **Confirm:** (a) the gate is a DISTINCT key from the `FarmUtilityLoopModule.operator`
  (the two-key separation — though this is a DEPLOY invariant with no on-chain handle: the proof is the loop
  operator key reverts `NotFarmUtilityAllocator`); (b) a `defundFarmUtility` while the farm-utility's cash is
  still borrowed out REVERTS on the EVK withdraw leg (`E_InsufficientCash`) — JIT/redemption-isolation, not a
  silent under-defund.

## Specific attacks to run
1. **Grep every reallocate sizing read.** Find every `_eeSupplyAssets` call (`:426-427` in `_eeMove`, `:558`
   and `:560` in `closeLine`'s defund) and confirm EACH reads `previewRedeem(config.balance)` — never
   `balanceOf` / `convertToAssets` / `maxWithdraw`. The X-Ray's `test_SEC11_PreFixSizing_Reverts_OnDonation`
   proves the OLD (live-balance) sizing reverts on a donation; confirm the CURRENT code uses the tracked measure
   everywhere. A single reallocate that slipped back to a live read is the headline finding.
2. **The donation attack.** Construct: an attacker donates 1 EVK share directly into the EE pool (or into the
   line vault). Trace whether `fund` (`:445`), the close defund (`:558-565`), or `fundFarmUtility` (`:636`) then
   reverts `InconsistentReallocation` (the withdraw/supply deltas no longer net). They SHOULDN'T — the tracked
   measure ignores direct transfers (`IEulerEarn.sol:69,73`). Confirm the donation is a no-op for sizing.
3. **The zero-sum/ordering invariant.** `_eeMove` (`:425-432`) reads `from` BEFORE `to` (`:426-427`), then
   builds `allocs[0] = {from, fromBalance - amount}` (withdraw leg first), `allocs[1] = {to, toBalance + amount}`
   (`:429-430`). Confirm the withdraw leg is FIRST so the pass has cash before it deposits, and that the move is
   exactly zero-sum (`-amount` / `+amount`). A `fund` with `amount > fromBalance` underflows `fromBalance -
   amount` (`:429`) — confirm that reverts (Solidity 0.8 checked sub) rather than wrapping. Is an over-fund a
   griefable revert or a benign bound?
4. **Defund ordering vs prune.** `closeLine` runs the defund (`:558-565`) BEFORE the supply-queue prune
   (`:575-583`) and the withdraw-queue reclaim (`:599-617`) — so the pruned/removed market is already emptied.
   Confirm the defund MUST precede the prune (an EE withdraw-queue removal of a NON-empty market engages the EE
   timelock, `EulerEarn.sol:366-370` — the comment at `:592-594` claims the defund emptying it avoids that).
   Find an ordering where a still-funded market hits the removal path and either reverts or strands.
5. **Two-key gate.** Confirm `fundFarmUtility`/`defundFarmUtility` revert `NotFarmUtilityAllocator` for ANY
   caller != `farmUtilityAllocator` — including the controller and the owner (it is NOT `onlyController` /
   `onlyOwner`). The cross-suite `FarmUtilityLoopModule.t.sol` proves the loop operator key reverts. The
   distinctness from the loop operator is the KNOWN residual (no on-chain handle) — confirm the gate fires;
   don't re-flag the residual as a vuln.
