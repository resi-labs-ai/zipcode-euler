# Mission 1 — The venue-neutrality adversary (does an Euler type leak, and does conformance hold?)

MISSION: 1 — venue-neutrality discipline (P-4) + dual-sided conformance (P-1/P-2/P-3)

## Who you are
You attack the one meaningful property of a pure interface: the venue-neutrality discipline. The claim is that
ONLY `bytes32`/`address`/`uint*`/an opaque `lineRef` cross this boundary — NO Euler type — so the controller
stays venue-agnostic and a second venue is just another adapter. And that conformance is real on BOTH sides:
the adapter implements every member, the controller consumes ONLY through the interface type. Your goal: ONE
Euler type leaking into a signature/event, or a conformance/consumption break. (An interface has no runtime
surface — a runtime finding here is impossible; soundness is the expected result.)

## The claims you are attacking
- **P-4 (venue-neutrality).** Read EVERY one of the 7 function signatures (`openLine:44`, `setLineLimits:49`,
  `fund:52`, `draw:55`, `observeDebt:58`, `closeLine:61`, `liquidate:64`) and 5 events (`LineOpened:17`,
  `LineLimitsSet:27`, `LineFunded:30`, `LineDrawn:33`, `LineClosed:36`). **Confirm EVERY parameter and return is
  a primitive (`bytes32`/`address`/`uint16`/`uint256`) or the opaque `lineRef` (an `address`)** — NO `IEVault`,
  `MarketAllocation`, `BatchItem`, EVC, `EulerRouter`, or `LineAccount` type appears anywhere. A single Euler
  type in a signature is the headline finding (it couples the controller to Euler, defeating §4.7).
- **The opaque-`lineRef` discipline.** `lineRef` is an `address` the controller treats as a HANDLE — it happens
  to be the borrow vault, but the controller never introspects it. **Confirm the interface declares it as a bare
  `address`** (not `IEVault` or any Euler vault type), so a different venue can return a different KIND of handle
  without breaking the controller. This is the load-bearing design choice that makes the seam genuinely
  venue-agnostic.
- **P-1 (compile-time conformance).** `EulerVenueAdapter is IZipcodeVenue` — Solidity's `is` check makes the
  contract compile iff EVERY member is implemented with the exact signature. **Confirm** the adapter implements
  all 7 functions + emits all 5 events with byte-matching signatures (`test_InterfaceImplemented` casts
  `IZipcodeVenue(address(adapter))`).
- **P-3 (consumption through the seam).** `ZipcodeController` holds `venue` and calls `IZipcodeVenue(venue_).{...}`
  through the INTERFACE TYPE (`ZipcodeController.sol:58`, `:240-279`). **Confirm the controller NEVER casts
  `lineRef` to an Euler type, never imports an Euler type for the venue path, and never calls a venue method
  outside the interface** — any Euler-typed call from the controller is a leak.

## Specific attacks to run
1. **Grep for Euler types in the interface.** Scan `IZipcodeVenue.sol` for any of: `IEVault`, `IEVC`, `IEVC.`,
   `MarketAllocation`, `BatchItem`, `EulerRouter`, `IEulerEarn`, `LineAccount`, `GenericFactory`. There should be
   ZERO. Confirm the only imports (if any) and every type used are primitives. (The file has NO imports — confirm.)
2. **The senior-surface exclusion.** `IZipcodeVenue` carries NO `seniorPool()` getter — that lives on the
   adapter (`EulerVenueAdapter.seniorPool() :522`), the one venue-specific getter the `SiloRegistry` needs
   (CTR-10b). **Confirm this is an INTENTIONAL omission** (keeping the line-only interface venue-neutral, §4.7),
   NOT a missing member. Do NOT report it as a gap.
3. **The consumption surface.** Open `ZipcodeController.sol` (`:58`, `:240-279`) and confirm every venue call is
   `IZipcodeVenue(venue_).method(...)` — the controller holds the venue as the interface type and never as
   `EulerVenueAdapter`. Confirm `lineRef` returned by `openLine` is stored/passed back as a bare `address`,
   never introspected or cast. A controller that reaches into the adapter's Euler internals would break
   venue-neutrality even though the interface itself is clean.
4. **`liquidate` as a declared stub.** The interface declares `liquidate(address)` (`:64`) even though the Euler
   impl always reverts `NotImplemented` (§4.4e). **Confirm** declaring it keeps the interface complete for venues
   that DO liquidate on-chain — an intentional design choice, not a dead member.

## What is already proven (don't re-report as new)
`test_InterfaceImplemented` (compile-time conformance, P-1), the full 53-test `EulerVenueAdapter.t.sol` behavioral
suite (every member driven + every event asserted, P-2), and `ZipcodeController.t.sol` (venue-agnostic
consumption, P-3). Confirming no Euler type crosses the boundary + conformance holds both sides goes in your
Summary. A finding must be a concrete Euler-type leak in a signature/event, a conformance divergence, or a
controller-side introspection of `lineRef` — NOT a runtime "vuln" (impossible on a pure interface) and NOT the
intentional senior-surface omission.
