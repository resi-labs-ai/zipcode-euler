# 8-B1 — szipUSD Baal+Zodiac substrate scaffold (wiring map)

> Source of truth = the kept code: `contracts/script/SummonSubstrate.s.sol` + the
> `contracts/src/interfaces/{baal,safe}/*` it drives. Ticket `tickets/sodo/8-B1-substrate-scaffold.md`
> + report `reports/8-B1-report.md` are intent — the script is the final form. This doc reads the code
> as authoritative and records the summon sequence, the two-Safe shape, and the authority model every
> downstream 8-B component bolts onto.

## Role
8-B1 is **not a compiled-from-source contract** — it is a Foundry **summon SCRIPT** that drives the
**live** `BaalAndVaultSummoner` on Base 8453 (Baal is solc 0.8.7, deployed; we interact via minimal local
0.8.24 interfaces, never compile — same Strategy-A rule as the Euler/Safe deps, WOOF-00). One call produces
the entire szipUSD junior-vault substrate in one flow: a **Baal (Moloch v3) DAO**, its **main Gnosis Safe**
(the Baal avatar — ragequit target, holds the FREE equity), a **non-ragequittable juniorTrancheSidecar Safe** (holds the
COMMITTED equity — the §11 structural freeze), and the **Loot + Shares** ERC20 clones (Loot
soulbound/paused at genesis; **Shares = 0 forever**). The script then injects the **team multisig as a Safe
owner/signer on BOTH Safes** so the otherwise governance-inert substrate is driveable. The "wiring" of 8-B1
is therefore (a) the summon sequence + the authority injection, and (b) the summoned-Safe addresses every
downstream engine module / Gate / oracle pins to.

## Wiring — internal (the summon sequence)

The whole flow is `_summon` → `_summonCore` (summon + main-Safe owner-add, caller-agnostic) → `_addOwnerToSidecar`.

**1. Precompute the main Safe (`computeMainSafe`).** The summoner deploys the main Safe via Safe-1.3.0
`createProxyWithNonce(gnosisSingleton, "", saltNonce)` (empty initializer). The script re-derives the address
**from live chain state** — never hardcoded:
- `factory = BaseAddresses.BAAL_SAFE_PROXY_FACTORY` (the summoner's factory field is `internal`/no getter, so it is pinned + caught by the assert below),
- `singleton = IBaalSummoner(BAAL_SUMMONER).gnosisSingleton()` (read live; = `0x69f4…2938`),
- `creationCode = ISafeProxyFactory(factory).proxyCreationCode()` (read live),
- `salt = keccak256(abi.encodePacked(keccak256(bytes("")), saltNonce))`,
- `initCodeHash = keccak256(creationCode ++ abi.encode(uint256(uint160(singleton))))`,
- `vm.computeCreate2Address(salt, initCodeHash, factory)`.

**2. Build init params + the 6 ordered init-actions (`_buildInitActions`).** `initParams =
abi.encode(name, symbol, address(0)×4)` — all-zero safe/forwarder/loot/shares ⇒ the summoner deploys fresh
ones; forwarder 0 ⇒ no EIP-2771. `name`/`symbol` (`"Zipcode szipUSD Junior"`/`"zJR"`) name the **internal**
Loot/Shares tokens only — never user-facing (Loot is Gate-held, Shares = 0). The summoner multisends the
actions as one DelegateCall during `Baal.setUp`, executed BY the Safe (avatar) so every `baalOnly` guard
passes:
1. `setAdminConfig(true, true)` — confirm Loot+Shares paused (no-op; `deployTokens` already paused them) ⇒ **soulbound from genesis**.
2. `setGovernanceConfig(...)` — permanently inert: `voting=2 days`, `grace=1 day`, `offering=0`, `quorum=0`, **`sponsor=type(uint256).max`** (un-meetable), `minRetention=0`.
3. `setShamans([], [])` — **no shaman at summon** (the Exit Gate gets `manager(2)` later via the team-admin signer).
4. `mintShares([], [])` — **zero Shares forever** (ragequit purity + governance inertness).
5. `mintLoot([], [])` — **zero Loot at summon** (genesis seed mints Loot later, only after the Gate holds `manager`).
6. **The authority injection:** `executeAsBaal(juniorTrancheSafe, 0, <selfAddOwnerPayload>)`.

**The authority-injection chain (action 6).** `executeAsBaal` runs raw `_to.call{value}(_data)` **as the
Baal** — so a direct `addOwnerWithThreshold` would revert (caller = Baal ≠ Safe). It must wrap through the
Safe's self-authorized module path. `_selfAddOwnerPayload` builds:
`execTransactionFromModule(juniorTrancheSafe, 0, addOwnerWithThreshold(team, 1), Call)`. Full chain:
Safe → `Baal.executeAsBaal` (avatar ✓) → Baal → `juniorTrancheSafe.execTransactionFromModule(juniorTrancheSafe, 0, addOwner, Call)`
(Baal is the Safe's enabled module ✓) → Safe → `juniorTrancheSafe.addOwnerWithThreshold(team, 1)` (`msg.sender == self`,
self-authorized ✓). **Threshold stays 1.** This requires the main-Safe address at encode time — hence the
precompute in step 1.

**3. Summon + resolve + fail-closed (`_summonCore`).** `summonBaalAndVault(initParams, actions, saltNonce,
bytes32(0), VAULT_NAME)` returns `(baal, juniorTrancheSidecar)` — `daoAddress = Baal`, `vaultAddress = juniorTrancheSidecar`. Then:
- `juniorTrancheSafe = IBaal(baal).avatar()` and **`if (predictedMainSafe != juniorTrancheSafe) revert MainSafeMismatch(...)`** — the compute-and-assert. This **fails closed** on any factory/singleton/salt drift (it actually caught the initial wrong-factory guess during the build → the `0xC228…` correction).
- `loot = IBaal(baal).lootToken()`, `shares = IBaal(baal).sharesToken()`.
- **`if (!ISafe(juniorTrancheSafe).isOwner(teamMultisig)) revert TeamOwnerNotSet(juniorTrancheSafe)`** — asserts the injection landed as an *effect* (the inner module-exec returns bool, never just "didn't revert").

**4. Sidecar owner-add (`_addOwnerToSidecar`).** `summonBaalAndVault` deploys the juniorTrancheSidecar **after** the Baal
`setUp` (via `deployAndSetupSafe(dao)`), so it cannot ride the init-actions — it ships **owned 1/1 by Baal,
no team owner** (the one genuine inert window). `_summon` closes it in the same flow (never deferred): the
team is now a main-Safe owner, so it drives the main Safe to reach the juniorTrancheSidecar through Baal:
`juniorTrancheSafe.execTransaction(baal, 0, executeAsBaal(juniorTrancheSidecar, 0, execTransactionFromModule(juniorTrancheSidecar, 0,
addOwnerWithThreshold(team, 1), Call)), Call, …)`. The signature is the Safe **pre-validated single-owner**
form — `signatures = abi.encodePacked(bytes32(uint160(team)), bytes32(0), uint8(1))` — valid because
`msg.sender == owner` (no ECDSA / no prior `approveHash`); all other `execTransaction` params zero. The
broadcaster MUST therefore be the team signer (`run()` models the MVP path; production replaces this with
real m-of-n sigs, identical on-chain shape). Post-assert: **`if (!ISafe(juniorTrancheSidecar).isOwner(team)) revert
TeamOwnerNotSet(juniorTrancheSidecar)`**.

**Invariants this establishes (the substrate contract with downstream):**
- **Two Safes, distinct.** `avatar() == target() == juniorTrancheSafe ≠ juniorTrancheSidecar`; both have `isModuleEnabled(baal) == true`, `getThreshold() == 1`, and **`isOwner(team) == true`**.
- **Zero Shares forever.** `totalShares() == 0` and `shares.totalSupply() == 0`. Authority is **Safe ownership, not votes** — so the Baal is governance-inert by design (no proposal can be sponsored: `sponsorThreshold = max`, `getVotes = 0`).
- **Zero Loot at summon**, both tokens paused/soulbound, `owner() == baal`.
- **Baal as the ragequit module**, not the authority: the Baal is an enabled module on both Safes (the ragequit path), but it can only act via a proposal (impossible at zero Shares) — so the team-admin Safe-owner is the sole live authority.

## Wiring — cross-component (which downstream contracts consume the summoned addresses)

The substrate exposes five addresses (`Substrate { baal, juniorTrancheSafe, juniorTrancheSidecar, loot, shares }`) that the rest
of the §8-B build pins to. Per `reports/baal-spec.md §13` the build order is: 8-B1 substrate → `SzipNavOracle`
→ Exit Gate + szipUSD → engine 8-B5…8-B10 / 8-B14 → 8-Bw / loss side / queue. The seams 8-B1 leaves open are
all reachable **only via the team-admin Safe-owner path** (`Safe.execTransaction`), never via a Baal proposal:

- **Exit Gate (baal-spec §4/§5) → `baal` + `loot` + `manager(2)`.** The Gate is the sole real-Loot custodian + szipUSD minter. It is granted `manager(2)` by **team-admin → `juniorTrancheSafe.execTransaction` → `Baal.setShamans([gate], [2])`** (NOT a proposal, NOT a raw post-deploy `setShamans` by anyone). Genesis Loot is minted only **after** the Gate holds `manager`. **Obligation 8-B1 creates:** the Gate (and every `manager`-holder) MUST be structurally unable to call `mintShares` (`manager` grants both mint paths; any Shares mint breaks the zero-Shares invariant) — the Gate only ever calls `mintLoot`/`burnLoot`, and every downstream fork test asserts `totalShares() == 0` (security F4.2).
- **Engine strategy modules (8-B5…8-B10, 8-B14) → `juniorTrancheSafe`, `enableModule`'d.** Each engine module is a Zodiac module the team-admin enables on the **main** Safe (it runs on the free equity); they act via `execTransactionFromModule`. The CRE robot drives their `onlyOperator` entrypoints through the CRE operator module.
- **`DurationFreezeModule` (the §8 freeze/rotation module) → BOTH Safes, enabled.** `commit` is the only juniorTrancheSidecar-funding path (proven DISCHARGED); it is enabled on the main Safe and the juniorTrancheSidecar. **Obligation 8-B1 creates (F6.2):** do NOT fund/route value into the juniorTrancheSidecar until `isOwner(team) == true` on it (8-B1's `_addOwnerToSidecar` lands this) and the team has proven it can drive it — `commit` fails closed (`execTransactionFromModule` reverts) until the module is enabled on the team-owned juniorTrancheSidecar.
- **`SzipNavOracle` (baal-spec §3) → reads BOTH Safes.** The NAV oracle sums basket balances across the main Safe **and** the juniorTrancheSidecar (free + committed) to price issuance/exit.
- **CRE operator module → enabled on both Safes, Roles-scoped (item 10).** The hot key that runs the strategy entrypoints. Enabling a module = full Safe power ⇒ only the admin may change the set.

## Item-10 deploy facts (the obligations 8-B1 records for the deploy/wiring window)

8-B1 is summoned at deploy by the real team. Its `run()` env-path is the MVP model; the obligations it files
into `tickets/PROGRESS.md` (security F3.1/F3.2/F5.1/F1.1) for the production deploy:

- **F3.2 — real `TEAM_MULTISIG` MUST be a true k-of-n (k≥2) Gnosis Safe, never an EOA / 1-of-n.** The main Safe's threshold-1 delegates ALL admin security to the team multisig's *internal* quorum; `run()` should reject an EOA (`code.length > 0`) or loudly flag it.
- **F3.1 — remove Baal as a Safe OWNER (`removeOwner`) once human signers settle.** 8-B1 keeps Baal as a vestigial threshold-1 owner for build simplicity; a contract left as a threshold-1 owner is a standing signature-path surface. Post-bootstrap, keep Baal **only as the ragequit module**, not an owner.
- **F5.1 — the CRE operator module MUST be enabled Roles-modifier-v2-scoped** (only its strategy entrypoints; NEVER `enableModule`/`addOwner`/`setShamans`/`mintShares`), never as a bare unscoped module. The `mockModule` enable in 8-B1's test is **test-only**, not the production wiring template.
- **F1.1 — use an unpredictable, single-use `saltNonce` + private (flashbots-style) submission.** The main-Safe CREATE2 address depends only on `(factory, singleton, saltNonce)` (empty initializer), so a fixed/sequential salt is front-run-griefable (a mirror tx pre-occupies the slot → summon reverts; fail-closed but a cheap grief). The fork test's constant salts (`SALT=1`/`SALT2=2`) are test-only.
- **The `BAAL_SAFE_PROXY_FACTORY 0xC228…10BC` precompute.** Build-discovered (code is truth): the summoner's own Safe proxy factory is read from BaalSummoner **storage slot 208**, NOT `SAFE_PROXY_FACTORY_1_3_0 0xa6B7…`. The deploy must keep this constant; the `compute == avatar` assert fails closed on any drift.
- **Settle-the-set sequence.** Item 10: the team enables the CRE operator + strategy modules on both Safes, grants the Gate `manager` via `setShamans`, applies role-locks, then removes Baal as owner — all once the module set is settled. (8-B1 deliberately does NOT `lockManager`/`lockGovernor`/`lockAdmin` — the Gate still needs `manager` granted later.)

## Gotchas
- **`executeAsBaal(juniorTrancheSafe, 0, addOwnerWithThreshold(...))` directly REVERTS** — caller is the Baal, not the Safe. It MUST wrap through `execTransactionFromModule(juniorTrancheSafe, …)` (Safe self-auth). The same wrap is reused for every later team-driven mutation.
- **The substrate is born frozen if not driven at summon** — this is the bug 8-B1 exists to prevent. The summoner forces each Safe owned 1/1 by the Baal, and at zero Shares the Baal can only act via an impossible proposal. The team-owner injection (the one authority-injection window) is mandatory; `_summon`/`run()` MUST NOT exit "successful" having left either Safe Baal-only/undriveable (both the main-Safe and juniorTrancheSidecar owner-adds hard-revert via `TeamOwnerNotSet`).
- **Never mint any Shares, ever** — no "bootstrap shares." Authority is Safe ownership, not votes; any Shares mint destroys the governance-inertness invariant.
- **Don't compile any `reference/Baal` / Safe / Zodiac source** — OZ-4 / pragma-0.8.7 collision with the Euler OZ-5 build (WOOF-00 Strategy A). Drive the live deployments through the local 0.8.24 interfaces only.
- **Use the return tuple + getters for address extraction, NOT the `SummonBaal` event** — a Solidity script can't parse an external call's events; the addresses come from the return tuple (`baal`, `juniorTrancheSidecar`) and `IBaal` getters (`avatar`, `lootToken`, `sharesToken`).
- **`submitProposal` does NOT revert at zero Shares / zero offering** (it succeeds unsponsored) — the genuine dead-end is downstream: `sponsorProposal`/`processProposal` revert `!sponsor`. The negative proof is "submit succeeds but can't be sponsored/processed," plus a direct `setShamans` reverting `!baal` (proving the team-owner→Safe→Baal chain is the ONLY path).
